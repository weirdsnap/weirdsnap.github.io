# 给游戏助手后端配 HTTPS：一团乱七八糟的折腾

前几天给博客加了个 Hollow Knight 游戏助手页面，前端用 Vue 3 写了个聊天界面，后端跑在服务器 `localhost:8765` 上。本地调试的时候一切正常，结果推到 GitHub Pages 后，页面打不开了——不对，页面能打开，但一提问就报错。

打开控制台一看：

```
Mixed Content: The page at 'https://weirdsnap.github.io/...' was loaded over HTTPS,
but requested an insecure resource 'http://api.weirdsnap.top/api/ask'.
```

哦，GitHub Pages 强制 HTTPS，而我的 API 是 HTTP。浏览器直接拦了。

## 方案一：Cloudflare Tunnel

第一反应是用 Cloudflare Tunnel。不需要开端口，不需要备案，还能自动 HTTPS，听起来完美。

服务器上装了 `cloudflared`，执行 `cloudflared tunnel login`，弹出授权页面。然后我就开始等。等了好久。页面一直在转。

等得不耐烦了，开始想别的办法。

## 方案二：非标准 HTTPS 端口

我的域名 `api.weirdsnap.top` 已经解析到服务器 IP，但云厂商把 80 和 443 封了——没备案不让用。不过听说有些云厂商只封标准端口，8443 这种非标准端口可能不管用。

问题是 HTTPS 证书。Let's Encrypt 默认用 HTTP-01 验证，需要 80 端口。如果 80 也被封，就得用 DNS 验证，更麻烦。

于是我先 `curl` 了一下 80 端口：

```bash
curl -I http://api.weirdsnap.top:80
```

结果：

```
HTTP/1.1 200 OK
Server: Caddy
```

80 居然是通的！而且服务器上已经有 Caddy 在跑了。那 HTTP-01 验证就能用，8443 方案可行。

## 配置 Caddy

Caddyfile 加一段：

```caddyfile
api.weirdsnap.top:8443 {
    reverse_proxy localhost:8765
}
```

然后放行 8443：

```bash
sudo ufw allow 8443/tcp
sudo systemctl reload caddy
```

验证：

```bash
curl https://api.weirdsnap.top:8443/api/status
```

返回：

```json
{"status": "ok", "version": "beta3", "uptime": 338.1}
```

成了。

## 前端改地址

把前端 `API_URL` 从协议相对地址改成带端口的：

```js
// 之前
const API_URL = '//api.weirdsnap.top/api';

// 之后
const API_URL = '//api.weirdsnap.top:8443/api';
```

推到 GitHub Pages，等部署生效，刷新页面，提问，正常返回。

## 这一团乱七八糟教会我什么

1. **GitHub Pages 的 HTTPS 是强制的**，前端调用 HTTP API 必被 Mixed Content 拦截。要么 API 也上 HTTPS，要么换 HTTP 部署。
2. **大陆云服务器 443 不一定能用**，但 80 有时候可以，非标准端口（8443）往往也能走。
3. **Caddy 真的省事**。自动证书、自动续期，配置一行反向代理就搞定。如果 80 端口能用来做 HTTP-01 验证，比 DNS 验证简单太多。
4. **Cloudflare Tunnel 不是唯一解**。它适合不想折腾端口的情况，但如果授权慢、网络抽风，直接上 Caddy + 非标准端口也是条路子。
5. **先测试，再假设**。我一直以为 80 被备案限制死了，实际 `curl` 一下才发现是通的，整个方案才定下来。

现在助手页能正常用了。下次再配这种前后端分离的玩具项目，应该能少踩几个坑。
