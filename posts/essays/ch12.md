# DeepSeek 思考模式：一个被误解的错误

今天在折腾 nanobot 接入 DeepSeek V4 的时候，遇到了一个奇奇怪怪的错误：

```
The reasoning_content in the thinking mode must be passed back to the API
```

第一反应是：DeepSeek API 有问题？还是 LiteLLM 转发的时候搞了什么鬼？或者是 nanobot 代码有 bug？

带着这三个疑问开始排查，最后发现——**都不是**。

## 错误的本质

这是 DeepSeek 的"思考模式"（Thinking Mode）搞的鬼。

思考模式是 DeepSeek 的一个高级特性：模型在给出最终回答前，会先输出一段思维链（`reasoning_content`），展示它是怎么想的。就像一个人先"想清楚"再"说出来"。

问题是这个 `reasoning_content` 有个特殊的管理规则：

- **如果你在对话中保留了它，必须在后续请求里**原样传回去**——这时它相当于模型"上一轮的思路"，必须连贯
- **如果你不需要它，可以从历史消息中删掉**——只要删干净，模型也不会抱怨

而报错恰恰是因为历史消息不完整：传了 `reasoning_content` 的一部分，或者传了但在某些场景下没传对。

## 实际测试

为了搞清楚到底怎么处理，我做了个对照实验：

| 场景 | 移除 reasoning_content | 保留 reasoning_content |
|------|----------------------|----------------------|
| 普通对话 | ✅ 正常 | ✅ 正常 |
| 使用工具调用后 | ✅ 正常 | ✅ 正常 |

**结论出乎意料：两种方式都正常。**

这说明这个问题在 DeepSeek V4 的实际表现里，**比文档描述的宽容得多**。不是"不移除就报错"，也不是"工具调用不移除就报错"——而是历史消息拼接出现更底层的问题（比如 LiteLLM 的 key 过滤逻辑把 `reasoning_content` 部分截断或格式转换出错）才会触发。

## 是什么触发了这个错误

后来定位到根因：**LiteLLM 的消息处理逻辑**。

LiteLLM 在处理请求消息时，会对消息键（message keys）做了一层过滤，只保留它认识的字段。DeepSeek 的 `reasoning_content` 不是标准的 OpenAI 协议字段，可能在过滤过程中被部分丢弃或格式异常。当 DeepSeek 收到这些"残缺"的历史消息时，就报了这个错。

解决方案是修改 LiteLLM 的消息键过滤白名单，加上 `reasoning_content`，或者直接通过 `extra_body` 参数原生透传。

## 一个更简单的方案：关掉思考模式

如果你不需要 DeepSeek 的思维链展示（对大多数应用场景来说确实不需要），直接在请求里禁掉就好：

```python
extra_body = {"thinking": {"type": "disabled"}}
```

这样 API 压根不会返回 `reasoning_content`，所有问题自然消失，而且响应还更快。

## 反思

这个问题的排查过程让我想到：

1. **先看自己的代码，别急着怪别人**——我一开始以为是 DeepSeek 或 LiteLLM 的 bug，结果根因在两者之间的兼容层
2. **有些"高级特性"不用反而更好**——思考模式听起来高大上，但对工具调用为主的场景来说，除了增加复杂性几乎没收益
3. **实测比文档靠谱**——DeepSeek 文档说必须怎么怎么处理，实测发现其实没那么严格

记录一下这次"错怪好人"的调试经历，下次遇到类似的 API 错误，可以更快地定位问题。

## 后记：v0.2.2 更新与新的发现

这篇文章发出去之后，我决定干脆把 nanobot 从 v0.1.4.post3 升级到 v0.2.2——新版本用原生 provider 取代了 LiteLLM，按理说 DeepSeek 的 `reasoning_content` 兼容问题应该一劳永逸解决了。

升级后的确如此：LiteLLM 的问题彻底没了，`reasoning_content` 不再被过滤或误处理。

但新的问题冒了出来：DeepSeek V4 在思考模式下，有时候返回的 `content` 是空字符串。排查发现不是 provider 的 bug——**这是思考模式的固有行为**。

DeepSeek 的思考模式把输出分成了两部分：

1. 先输出 `reasoning_content`（思考链）
2. 再输出 `content`（最终回答）

两者的 token 都从 `maxTokens` 预算里扣。如果思考过程太长把预算吃完了，模型还没来得及输出 content 就被截断（`finish_reason: length`），content 自然就是空的。

直调 API 验证：

| maxTokens | 简单请求 | 复杂请求 |
|-----------|---------|---------|
| 50 | ⚠️ thinking 吃完预算，content 空 | ⚠️ 同样问题 |
| 100 | ✅ 正常 | ⚠️ 偶尔 content 空 |
| 4096+ | ✅ 正常 | ✅ 正常 |

同时也发现，在之前争论的 maxTokens 问题上，我原来的 128K 确实太夸张了——思考模式用不了那么多 token。最终调成了 16384（16K），既给思考链留足了空间，回复也不会被截断。

到此为止，从 maxTokens 到思考模式到 LiteLLM 再到 v0.2.2，这个系列的折腾终于完整了：

- maxTokens 不是上下文窗口（8K → 128K 的教训）
- 思考模式需要在自己的 token 预算内规划（128K → 16K 的矫正）
- API 兼容层的问题要从源头解决（v0.2.2 原生 provider 的胜利）
