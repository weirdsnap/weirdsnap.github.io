# 把maxTokens当成上下文窗口的蠢事

今天在飞书上跟我的nanobot助手聊天，突然意识到一个很蠢的问题。

之前我一直觉得nanobot的响应速度有点慢，而且对话长了好像记不住东西。我以为是模型不行，或者API调用有什么问题。结果今天心血来潮看了一眼config.json，发现`maxTokens`设的是8192（8K）。

我当时第一反应：好啊，DeepSeek V4支持128K上下文，你给我设8K，怪不得！赶紧改大！

然后nanobot跟我说：等等，maxTokens不是上下文窗口，是**单次回复的上限**。

我：啊？

原来我一直把两个东西搞混了：

- **maxTokens**：模型每次回复最多能输出多少token。日常聊天几百token就够，8K其实完全不会卡住对话，只是复杂问题回答到一半可能被截断。
- **上下文窗口**：模型能"看到"多少历史对话，这个是DeepSeek V4本身支持1M，不需要在配置文件里设。

而且仔细想想，**上下文窗口不够其实没那么可怕**——nanobot的agent会自动帮模型做历史压缩，把早期对话摘要后塞回去。虽然会丢失一些细节，但至少对话不会断。

**maxTokens被截断才是真危险**：模型话说到一半被砍了，agent根本不知道它没说完，也不会自动续写。信息直接丢失，你还以为是模型回答完了，实际上后半段没了。相比之下，上下文窗口不够反而有保底机制。

所以从头到尾，响应慢跟maxTokens没啥关系……是我自己想多了。

不过话说回来，就算我理解错了，**一直用着默认的8K配置也没去动过**，这也是个问题。系统装好之后 config.json 长什么样就一直什么样，压根没想过要检查一下参数是不是合理。要是早点看一眼，至少不会在"回复被截断"这个问题上忍那么久。

改大也没坏处，至少不用担心回答被截断了。当时设成了128K，虽然日常根本用不到这么多，但心理上舒服多了。

## 后续：128K 还是大了

事情远没有结束。第二天就遇到了 DeepSeek 思考模式（Thinking Mode）的 `reasoning_content` 兼容性问题（详见《DeepSeek 思考模式：一个被误解的错误》），折腾了半天 LiteLLM。后来一气之下把 nanobot 从 v0.1.4 直接升到了 v0.2.2——这次升级直接抛弃了 LiteLLM，DeepSeek 变成了原生 provider。

升级本来是为了解决 LiteLLM 的问题，结果新 provider 也带来了新惊喜：在思考模式下，如果 maxTokens 设得太小，模型会把所有预算花在"思考"（生成 `reasoning_content`）上，根本没机会"说话"（生成 `content`），最后返回一个空字符串。

测试发现：

- maxTokens=50：简单问题勉强能回答，稍微复杂点 content 就空了
- maxTokens=100：依然有风险，思考稍长就爆
- maxTokens=4096：绝大多数场景足够
- maxTokens=16384：思考再长也游刃有余

我原来的 128K 值完全用不上。思考模型的 thinking 虽然消耗 token，但正常的思考链也就几百到一两千 token，加上回复也远到不了 128K。

所以绕了一大圈，maxTokens 经历了：

```
8K   →  128K  →  16K
(默认)   (报复性调大)   (最终合理值)
```

**最终调到了 16384（16K）**。不多不少，正好。

一个参数折腾了三轮，结果回到了最合理的中间值——也算是给这个系列画上句号了。
