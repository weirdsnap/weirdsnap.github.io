# GitHub Pages 不更新了：一次部署踩坑

今天给博客加了个游戏助手页面，前端推上去之后发现怎么刷新都不生效。一开始以为是浏览器缓存，清了好几遍，Safari 开发菜单都快被我点烂了，还是没变化。

用 `curl` 一查，线上文件居然是旧版：

```js
const API_URL = 'http://114.132.189.56:8765/api';
```

但本地仓库早就改成：

```js
const API_URL = 'https://api.weirdsnap.top:8443/api';
```

而且不仅是这个文件，`posts/index.json` 里新加的文章也没有。

## GitHub Pages 最后一次部署是 5 小时前

去仓库的 Actions 页面一看，`pages-build-deployment` workflow 最后一次跑是 5 小时前。也就是说，后面所有的 push 都没触发 Pages 构建。

这就很离谱。GitHub Pages 默认应该是 push 到部署分支就自动构建的，但它突然就停了。

我尝试了几个办法：

1. **清缓存、强制刷新** —— 没用，因为线上文件本身就是旧的
2. **加 query string 绕过 CDN** —— 没用，文件内容就是旧的
3. **手动切换 Settings → Pages 的 Source** —— 改成 None 再改回 master，理论上能强制重新绑定 Pages 构建，但总不能每次 push 都手动切吧

## 改用 GitHub Actions 部署

最终决定不用 GitHub 默认的那个黑盒 Pages 构建了，改成 GitHub Actions 自己部署。

加了一个 `.github/workflows/pages.yml`：

```yaml
name: Deploy to GitHub Pages

on:
  push:
    branches: [master, main]
  workflow_dispatch:

permissions:
  contents: read
  pages: write
  id-token: write

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
      - uses: actions/setup-python@v5
        with:
          python-version: '3.x'
      - run: python3 scripts/build_index.py
      - run: |
          if [ -n "$(git status --porcelain posts/index.json)" ]; then
            echo "Error: posts/index.json is out of date"
            exit 1
          fi
      - run: python3 -c "import json; json.load(open('posts/index.json'))"
      - run: python3 scripts/validate.py
      - uses: actions/upload-pages-artifact@v3
        with:
          path: .

  deploy:
    needs: build
    runs-on: ubuntu-latest
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    steps:
      - uses: actions/deploy-pages@v4
```

然后去 Settings → Pages 把 Source 从 **Deploy from a branch** 改成 **GitHub Actions**。

这样以后每次 push master，先跑 CI 检查，全部通过后再自动部署 Pages，不再依赖 GitHub 那个看不见的默认构建。

## CI 又失败了

改成 Actions 后的第一次 push，workflow 直接失败：

```
Error: posts/index.json is out of date. Run 'python3 scripts/build_index.py' and commit the result.
```

原因是我在另一个地方改了 `posts/ai/models/ch06.md`，内容字数从 2070 变到 3400，但忘了重新生成 `posts/index.json`。Blog CI 每次都会重新生成索引并比对，发现对不上就失败了。

这其实是个好习惯，只是我之前偷懒了。重新生成索引、提交、push，然后 workflow 全绿，Pages 也终于更新了。

## 总结

| 问题 | 原因 | 解决 |
|------|------|------|
| 线上页面不更新 | GitHub Pages 默认构建卡住，5 小时没跑 | 改用 GitHub Actions 部署 |
| 手动切 Source 麻烦 | 不能每次都手动触发 | Actions workflow 自动处理 |
| CI 失败 | 改了文章没重新生成 `index.json` | push 前运行 `build_index.py` |

这次踩坑让我意识到：**不要把部署交给不可控的黑盒机制**。GitHub Pages 默认构建虽然方便，但卡住的时候你根本不知道发生了什么。改成 Actions 之后，每一步都看得见、可控、还能加 CI 门禁，反而更靠谱。

以后改文章记住两件事：

1. 跑 `python3 scripts/build_index.py`
2. 把 `posts/index.json` 一起 commit push

CI 通过 → Pages 自动部署，完事。
