# MoE 与多模态概览

前面的模型基础文章讲了 Transformer 的核心组件：Attention、位置编码、FFN、激活函数和归一化。这一章我们看两个更进阶的方向：

- **MoE（Mixture of Experts）**：用"条件计算"把模型做大，但每次只激活一部分参数
- **多模态模型**：让大模型不仅能处理文本，还能看图像、听音频

这两个方向都是当前大模型发展的重要趋势，也对推理部署提出了新的挑战。

---

## 1. 为什么需要 MoE？

大模型的能力往往和参数量正相关。但参数越多，训练和推理成本越高。

MoE 的核心思想是：**把模型做大，但不让所有参数都参与每次计算**。

```
传统稠密模型：
  输入 → 所有层参与计算 → 输出
  参数量 = 激活参数量

MoE 稀疏模型：
  输入 → 路由选择少数专家 → 只有被选中的专家参与计算
  总参数量 >> 激活参数量
```

这样可以用更少的计算量获得更大的模型容量。

---

## 2. MoE 的基本结构

MoE 层通常替换 Transformer 中的 FFN 层。一个 MoE 层包含：

```
1. 多个专家网络（Experts）：每个专家本质上是一个小 FFN
2. 一个路由网络（Router / Gate）：决定输入应该由哪些专家处理
```

```python
# 简化示意
class MoELayer(nn.Module):
    def __init__(self, d_model, num_experts, top_k):
        super().__init__()
        self.experts = nn.ModuleList([
            nn.Linear(d_model, d_model) for _ in range(num_experts)
        ])
        self.router = nn.Linear(d_model, num_experts)
        self.top_k = top_k

    def forward(self, x):
        # 路由分数
        router_logits = self.router(x)  # [batch, seq, num_experts]
        weights, selected_experts = torch.topk(
            torch.softmax(router_logits, dim=-1),
            self.top_k,
            dim=-1
        )  # weights / selected_experts: [batch, seq, top_k]

        # 只计算 top-k 个专家的输出，按对应权重加权求和
        output = torch.zeros_like(x)
        for k in range(self.top_k):
            expert_idx = selected_experts[..., k]      # [batch, seq]
            expert_weight = weights[..., k:k+1]        # [batch, seq, 1]
            for i, expert in enumerate(self.experts):
                mask = (expert_idx == i)
                if mask.any():
                    output[mask] += expert(x[mask]) * expert_weight[mask]

        return output
```

### 2.1 关键超参数

| 参数 | 含义 | 典型值 |
|------|------|--------|
| `num_experts` | 专家总数 | 8, 16, 64, 128 |
| `top_k` | 每个 token 激活的专家数 | 1, 2, 8 |
| `d_model` | 模型维度 | 4096, 8192 |

总参数量 ≈ `num_experts × 单个专家参数量`。

激活参数量 ≈ `top_k × 单个专家参数量`。

---

## 3. 路由机制

路由是 MoE 的核心。好的路由应该：

1. 把 token 分配给最合适的专家
2. 保持各专家负载均衡
3. 计算开销小

### 3.1 Top-K 路由

最常用。对输入计算每个专家的分数，选分数最高的 K 个。

```
router_logits = x @ W_router  # 每个专家的分数
probs = softmax(router_logits)
top_k_probs, top_k_indices = topk(probs, k)
```

### 3.2 负载均衡问题

如果路由总是把 token 分配给少数几个"热门"专家，其他专家就会被浪费。

这会导致：
- 训练不稳定
- GPU 负载不均衡
- 模型容量没有被充分利用

解决方法是给路由损失加上**负载均衡损失（Load Balancing Loss）**：

```
loss_total = loss_task + α × loss_load_balance
```

负载均衡损失鼓励每个专家处理的 token 数量大致相等。

## 4. MoE 对训练的影响

### 4.1 训练通信量大

MoE 的专家通常分布在不同 GPU 上。一个 token 要交给哪个专家处理，就需要把数据传到对应的 GPU 上。

```
GPU 0: token A → 路由 → 专家 3 在 GPU 2 → 传数据到 GPU 2
GPU 1: token B → 路由 → 专家 5 在 GPU 3 → 传数据到 GPU 3
```

这被称为**All-to-All 通信**。当专家数量多、模型大时，通信开销会非常大。

### 4.2 需要更多显存

虽然每次只激活部分参数，但所有专家的权重都要加载到显存里。

```
稠密 70B 模型：推理时显存占用 ≈ 140 GB（FP16）
MoE 8×7B 模型：总参数量 56B，但所有专家权重都要存 ≈ 112 GB（FP16）
```

所以 MoE 模型虽然激活参数少，但显存占用不一定少。

---

## 5. MoE 对推理的影响

### 5.1 吞吐量 vs 延迟

MoE 在训练时可以降低计算量，但在推理时的收益取决于 batch size：

- **大 batch**：多个 token 可能命中同一专家，可以批量计算，吞吐高
- **小 batch / 单用户**：token 分散到不同专家，每次只算一点点，GPU 利用率低，延迟可能反而更高

### 5.2 专家并行

大规模 MoE 通常用**专家并行（Expert Parallelism, EP）**：每个 GPU 只存一部分专家。

```
GPU 0: 专家 0, 1
GPU 1: 专家 2, 3
GPU 2: 专家 4, 5
GPU 3: 专家 6, 7
```

路由决定 token 去哪个 GPU，通信量随 top_k 和 batch size 增加。

### 5.3 推理优化方向

- **专家缓存**：把热门专家常驻显存
- **动态批处理**：尽量把要去同一个专家的 token 拼在一起
- **稀疏 Attention**：和 MoE 结合进一步减少计算

## 6. 主流 MoE 模型

| 模型 | 架构 | 专家数 | top_k | 特点 |
|------|------|--------|-------|------|
| **Switch Transformer** | T5 + MoE | 2048 | 1 | 第一个大规模 MoE，每个 token 只路由到 1 个专家（top-1） |
| **Mixtral 8×7B** | Llama + MoE | 8 | 2 | 总参数量 46.7B，激活 12.9B |
| **Mixtral 8×22B** | Llama + MoE | 8 | 2 | 总参数量 141B，激活 39B |
| **Grok-1** | MoE | 8 | 2 | 314B 总参数 |
| **DeepSeek-V2/V3** | MLA + MoE | 64~256 | 6 | 高效 KV Cache + 大量专家 |
| **Qwen2.5-MoE** | Qwen + MoE | - | - | 阿里云 MoE 模型 |

Mixtral 8×7B 是最经典的例子：总参数量接近 47B，但每次只激活约 13B，推理速度接近 13B 稠密模型，但效果接近 70B 稠密模型。

## 7. 多模态基础

### 7.1 什么是多模态模型

多模态模型能同时处理两种或多种类型的数据，比如：

- 文本 + 图像（最常见）
- 文本 + 音频
- 文本 + 视频
- 文本 + 图像 + 音频

典型代表：

```
GPT-4V / Claude 3：图像理解
LLaVA：开源视觉语言模型
Qwen-VL：阿里视觉语言模型
Whisper：语音转文本
```

### 7.2 常见架构

多模态模型通常由几个部分组成：

```
图像编码器（Vision Encoder）：把图像变成视觉 token
投影层（Projection）：对齐视觉和文本特征空间
大语言模型（LLM）：处理拼接后的文本 + 视觉 token
```

```
输入图像 → Vision Encoder → 视觉 token
输入文本 → Text Tokenizer → 文本 token
                          ↓
                    拼接 → LLM → 输出文本
```

### 7.3 视觉编码器

最常用的视觉编码器是 **ViT（Vision Transformer）**：

```
图像 [224×224×3] → 切分成 16×16 的 patch → 每个 patch 变成一个 token
```

一张 224×224 的图像会产生 `(224/16)^2 = 196` 个视觉 token。如果图像更大，token 数会平方增长。

---

## 8. 多模态对推理的挑战

### 8.1 视觉 token 数量巨大

一张图可能产生几百甚至上千个视觉 token，直接拼进文本序列：

```
文本 prompt：50 tokens
一张图：196 tokens
总输入：246 tokens
```

如果一次发 4 张图：

```
总输入：50 + 4 × 196 = 834 tokens
```

Prefill 阶段的计算量和序列长度平方成正比，多图会显著增加延迟。

### 8.2 KV Cache 膨胀

视觉 token 也要存 KV Cache。一张图的 KV Cache 可能比整个文本 prompt 还大。

```
196 视觉 token × 隐藏维度 × 层数 × 2（K+V）
```

### 8.3 图像预处理复杂

推理时需要：
- 调整图像尺寸
- 归一化
- 切 patch
- 可能还要处理多图、图文交错

这些预处理步骤需要额外的工程支持。

### 8.4 不同模态的量化难度不同

视觉编码器通常对量化更敏感。图像特征的微小损失可能影响理解效果。

---

## 9. 一句话总结

> MoE 让大模型在不增加每次计算量的前提下扩大容量，代价是显存和通信；多模态让模型能处理图像、音频等信息，代价是视觉 token 带来的 Prefill 和 KV Cache 压力。两者都是当前大模型前沿，也是推理系统需要重点优化的方向。
