# Embedding 模型：单语言与多语言的核心差异

## 1. 前言

最近在做一个小项目——一个空洞骑士的中文 RAG 助手。用户用中文提问，RAG 去知识库里检索相关信息，然后 LLM 组织回答。

听起来很简单对吧？但有个问题卡了我整整两天：**中文提问全搜不到**。

无论搜什么——"螳螂领主"、"苍白之王"、"国王之道"——返回结果都是同一个垃圾 fragment。但换成英文 "Mantis Lords"、"Pale King"，瞬间搜到正确文档。数据明明是对的，就是检索链路断了。

问题出在 Embedding 模型上。

## 2. Embedding 是什么？

先简单回顾一下。Embedding 模型把文本变成向量（一串浮点数），语义相似的文本在向量空间中距离更近。

```
"猫" → [0.12, -0.34, 0.56, ...] (384维向量)
"狗" → [0.11, -0.30, 0.52, ...] (距离近 ✓)
"量子力学" → [0.89, 0.23, 0.11, ...] (距离远 ✓)
```

RAG 的工作流程是：

1. 把知识库所有文档提前转成向量，存进向量库
2. 用户提问时，把问题也转成向量
3. 在向量库里找最相似的 K 个文档
4. 把这些文档扔给 LLM 生成回答

**关键**：问题和文档必须在**同一个语义空间**里。如果模型的语义空间里根本没有"中文"这个维度，那中文问题就变成了噪声，命中率直接为零。

## 3. 问题的根因：两个 Embedding 模型

### bge-small-en-v1.5（英文专用）

这是 BAAI 出品的小型英文 Embedding 模型，只有 33M 参数，磁盘占用 ~33MB。

**训练数据**：纯英文语料（维基百科、Reddit、科学论文等）
**词汇表**：只有英文 token（约 3 万），模型压根不认识中文字符

当我输入中文时，实际上发生了什么？

```
输入: "螳螂领主"
tokenizer 视角: [æ, ŋ, ̂, ...] → 一套连模型自己都不理解的 Unicode 序列
embedding 结果: [0.001, -0.002, 0.001, ...] → 接近零向量的噪声
```

模型把中文当成了一串陌生的字节模式，找不到任何语义规律，只能输出一个"我不知道这是什么"的模糊向量。这在实践中表现为：**任何中文句子 embedding 出来的向量都差不多**，全部挤在向量空间的一个小角落里。

我验证的对比数据：

| Query | 英文检索结果 | 中文检索结果 |
|-------|------------|------------|
| Mantis Lords / 螳螂领主 | ✅ 正确找到 Mantis Lords | ❌ 返回 Vengeful Spirit |
| Pale King / 苍白之王 | ✅ 正确找到 Pale King | ❌ 返回 Vengeful Spirit |
| Fury of the Fallen / 亡者之怒 | ✅ 正确找到护符 | ❌ 返回 Vengeful Spirit |
| King's Pass charms / 国王之道 护符 | ✅ 正确找到相关护符 | ❌ 返回 Vengeful Spirit |

不管搜什么中文，返回的都是同一个最短的垃圾 chunk——因为所有中文 embedding 都在同一片噪声区，而那个短 chunk 恰好最接近噪声中心。

### paraphrase-multilingual-MiniLM-L12-v2（多语言）

这是 Sentence-BERT 系列的多语言版本，118M 参数，磁盘占用 ~470MB（14 倍于 bge-small-en）。

**训练数据**：涵盖 50+ 种语言的平行语料，每句话有中文、英文、日文等多语言翻译配对
**核心训练目标**：cross-lingual alignment——让"螳螂领主"和"Mantis Lords"在向量空间中尽可能靠近

```
英文模型的空间布局：
  [Mantis Lords]         ← 英文正确位置
  [螳螂领主]             ← 乱码噪声，飘在远处
  距离: 非常远 → 检索失败 (0%)

多语言模型的空间布局：
  [Mantis Lords]     ← 英文位置
  [螳螂领主]         ← 中文位置（隔壁）
  距离: 非常近 → 检索成功 ✓
```

打个比方：**bge-small-en 是个只懂英语的图书管理员**。你用中文问他"螳螂领主在哪"，他听不懂，只能随手抽一本书递给你。**paraphrase-multilingual 是个懂 50 种语言的管理员**，你用中文问、用英文问、用日文问，他都能把你带到正确的书架前。

## 4. 核心差异对比

### 4.1 模型参数与性能

| 维度 | bge-small-en-v1.5 | paraphrase-multilingual-MiniLM-L12-v2 |
|------|-------------------|--------------------------------------|
| 参数量 | 33M | 118M |
| 磁盘大小 | ~33 MB | ~470 MB |
| 向量维度 | 384 | 384 ✅ |
| 推理速度 | 快 | 慢 2-3x |
| 语言支持 | 仅英文 | 50+ 语言 |

两者都是 384 维度，这个没变——向量库的 FAISS 索引结构不需要改。

### 4.2 语言覆盖

英文模型只认识英文 token（约 3 万词汇），中文对它来说是"乱码"。多语言模型的词汇表扩展到数十万 token，覆盖了中文、日文、韩文、阿拉伯文等主要语言的常用字符——这些 token 在预训练阶段就学会了语义映射。

### 4.3 Cross-lingual Alignment

这是最核心的区别。多语言模型在训练时使用了平行语料：

```
"今天天气真好" ↔ "The weather is nice today"
```

模型被训练让这两个句子的 embedding 尽可能相似。经过数十亿对这样的训练后，模型的语义空间被"拉伸"成一种多语言对齐的结构——不同语言中表达相同意义的句子，在向量空间中落在附近的位置。

英文模型完全没有这个对齐过程，中文对它来说就是"never seen before"的数据。

### 4.4 中文支持的质量上限

多语言模型的中文能力不是无限的。paraphrase-multilingual 的中文训练语料占比大概 10-15%，所以：

- **日常表达**（"怎么去白色宫殿"、"螳螂领主在哪"）→ 效果很好 ✓
- **游戏专有名词**（"虚空之光"对应 "Voidlight"、"梦之盾"对应 "Dreamshield"）→ 如果中文名在语料中不常见，可能匹配不够精准，需要混合检索策略辅助
- **英文搜索**仍然正常（多语言模型也包含英文）

## 5. 实践中的切换经验

切换模型时要注意的问题：

### 5.1 必须重建向量库

旧的向量库是用英文模型生成的，其中的向量空间结构完全不能复用。必须：

```
rm -rf vectorstore/
python scripts/ingest.py
```

我第一次就踩了坑，以为只是换个模型的事——结果旧向量库依然用英文模型的维度空间，中文检索自然还是崩。

### 5.2 模型下载

多语言模型 ~470MB，从 HuggingFace 下载在国内需要镜像：

```bash
export HF_ENDPOINT=https://hf-mirror.com
```

首次下载慢，但会缓存到 `~/.cache/fastembed/`，之后秒加载。

### 5.3 版本控制

向量库通常是二进制文件（FAISS 索引），不适合进 Git。而代码层面的模型名称配置应该进 Git：

```python
# config.py
FASTEMBED_MODEL = os.getenv("FASTEMBED_MODEL", "sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2")
```

这样队友 `git pull` 后只需在自己的环境里重建向量库就能用。

## 6. 总结

| 场景 | 英文模型 | 多语言模型 |
|------|---------|-----------|
| 纯英文 RAG | ✅ 够用，还更快 | ✅ 可以但略慢 |
| 多语言 RAG | ❌ 中文检索为零 | ✅ 中文英文都能用 |
| 项目规模 | 小、个人项目 | 需要 ~470MB 模型 |
| 精确匹配 | 英文好 | 中文够用，英文略差 |

如果你的 RAG 系统需要支持中文（或其他非英文语言），**不要试图用英文 Embedding 模型凑合**。这不是调参能解决的问题——模型根本不认识那些字符。直接上多语言模型，一步到位。

不然你查 "亡者之怒"，它给你返回 "Vengeful Spirit 的备注栏"——那个下午我 debug 得很痛苦。
