# 分布式文件系统与 Checkpoint 加载

前面五章我们主要聊了网络：GPU 之间怎么通信、用什么网络、集群拓扑怎么设计。但网络只是基础设施的一部分。

在大规模 AI 训练中，另一个经常成为瓶颈的是**存储**：

- 训练数据集动辄几十 TB 甚至 PB 级，存在哪？
- 几千张 GPU 同时读取数据，怎么保证不抢带宽？
- 模型训练几周，中间 checkpoint 怎么写才能既快又可靠？
- 训练中断后恢复，checkpoint 加载会不会成为启动瓶颈？

这些问题都需要分布式文件系统来解决。

---

## 1. AI 训练对存储的需求

### 1.1 数据量大

- GPT 类预训练：数据集通常 **1TB - 100TB**
- 多模态训练：图片、视频数据更大，**100TB - PB 级**
- 这些数据无法全部放进单台服务器的本地 SSD

### 1.2 并发读取高

- 数据并行场景下，每张 GPU 都要持续读取数据
- 1000 张 GPU 同时读，存储系统需要提供 **TB/s 级聚合吞吐**

### 1.3 Checkpoint 写入突发

- 训练过程中定期保存 checkpoint（比如每 1 小时或每 1000 步）
- 一个 70B 模型的 FP16 checkpoint 大约 **140GB**
- 如果是 3D 并行，所有 rank 同时写自己的分片，写入量巨大

### 1.4 加载性能影响恢复时间

- 训练中断后重启，需要从 checkpoint 恢复
- 如果 checkpoint 存在慢存储上，加载时间可能长达几十分钟

## 2. 常见分布式文件系统

### 2.1 Lustre

Lustre 是高性能计算（HPC）领域最常用的并行文件系统之一，也是很多早期 AI 集群的选择。

```
Client ── Network ── MDS (Metadata Server)
                 └── OSS (Object Storage Server) × N
                        └── OST (Object Storage Target)
```

- **MDS**：管理文件元数据（目录、权限、文件位置）
- **OSS/OST**：管理实际数据存储
- 支持 **POSIX 接口**，对应用透明
- 可以水平扩展 OSS 数量来提升聚合带宽

**优点**：成熟、高吞吐、POSIX 兼容、生态丰富
**缺点**：元数据可能成为瓶颈，小文件性能差，运维复杂

### 2.2 GPFS / IBM Spectrum Scale

IBM 的 GPFS（现在叫 Spectrum Scale）是另一个企业级并行文件系统。

- 同样采用元数据和数据分离架构
- 支持 **GPFS Native RAID** 和多种后端存储
- 在企业级 AI 集群中使用较多

**优点**：稳定、企业级支持强
**缺点**：成本高，部署复杂

### 2.3 BeeGFS

BeeGFS 是一个开源的并行文件系统，设计上更注重易部署和高性能。

- 组件包括 Management、Metadata、Storage、Client
- 支持 NVMe、SSD、HDD 分层存储
- 在中小规模 AI 集群中越来越流行

**优点**：开源、易部署、性能好
**缺点**：大规模部署经验相对 Lustre 较少

### 2.4 WekaFS / VAST Data

这些是新兴的商业并行文件系统，专门为 AI 工作负载优化。

- **WekaFS**：基于 NVMe 的全闪存并行文件系统，延迟极低
- **VAST Data**：使用QLC闪存+软件优化，提供高吞吐低成本方案

**优点**：为 AI 优化，性能高
**缺点**：商业软件，成本高

## 3. 对象存储与数据湖

### 3.1 对象存储的角色

除了并行文件系统，对象存储（如 S3、MinIO、Ceph）也常用于 AI 训练：

- **数据集归档**：冷数据放在对象存储，成本低
- **训练前预处理**：原始数据从对象存储读取，处理后写入并行文件系统
- **Checkpoint 长期保存**：历史 checkpoint 归档到对象存储

### 3.2 S3 的问题

直接用 S3 作为训练数据源，通常性能不够：

- S3 高延迟（毫秒级），不适合每张 GPU 频繁小读取
- 没有 POSIX 语义，很多训练框架需要额外适配

所以常见做法是：**对象存储存原始数据，并行文件系统存训练用数据**。

```
Raw Data (S3/MinIO)
    ↓
Preprocessing
    ↓
Training Data (Lustre/BeeGFS)
    ↓
GPU Training
```

## 4. Checkpoint 保存与加载

### 4.1 Checkpoint 里存什么

一个典型的训练 checkpoint 包括：

- 模型权重（model state dict）
- 优化器状态（optimizer state）
- 学习率调度器状态
- 随机数种子和训练步数
- 数据加载器位置（避免重复训练已看过的数据）

### 4.2 保存策略

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| **同步保存** | 所有 rank 同时写 | 简单但容易造成 I/O 突发 |
| **异步保存** | 主进程写，其他继续训练 | 减少训练中断，但实现复杂 |
| **仅 rank 0 保存** | 只有主进程写权重 | 简单，但 3D 并行时不够 |
| **分片保存** | 每个 rank 写自己的分片 | 3D 并行常用 |

### 4.3 加载优化

加载 checkpoint 时，常见瓶颈：

- 所有 rank 同时读同一个大文件，存储带宽被抢
- checkpoint 需要先落到本地 SSD，再从本地读到 GPU
- 元数据操作（列目录、查文件）太慢

优化方法：

- **分片加载**：每个 rank 只读自己需要的分片
- **本地缓存**：把 checkpoint 预加载到计算节点的本地 NVMe SSD
- **并行读取**：利用多个存储节点和多个网络链路同时读取
- **定期保存到本地**：训练中间状态优先写本地，异步同步到共享存储

## 5. 存储性能指标

评估存储系统是否满足 AI 训练需求，主要看三个指标：

| 指标 | 含义 | AI 训练关注点 |
|------|------|--------------|
| **吞吐（Throughput）** | 单位时间能读多少数据 | 大数据集顺序读取、checkpoint 加载 |
| **IOPS** | 每秒能处理多少 IO 请求 | 小文件随机读取 |
| **延迟（Latency）** | 单次请求响应时间 | 影响 GPU 等待数据的空转时间 |

对于 AI 训练：

- 数据加载：**高吞吐**最重要
- checkpoint 保存：**高吞吐 + 突发写入能力**
- 小文件元数据操作：**高 IOPS** 和 低延迟元数据服务

---

## 6. 小结

- AI 训练需要**高吞吐、高并发、大容量**的存储系统
- **Lustre、GPFS、BeeGFS** 是主流并行文件系统选择
- **对象存储**适合存原始数据和归档，但通常不直接用于训练读取
- **Checkpoint** 的保存和加载需要专门优化，避免成为训练瓶颈
- 评估存储性能看 **吞吐、IOPS、延迟** 三个维度
