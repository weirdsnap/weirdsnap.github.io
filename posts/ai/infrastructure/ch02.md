# InfiniBand 与 RoCEv2：两种 RDMA 网络实现

上一章我们聊了 RDMA 的核心思想：**让网卡直接读写远端内存，绕过 CPU 和操作系统内核**。但 RDMA 本质上是一组能力，不是某一种具体的网络技术。它必须运行在物理网络之上。

目前业界主流的 RDMA 网络实现有两种：

- **InfiniBand（IB）**：专为 HPC 和 AI 设计的专用网络
- **RoCEv2（RDMA over Converged Ethernet v2）**：在标准以太网上跑 RDMA

它们都能提供微秒级延迟和百 Gbps 级带宽，但设计理念、生态和适用场景差别很大。选型时一旦选错，后期改造成本极高。

---

## 1. InfiniBand：为 AI 而生的专用网络

### 1.1 完整的专用协议栈

InfiniBand 不是"在以太网上跑一个协议"，而是从物理层、链路层到传输层都重新设计的网络技术。它的协议栈完全独立于 TCP/IP。

```
应用层（ verbs / MPI / NCCL ）
    ↓
传输层（ RC / UD / RD ）
    ↓
网络层（ 子网管理、路由 ）
    ↓
链路层（ 链路编码、流控、QoS ）
    ↓
物理层（ IB 线缆、光模块、网卡、交换机 ）
```

这种垂直整合带来一个显著好处：**每一层都为低延迟、高吞吐、高可靠性优化**，不需要兼容传统以太网的复杂历史包袱。

### 1.2 关键特性

| 特性 | 说明 |
|------|------|
| **极低延迟** | 端到端延迟约 1~2 μs，比 RoCEv2 通常低 20~50% |
| **高带宽** | 目前 NDR 400Gbps，XDR 800Gbps 已商用 |
| **子网管理器（Subnet Manager）** | 集中管理网络拓扑、路由和拥塞控制，简化大规模集群运维 |
| **原生 RDMA 支持** | 网卡、交换机、协议栈都原生支持 RDMA，无需额外配置 |
| **无损网络** | 链路层内置基于 credit 的流控，避免丢包 |

### 1.3 主要生态

InfiniBand 的硬件生态基本被 **NVIDIA（通过收购 Mellanox）** 主导：

- **网卡**：ConnectX-6 / ConnectX-7 / ConnectX-8
- **交换机**：QM9700 / QM9790 系列
- **线缆**：DAC 铜缆、AOC 有源光缆

在 NVIDIA 的 DGX  SuperPOD 方案中，InfiniBand 是默认网络选择。

### 1.4 适用场景

- 超大规模 AI 训练集群（千卡、万卡级）
- 对延迟极度敏感的 HPC 应用
- 预算充足、希望"开箱即用"的场景

主要缺点：**贵**。专用交换机、专用线缆、专用网卡，整体成本通常比 RoCEv2 高 30%~100%。

## 2. RoCEv2：在以太网上跑 RDMA

### 2.1 为什么需要 RoCE？

很多数据中心已经部署了大量以太网交换机和线缆。如果为了 RDMA 全部换成 InfiniBand，成本太高、风险太大。

RoCE 的想法是：**能不能在现有以太网基础设施上，也能跑 RDMA？**

- **RoCEv1**：只在二层以太网上跑 RDMA，无法跨三层路由，实际应用很少
- **RoCEv2**：基于 UDP/IP，可以跨子网、跨路由，成为实际主流

### 2.2 RoCEv2 协议栈

```
应用层（ verbs / NCCL ）
    ↓
RDMA 层
    ↓
UDP + IP
    ↓
以太网 MAC + PHY
```

RoCEv2 把 RDMA 数据封装在 UDP 包里，通过 IP 路由。这意味着它可以跑在标准以太网交换机上。

### 2.3 关键挑战：以太网会丢包

RDMA 对丢包非常敏感。传统 TCP 丢包后可以重传，但 RDMA 的 kernel bypass 设计让重传代价很高。一旦丢包，性能会断崖式下跌。

以太网本身不是无损网络，所以 RoCEv2 必须依赖两套机制来避免丢包：

#### 2.3.1 PFC（Priority Flow Control，优先级流控）

PFC 是 IEEE 802.1Qbb 标准。当交换机端口缓冲区即将满时，向上游设备发送"暂停"帧，让上游暂时停止发送特定优先级的流量。

```
接收端交换机缓冲区快满
        ↓
发送 PFC 暂停帧给上游
        ↓
上游网卡/交换机暂停该优先级流量
        ↓
缓冲区压力缓解后发送恢复帧
```

PFC 能把丢包控制在链路层，但有一个副作用：** pause 帧可能向上游级联传播**，导致网络中出现"反压风暴"，甚至阻塞整个网络。

#### 2.3.2 ECN（Explicit Congestion Notification，显式拥塞通知）

ECN 是 IP 头和 TCP/UDP 选项中的一个标志位。当交换机检测到拥塞时，在通过的包上标记 ECN 位。接收端网卡收到后，通过拥塞通知包（CNP）告诉发送端降低速率。

```
交换机拥塞 → 标记 ECN
    ↓
接收端网卡收到 ECN 包
    ↓
发送 CNP 给发送端
    ↓
发送端降低注入速率
```

ECN 比 PFC 更"温和"，是一种端到端的拥塞控制信号，不会像 PFC 那样引发级联暂停。

实际部署中，通常**PFC + ECN 一起用**：PFC 做最后一道防线防止丢包，ECN 做主要的拥塞控制。

### 2.4 配置复杂度

RoCEv2 的部署配置比 InfiniBand 复杂得多：

- 交换机要配置 PFC、ECN、DSCP、队列调度
- 网卡要配置 RoCE 版本、PFC 优先级、ECN 阈值
- 网络中要划分无损队列，避免普通 TCP 流量抢占 RoCE 流量
- 需要监控丢包、PFC 暂停次数、ECN 标记率

一句话：**RoCEv2 省钱但费人**。

### 2.5 适用场景

- 已有大量以太网设备的数据中心
- 预算有限的 AI 训练/推理集群
- 规模不是特别大（几十到几百卡）的场景

## 3. InfiniBand vs RoCEv2 对比

| 维度 | InfiniBand | RoCEv2 |
|------|-----------|--------|
| **协议栈** | 专用，从物理层到传输层 | 基于标准 UDP/IP + 以太网 |
| **延迟** | ~1-2 μs | ~2-5 μs |
| **带宽** | NDR 400G / XDR 800G | 100G / 200G / 400G / 800G（取决于以太网设备） |
| **无损网络** | 原生支持 | 依赖 PFC + ECN 配置 |
| **交换机** | 专用 IB 交换机 | 标准以太网交换机（需支持 PFC/ECN） |
| **网卡** | IB 网卡（如 ConnectX-7 IB） | 支持 RoCE 的以太网卡 |
| **管理复杂度** | 低，子网管理器集中管理 | 高，需要精细化网络调优 |
| **成本** | 高 | 较低 |
| **生态开放性** | 相对封闭，NVIDIA 主导 | 开放，多厂商支持 |
| **典型规模** | 千卡、万卡级 | 几十到几百卡 |

## 4. 选型建议

### 选 InfiniBand，如果：

- 你在建一个 **千卡以上** 的超大规模训练集群
- 对 **延迟极度敏感**，比如某些强化学习训练、大规模 MoE 训练
- 预算充足，想要**开箱即用**的体验
- 你已经在用 NVIDIA DGX / SuperPOD 方案

### 选 RoCEv2，如果：

- 你已有成熟的 **以太网数据中心**
- 集群规模在 **几百卡以内**
- 预算有限，希望复用现有网络设备
- 团队有足够网络运维能力调优 PFC/ECN

### 一个常见误区

> "RoCEv2 比 InfiniBand 差很多，所以大规模训练必须用 IB。"

不完全对。在 **千卡以内** 的场景，调优良好的 RoCEv2 网络性能可以非常接近 InfiniBand。真正拉开差距的是：

- 万卡级集群的网络拓扑复杂度
- 团队的网络调优能力
- 对延迟和稳定性的极致要求

很多云厂商和大型企业的 AI 集群，实际上是在 RoCEv2 上跑的。

---

## 5. 小结

- **RDMA 是能力，IB 和 RoCEv2 是两种不同的实现路径**
- **InfiniBand** 是专用网络，延迟最低、管理最简单、成本最高
- **RoCEv2** 基于以太网，成本低、生态开放，但需要精细配置 PFC/ECN 才能保证无损传输
- 选型没有绝对答案，关键看 **规模、预算、团队能力** 三个因素
