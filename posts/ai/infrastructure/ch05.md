# AI 数据中心网络架构

前面四章我们聊了单机内的 GPU 互联（NVLink/NVSwitch）、多机之间的 RDMA 能力、InfiniBand/RoCEv2 网络实现，以及上层的 NCCL 集合通信库。

但当集群规模从几台机器扩展到几百台、几千台时，问题就不再是"两张 GPU 怎么通信"，而是：

> **整个数据中心的网络拓扑，能不能让所有 GPU 之间都保持高带宽、低延迟、不拥塞的通信？**

这就是 AI 数据中心网络架构要解决的问题。

---

## 1. AI 集群的网络需求

### 1.1 和普通数据中心的区别

普通数据中心（比如Web服务）的流量特征是：

- 大量南北向流量（用户请求进来，响应出去）
- 东西向流量相对较小，且多为请求-响应式
- 对延迟敏感的是尾延迟（p99），不是每一跳的微秒级延迟

AI 训练集群完全不同：

- **东西向流量占主导**：每次迭代都要 AllReduce 整个模型的梯度
- **流量巨大且突发**：几十 GB 甚至上百 GB 的数据要在毫秒级同步完
- **全连接通信**：任意两个 GPU 之间都可能需要通信
- **对延迟和带宽都极度敏感**：网络慢一点，GPU 就空等

### 1.2 关键指标

设计 AI 集群网络时，主要关注三个指标：

| 指标 | 含义 | 典型要求 |
|------|------|---------|
| **带宽** | 单端口和全网可用带宽 | 单端口 100G/200G/400G，全网无阻塞或低收敛 |
| **延迟** | 端到端通信延迟 | 微秒级，跨机 < 10 μs |
| **直径** | 任意两节点间最多经过几跳交换机 | 越小越好，通常 2-3 跳 |

## 2. Fat-Tree：经典的无阻塞网络拓扑

### 2.1 为什么叫 Fat-Tree

Fat-Tree 最早用于超级计算机，后被广泛应用于 AI 集群。它的核心思想是：

> **越靠近根节点，链路越"胖"（带宽越大），从而避免传统树形网络的根节点瓶颈。**

在传统树形网络中，所有叶子节点的流量都要经过根节点，根节点很容易成为瓶颈。Fat-Tree 通过**增加上层链路的带宽和交换机数量**，让网络接近无阻塞。

### 2.2 三层 Fat-Tree 结构

一个典型的三层 Fat-Tree 包含：

- **Leaf（叶交换机）**：直接连接服务器
- **Spine（脊交换机）**：连接 Leaf
- **Core（核心交换机）**：连接 Spine（超大规模时才有）

```
        Core
       / | \
    Spine Spine Spine
    /|\  /|\  /|\
  Leaf ... Leaf ... Leaf
   |      |      |
 Node   Node   Node
```

### 2.3 无阻塞与收敛比

- **无阻塞（Non-blocking）**：任意输入端口到任意输出端口都能以线速通信，没有内部竞争
- **收敛比（Oversubscription）**：如果上行带宽小于下行带宽总和，就存在收敛

例如：一台 Leaf 交换机有 32 个 400G 下行端口连接服务器，但只有 16 个 400G 上行端口连 Spine，那么收敛比就是 2:1。

对于 AI 训练，通常希望收敛比尽量低，最好是 **1:1 无阻塞**。

### 2.4 Fat-Tree 的成本问题

Fat-Tree 接近无阻塞的代价是**交换机数量多**。一个 1:1 无阻塞的 Fat-Tree 需要大量上层交换机来避免收敛，Leaf 与 Spine 的比例取决于端口数和收敛比要求，高端口密度交换机很贵。

当集群规模达到万卡级时，纯 Fat-Tree 的成本会变得非常高。因此业界开始探索更高效的拓扑。

## 3. Rail-Optimized：为 AI 训练优化的拓扑

### 3.1 观察：GPU 之间并不是完全均匀通信

在典型的分布式训练中：

- **Tensor Parallelism（TP）**：只在单机内 8 卡之间通信，使用 NVLink
- **Data Parallelism（DP）**：当 DP 与 TP 组合时，跨节点的 DP 通信通常表现为同号 GPU 通信（GPU0 与 GPU0，GPU1 与 GPU1，等等）
- **Pipeline Parallelism（PP）**：按层切分的相邻节点之间通信

这意味着，**跨机的 DP 通信主要是"同 rail"通信**：同一台服务器里的第 1 张 GPU，主要和其他服务器里的第 1 张 GPU 通信。

### 3.2 Rail-Optimized 设计

Rail-Optimized 拓扑的核心思想是：

> **把同 rail 的 GPU（比如所有服务器的 GPU0）尽量用高带宽、少跳数的网络直连。**

```
Server 1: GPU0 GPU1 GPU2 GPU3 GPU4 GPU5 GPU6 GPU7
Server 2: GPU0 GPU1 GPU2 GPU3 GPU4 GPU5 GPU6 GPU7
Server 3: GPU0 GPU1 GPU2 GPU3 GPU4 GPU5 GPU6 GPU7
Server 4: GPU0 GPU1 GPU2 GPU3 GPU4 GPU5 GPU6 GPU7

Rail 0: 所有 GPU0 ── 直接连到同一组 Leaf/Spine
Rail 1: 所有 GPU1 ── 直接连到同一组 Leaf/Spine
...
```

这样，DP 的 AllReduce 通信主要发生在 rail 内部，路径更短、带宽更高、拥塞更少。

### 3.3 Rail-Optimized vs Fat-Tree

| 维度 | Fat-Tree | Rail-Optimized |
|------|---------|----------------|
| 设计目标 | 通用无阻塞 | 优化 AI 训练通信模式 |
| 交换机数量 | 多 | 相对较少 |
| 任意两节点通信 | 性能一致 | 同 rail 性能更好 |
| 适用场景 | 通用 HPC/AI | 大规模 AI 训练 |

Rail-Optimized 不是完全替代 Fat-Tree，而是在 AI 训练特定通信模式下做了优化。

## 4. NVIDIA Spectrum-X 与参考架构

### 4.1 Spectrum-X 是什么

NVIDIA 在 2023 年推出的 **Spectrum-X** 是一套面向生成式 AI 的以太网网络平台，专门针对 RoCEv2 做了优化。

它包含两部分：

- **Spectrum-4 交换机**：支持 RoCEv2、PFC、ECN、性能隔离
- **BlueField-3 DPU**：在网卡/ DPU 上做网络卸载和拥塞控制

### 4.2 Spectrum-X 的目标

让 RoCEv2 在以太网上达到接近 InfiniBand 的性能，同时保持以太网的开放生态。

```
GPU ── BlueField-3 DPU ── Spectrum-4 Switch ── 其他节点
         ↓                        ↓
      网络卸载                 拥塞控制 / 性能隔离
```

### 4.3 NVIDIA 参考架构

NVIDIA 为不同规模提供了参考架构：

- **DGX BasePOD**：小规模起步方案
- **DGX SuperPOD**：大规模训练集群，通常使用 InfiniBand
- **Spectrum-X 参考架构**：使用 RoCEv2 的以太网方案

这些参考架构不仅包括网络拓扑，还包括服务器配置、存储配置、软件栈版本、布线规范等。

## 5. 集群规模与网络设计

### 5.1 千卡级集群

- 通常使用 **Fat-Tree 或 2 层 Leaf-Spine 结构**
- 单端口 200G/400G
- 收敛比 1:1 或 2:1
- IB 或 RoCEv2 均可

### 5.2 万卡级集群

- 通常使用 **3 层 Leaf-Spine-Core 或 Rail-Optimized 拓扑**
- 单端口 400G/800G
- 必须 1:1 无阻塞或接近无阻塞
- 多轨道网络设计
- 通常使用 InfiniBand，部分使用优化过的 RoCEv2

### 5.3 计算/网络/存储配比

设计集群时，除了网络，还要考虑计算和存储的配比：

| 规模 | 计算节点 | 存储节点 | 网络端口 |
|------|---------|---------|---------|
| 256 卡 | 32 台 8 卡服务器 | 4-8 台 | 32-64 个 400G |
| 1024 卡 | 128 台 | 16-32 台 | 128-256 个 400G |
| 10000 卡 | 1250 台 | 100+ 台 | 1250+ 个 400G/800G |

存储节点数量取决于数据集大小和 checkpoint 频率。checkpoint 写入通常是突发性的，需要高吞吐存储。

## 6. 小结

- **AI 数据中心网络的核心挑战**：支撑大规模 GPU 之间的高带宽、低延迟、全连接通信
- **Fat-Tree** 是经典无阻塞拓扑，但交换机成本高
- **Rail-Optimized** 针对 AI 训练通信模式优化，减少同 rail GPU 之间的通信跳数
- **NVIDIA Spectrum-X** 是 RoCEv2 以太网方案的代表，试图在开放生态下接近 IB 性能
- 集群规模决定拓扑选择：千卡级可用 Fat-Tree，万卡级需要 Rail-Optimized 或多层无阻塞设计

网络架构讲完后，下一个大主题是**存储**：训练数据存在哪、checkpoint 怎么写、数据集怎么并行加载。这些同样会决定集群能不能跑满。
