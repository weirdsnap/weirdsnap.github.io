# NCCL：集合通信库

前面三章我们聊了 RDMA、InfiniBand/RoCEv2、NVLink/NVSwitch。这些都是底层网络或硬件能力。但对于写 AI 训练代码的工程师来说，通常不会直接操作 RDMA 的 verbs 接口，而是用一个更高层的库：

**NCCL（NVIDIA Collective Communications Library）**

NCCL 把多卡之间的通信抽象成几个简单的集合通信原语，比如 `AllReduce`、`AllGather`。它屏蔽了底层是 NVLink、PCIe 还是 InfiniBand 的差异，让上层代码可以专注于模型和算法。

---

## 1. 为什么需要 NCCL

### 1.1 没有 NCCL 的世界

写分布式训练代码时，每次同步梯度都要面对几个问题：

- 这些 GPU 是在同一台机器内，还是跨机器？
- 如果是同一台机器，走 NVLink 还是 PCIe？
- 如果是跨机器，走 InfiniBand 还是 RoCEv2？
- 数据怎么切分、怎么路由、怎么避免拥塞？

这显然不现实。不同硬件的编程接口完全不同，直接手写集合通信算法既复杂又容易出错。

### 1.2 NCCL 的作用

NCCL 是 NVIDIA 提供的**集合通信库**，它提供了一组标准的集合通信原语，并自动选择最优的通信路径和算法。

```
PyTorch / TensorFlow / JAX
        ↓
    c10d / XLA
        ↓
       NCCL
        ↓
   NVLink / IB / RoCE / PCIe
```

PyTorch 中的用法很简单：

```python
torch.distributed.all_reduce(tensor, op=torch.distributed.ReduceOp.SUM)
```

底层就会调用 NCCL 完成梯度同步。

## 2. NCCL 的核心原语

NCCL 实现了分布式训练中最常用的集合通信操作。理解这些原语，才能理解分布式训练的各种并行策略。

### 2.1 Broadcast

一张 GPU 上的数据广播给所有 GPU。

```
Before:              After:
GPU0: [A]            GPU0: [A]
GPU1: [-]    →       GPU1: [A]
GPU2: [-]            GPU2: [A]
GPU3: [-]            GPU3: [A]
```

典型用途：模型初始化时，把主进程的模型权重广播给所有进程。

### 2.2 Reduce

把所有 GPU 上的数据按某种操作（如求和、取最大）归约到一张 GPU 上。

```
Before:              After:
GPU0: [1]            GPU0: [10]
GPU1: [2]    →       GPU1: [-]
GPU2: [3]            GPU2: [-]
GPU3: [4]            GPU3: [-]
```

### 2.3 AllReduce

Reduce 的结果再广播给所有 GPU。这是**数据并行训练中最核心的操作**。

```
Before:              After:
GPU0: [g0]           GPU0: [g0+g1+g2+g3]
GPU1: [g1]   →       GPU1: [g0+g1+g2+g3]
GPU2: [g2]           GPU2: [g0+g1+g2+g3]
GPU3: [g3]           GPU3: [g0+g1+g2+g3]
```

在数据并行训练中，每张 GPU 计算本地梯度，然后通过 AllReduce 得到全局平均梯度。

### 2.4 AllGather

每张 GPU 持有数据的一部分，AllGather 后每张 GPU 都拥有完整的数据。

```
Before:              After:
GPU0: [A0]           GPU0: [A0|A1|A2|A3]
GPU1: [A1]   →       GPU1: [A0|A1|A2|A3]
GPU2: [A2]           GPU2: [A0|A1|A2|A3]
GPU3: [A3]           GPU3: [A0|A1|A2|A3]
```

典型用途：模型并行（Tensor Parallelism / Pipeline Parallelism）中收集中间结果。

### 2.5 ReduceScatter

先 Reduce，再 Scatter。每张 GPU 只得到最终结果的一部分。

```
Before:              After:
GPU0: [g0]           GPU0: [part0]
GPU1: [g1]   →       GPU1: [part1]
GPU2: [g2]           GPU2: [part2]
GPU3: [g3]           GPU3: [part3]
```

典型用途：ZeRO 优化中的梯度分区。

### 2.6 AllToAll

每张 GPU 把自己的一部分数据发给其他每张 GPU。

```
Before:              After:
GPU0: [A0 B0 C0 D0]  GPU0: [A0 A1 A2 A3]
GPU1: [A1 B1 C1 D1]  GPU1: [B0 B1 B2 B3]
GPU2: [A2 B2 C2 D2]  GPU2: [C0 C1 C2 C3]
GPU3: [A3 B3 C3 D3]  GPU3: [D0 D1 D2 D3]
```

典型用途：MoE（混合专家模型）中的 all-to-all 路由通信。

## 3. NCCL 如何适配不同硬件

NCCL 会自动探测集群中的 GPU 拓扑和网络设备，选择最优通信路径。

### 3.1 单机内通信

如果所有 GPU 在同一台机器内，NCCL 会优先使用：

- **NVLink + NVSwitch**：最高带宽、最低延迟
- **PCIe**：作为 fallback

### 3.2 跨机通信

如果 GPU 分布在不同机器上，NCCL 会通过网络通信：

- **InfiniBand**：NCCL 直接调用 IB verbs 接口
- **RoCEv2**：NCCL 通过 RoCE 接口通信
- **TCP/IP**：如果没有 RDMA 设备，NCCL 也可以回退到 TCP，但性能很差

### 3.3 混合场景

在大型集群中，经常是**单机内走 NVLink，跨机走 IB/RoCE** 的混合拓扑。NCCL 会自动构建一个包含所有 GPU 的通信图，并选择最优路径。

```
Node 1: GPU0-7 ──NVSwitch──┐
                            ├── IB/RoCE ── Node 2, Node 3, ...
Node 2: GPU0-7 ──NVSwitch──┘
```

## 4. NCCL 的通信算法

对于同一个集合通信操作，NCCL 会根据数据大小、GPU 数量、拓扑选择不同算法。

### 4.1 Ring（环形）

每张 GPU 只和左右邻居通信，数据像接力一样在环中传递。

```
GPU0 → GPU1 → GPU2 → GPU3 → GPU0
```

优点：实现简单，带宽利用率高，适合大数据量。

缺点：延迟较高，需要走多跳。

### 4.2 Tree（树形）

GPU 组成一棵或多棵树，数据向上聚合、向下广播。

```
       GPU0
      /    \
   GPU1    GPU2
   /
GPU3
```

优点：延迟低，适合小数据量。

缺点：根节点可能成为瓶颈，带宽利用率不如 Ring。

### 4.3 NVLink SHARP

在支持 NVSwitch 的架构上，NCCL 可以利用 **SHARP（Scalable Hierarchical Aggregation and Reduction Protocol）**，让 NVSwitch 直接做 reduce 操作，进一步减少通信量。

```
普通 AllReduce：  GPU 之间传 2×数据量
SHARP AllReduce： GPU 之间传 1×数据量（Switch 分担部分计算）
```

SHARP 能显著降低 AllReduce 的有效带宽需求。

## 5. 在 PyTorch 中使用 NCCL

PyTorch 的分布式后端 `nccl` 就是基于 NCCL 实现的。

```python
import torch
import torch.distributed as dist

# 初始化进程组
dist.init_process_group(backend='nccl')

# 创建 local rank 对应的 GPU
torch.cuda.set_device(dist.get_rank() % torch.cuda.device_count())

# 模拟本地梯度
grad = torch.randn(1024, 1024, device='cuda')

# AllReduce：所有进程梯度求和
dist.all_reduce(grad, op=dist.ReduceOp.SUM)

# 如果是平均梯度，再除以 world size
grad /= dist.get_world_size()
```

### 5.1 环境变量调优

NCCL 提供了很多环境变量用于调试和调优：

| 环境变量 | 作用 |
|---------|------|
| `NCCL_DEBUG=INFO` | 输出 NCCL 初始化信息和拓扑检测结果 |
| `NCCL_IB_DISABLE=1` | 禁用 InfiniBand，强制使用 TCP/IP |
| `NCCL_P2P_DISABLE=1` | 禁用 GPU 之间 P2P（NVLink/PCIe）通信 |
| `NCCL_TREE_THRESHOLD` | 控制 Tree 和 Ring 算法的切换阈值 |

### 5.2 常见错误

```text
NCCL error in: ... unhandled system error
```

常见原因：

- IB/RoCE 网络配置不正确
- 防火墙阻止了 NCCL 的通信端口
- GPU 拓扑和 NCCL 预期不符
- 不同节点的 NCCL 版本不一致

调试第一步通常是加 `NCCL_DEBUG=INFO` 看 NCCL 检测到了什么设备和拓扑。

---

## 6. 小结

- **NCCL 是 NVIDIA 的集合通信库**，屏蔽了底层 NVLink/IB/RoCE/PCIe 的差异
- 核心原语包括 **Broadcast、Reduce、AllReduce、AllGather、ReduceScatter、AllToAll**
- **AllReduce** 是数据并行训练的核心，**AllGather/ReduceScatter** 常用于模型并行和 ZeRO
- NCCL 会根据拓扑自动选择 **Ring/Tree/SHARP** 等算法
- 在 PyTorch 中通过 `backend='nccl'` 使用，调试时打开 `NCCL_DEBUG=INFO`

下一章可以聊聊 **AI 数据中心网络架构**：当集群规模扩大到几百、几千节点时，网络拓扑怎么设计才能保证任意两节点之间的高带宽和低延迟。
