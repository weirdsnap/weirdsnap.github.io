# 数据加载与预处理流水线

上一章聊了分布式文件系统怎么存数据。但数据存在存储系统里还不够，还需要高效地喂给 GPU。

在 AI 训练中，经常看到这样的现象：GPU 利用率不高，但显存和算力都充足。原因往往不是模型不够大，而是：

> **数据跟不上 GPU 的消耗速度。**

本章聊数据加载和预处理流水线，目标是让 GPU 尽可能少空等。

---

## 1. 数据加载为什么容易成为瓶颈

### 1.1 GPU 算力 vs 数据读取速度

以训练 GPT 类模型为例：

- 一张 H100 的算力：几百 TFLOPS
- 每次迭代需要读取：几 MB 到几十 MB 的 token 数据
- 如果读取延迟是毫秒级，GPU 就会空闲

当 GPU 数量增加时，问题更严重：

- 1000 张 GPU 同时读取，存储系统需要提供 **每秒 TB 级** 的数据吞吐
- 任何单点瓶颈都会被放大

### 1.2 数据加载的典型路径

```
存储系统（Lustre/S3）
    ↓ 读取原始数据
CPU 内存
    ↓ 解码、预处理、数据增强
CPU → GPU 传输（PCIe）
    ↓
GPU 显存
    ↓
模型训练
```

每一个环节都可能慢：

- 存储读取慢
- CPU 预处理慢
- CPU→GPU 传输慢
- 数据格式转换慢

## 2. PyTorch DataLoader 的核心参数

PyTorch 的 `DataLoader` 是数据加载的主力军。几个关键参数直接影响性能。

### 2.1 num_workers

```python
DataLoader(dataset, batch_size=32, num_workers=4)
```

`num_workers` 表示用多少个子进程预加载数据。

- `num_workers=0`：在主进程加载，训练会阻塞等待数据
- `num_workers=4/8`：常用值，可以并行读取和预处理
- 不是越大越好：太多 worker 会增加 CPU 和内存压力

选择原则：

- 从 4 开始试
- 如果 CPU 还有余量，逐步增加到 8 或 16
- 如果 CPU 被打满，说明预处理太重，需要优化预处理逻辑或离线预处理

### 2.2 pin_memory

```python
DataLoader(dataset, pin_memory=True)
```

`pin_memory=True` 会把数据预先放到 CPU 的**页锁定内存（pinned memory）**中，这样 CPU→GPU 的传输可以由 DMA 直接完成，速度更快。

建议：**只要用 GPU 训练，就开启 pin_memory**。

### 2.3 prefetch_factor

```python
DataLoader(dataset, num_workers=4, prefetch_factor=2)
```

表示每个 worker 预取多少个 batch。默认是 2。

- 太小：GPU 可能等数据
- 太大：占用太多 CPU 内存

### 2.4 persistent_workers

```python
DataLoader(dataset, num_workers=4, persistent_workers=True)
```

每个 epoch 结束后不销毁 worker 进程，避免反复创建进程的开销。多 epoch 训练时建议开启。

## 3. 预处理：在线 vs 离线

### 3.1 在线预处理

每次读取数据时实时做预处理：

```python
# 在线预处理示例
def __getitem__(self, idx):
    text = self.data[idx]
    tokens = self.tokenizer.encode(text)
    return tokens
```

**优点**：灵活，节省磁盘空间
**缺点**：CPU 压力大，容易成为瓶颈

### 3.2 离线预处理

先把数据预处理完，存成训练直接可用的格式：

```
Raw Text
    ↓
Tokenize
    ↓
Pack into fixed-length sequences
    ↓
Save as .bin / .idx / .arrow
```

训练时直接读取预处理后的 token ID：

```python
def __getitem__(self, idx):
    return self.token_ids[idx * seq_len : (idx+1) * seq_len]
```

**优点**：训练时 CPU 负载低，读取速度快
**缺点**：预处理需要时间和存储空间，灵活性差

### 3.3 选择建议

| 场景 | 建议 |
|------|------|
| 预处理简单 | 在线预处理 |
| 预处理复杂、重复性高 | 离线预处理 |
| 大规模训练 | 尽量离线预处理，减少训练时 CPU 开销 |

## 4. 分布式训练中的数据采样

### 4.1 DistributedSampler

在 PyTorch 分布式训练中，需要确保每张 GPU 看到不同的数据：

```python
from torch.utils.data.distributed import DistributedSampler

sampler = DistributedSampler(dataset, num_replicas=world_size, rank=rank)
dataloader = DataLoader(dataset, batch_size=..., sampler=sampler)
```

`DistributedSampler` 会把数据集切分成 `world_size` 份，每个 rank 只读取自己的那份。

### 4.2 shuffle 的处理

```python
sampler.set_epoch(epoch)  # 每个 epoch 换不同的随机种子
```

如果不调用 `set_epoch`，每个 epoch 的 shuffle 顺序会相同，影响训练效果。

### 4.3 大数据集的 shuffle 问题

当数据集大到无法全部加载到内存时，shuffle 变得很困难：

- 完全随机 shuffle 需要随机访问存储，性能差
- 解决方案：**分桶 shuffle** 或 **预 shuffle 后顺序读取**

例如，把数据分成 1000 个 shard，训练时随机选择 shard 顺序，但 shard 内部顺序读取。

## 5. 数据缓存策略

### 5.1 本地 SSD 缓存

把训练数据从共享存储复制到计算节点的本地 NVMe SSD：

```
Shared Storage (Lustre)
    ↓ 训练前一次性拷贝
Local NVMe SSD
    ↓ 训练时高速读取
GPU
```

优点：避免训练时争抢共享存储带宽
缺点：需要额外本地存储空间，数据更新时需要重新同步

### 5.2 内存缓存

如果数据集能放进内存，可以直接把整个数据集加载到内存：

```python
self.data = [self.dataset[i] for i in range(len(self.dataset))]
```

或者使用内存映射文件（mmap）实现部分缓存效果。

### 5.3 分层缓存

大规模场景下，常用分层缓存：

```
Hot data → 本地 NVMe SSD
Warm data → 内存
Cold data → 共享存储 / 对象存储
```

## 6. CPU-GPU 流水线

理想情况下，数据加载和 GPU 计算应该像流水线一样重叠：

```
Time →
CPU:  [加载 batch 1] [加载 batch 2] [加载 batch 3]
GPU:              [训练 1] [训练 2] [训练 3]
```

如果 CPU 加载时间大于 GPU 计算时间，GPU 就会空等。

优化方向：

1. **加快数据加载**：更多 worker、pin_memory、本地缓存
2. **减少预处理**：离线预处理、简化 transform
3. **增加计算量**：更大的模型、更大的 batch，让 GPU 计算时间更长

---

## 7. 小结

- 数据加载是大规模训练常见瓶颈，GPU 空等通常是因为数据没跟上
- `num_workers`、`pin_memory`、`prefetch_factor`、`persistent_workers` 是 DataLoader 的核心调优参数
- **在线预处理**灵活但 CPU 开销大，**离线预处理**更适合大规模训练
- 分布式训练要用 `DistributedSampler` 切分数据，并注意 `set_epoch`
- **本地 SSD 缓存**和**分层缓存**可以有效缓解共享存储带宽压力
- 最终目标是让 CPU 数据加载和 GPU 计算形成流水线，最大化 GPU 利用率
