# Agent 技术的最新进展（2024-2025）

## 从 Function Calling 到通用接口

2023 年 OpenAI 推出 Function Calling 时，工具调用还是**模型专属能力**——每个厂商有自己的格式、自己的 SDK、自己的生态。一年半过去，局面已经大不相同。

本文梳理 2024-2025 年 Agent 技术的三个关键进展：
1. **MCP**：标准化工具协议
2. **Computer Use**：模型直接操作计算机
3. **多模态工具调用**：从文本到视觉/音频

---

## 一、MCP：Model Context Protocol

### 1.1 问题：N×M 的集成噩梦

假设你有 10 个工具（GitHub、Slack、数据库、浏览器...），想接入 5 个模型（GPT-4、Claude、Gemini、Llama、Qwen）。

**没有 MCP 时**：每个工具要为每个模型写适配器，10×5=50 个适配器。

**有 MCP 时**：工具实现一次 MCP 协议，模型支持一次 MCP 协议，10+5=15 个实现。

### 1.2 MCP 是什么

MCP 是 Anthropic 2024 年 11 月推出的**开放协议**，定义了模型如何发现、调用外部工具的标准接口。

**核心思想**：把"工具描述 + 调用方式"标准化，类似 USB 接口——设备（工具）和主机（模型）按统一协议通信。

### 1.3 MCP 协议结构

```json
// 工具描述（Tool Schema）
{
  "name": "read_file",
  "description": "读取文件内容",
  "inputSchema": {
    "type": "object",
    "properties": {
      "path": {"type": "string", "description": "文件路径"}
    },
    "required": ["path"]
  }
}

// 工具调用请求
{
  "jsonrpc": "2.0",
  "method": "tools/call",
  "params": {
    "name": "read_file",
    "arguments": {"path": "/etc/hosts"}
  }
}

// 工具执行结果
{
  "jsonrpc": "2.0",
  "result": {
    "content": [
      {"type": "text", "text": "127.0.0.1 localhost"}
    ]
  }
}
```

### 1.4 MCP vs OpenAI Function Calling

| | OpenAI Functions | MCP |
|--|-----------------|-----|
| **发起方** | OpenAI | Anthropic（但开源开放） |
| **传输协议** | HTTP REST | JSON-RPC over stdio/sse |
| **工具发现** | 请求时传入 tools 参数 | 运行时动态发现 |
| **状态管理** | 无状态 | 支持会话状态 |
| **生态** | OpenAI 专属 | 跨厂商（Claude、Cursor、Windsurf 已支持） |

### 1.5 实际意义

MCP 让**工具开发者**只需实现一次协议，就能被所有支持 MCP 的模型/IDE 使用。Cursor、Windsurf、Claude Desktop 已经内置 MCP 客户端。

## 二、Computer Use：模型操作计算机

### 2.1 从"调用 API"到"操作 GUI"

传统 Function Calling 调用的是**结构化 API**（传 JSON，收 JSON）。Computer Use 更进一步：模型能**看屏幕截图、移动鼠标、点击按钮、输入键盘**。

### 2.2 Claude 3.5 Sonnet 的 Computer Use

```python
# Anthropic 官方示例
response = client.beta.messages.create(
    model="claude-3-5-sonnet-20241022",
    max_tokens=1024,
    tools=[{
        "type": "computer_use",
        "display_width_px": 1024,
        "display_height_px": 768,
    }],
    messages=[{
        "role": "user",
        "content": "打开计算器，计算 1234 * 5678"
    }]
)

# 模型返回：需要截图
# 后端截图 → 传给模型
# 模型返回：鼠标移动到 (x, y)，点击
# 后端执行 → 截图 → 传给模型
# ...循环直到任务完成
```

**工作流程**：
1. 模型请求截图
2. 后端截取屏幕，返回图片
3. 模型分析图片，决定下一步操作（鼠标移动/点击/键盘输入）
4. 后端执行操作
5. 重复 1-4 直到任务完成

### 2.3 与 RPA 的区别

| | 传统 RPA | Computer Use |
|--|---------|-------------|
| **定位** | 按预设规则执行 | 理解任务目标，自主决策 |
| **适应性** | 界面变化就失效 | 能应对界面变化 |
| **开发成本** | 高（需精确录制坐标） | 低（自然语言描述任务） |
| **可靠性** | 高（确定性执行） | 中（模型可能误判） |

### 2.4 应用场景

- **自动化测试**：模型自动点击 UI，验证功能
- **数据录入**：从网页提取数据，填入 Excel
- **跨系统操作**：在多个没有 API 的老系统之间搬运数据
- **你的游戏助手**：理论上可以让模型直接看游戏画面，给出操作指令（而不是只回答文本）

## 三、多模态工具调用

### 3.1 从文本工具到多模态工具

早期 Function Calling 的输入输出都是文本。2024 年起，主流模型开始支持**多模态工具调用**：

| 模型 | 能力 |
|------|------|
| GPT-4o | 图片输入 + 工具调用 |
| Claude 3.5 Sonnet | 图片输入 + Computer Use |
| Gemini 1.5 Pro | 图片/音频/视频输入 + 工具调用 |

### 3.2 典型场景：看图调用工具

```json
{
  "messages": [
    {
      "role": "user",
      "content": [
        {"type": "text", "text": "这张图片里的商品多少钱？帮我查一下评价。"},
        {"type": "image_url", "image_url": {"url": "https://example.com/product.jpg"}}
      ]
    }
  ],
  "tools": [
    {
      "type": "function",
      "function": {
        "name": "search_reviews",
        "parameters": {
          "type": "object",
          "properties": {
            "product_name": {"type": "string"}
          }
        }
      }
    }
  ]
}
```

**流程**：
1. 模型看图片，识别出商品名称
2. 模型调用 `search_reviews` 工具
3. 工具返回评价数据
4. 模型生成最终回答（结合图片内容 + 评价数据）

### 3.3 对游戏助手的启发

你的游戏助手目前是纯文本交互。如果接入多模态：

- 用户上传**游戏截图**，模型直接看图回答"这个 BOSS 怎么打"
- 用户上传**装备面板截图**，模型分析属性，推荐配装
- 用户上传**地图截图**，模型标记路线

**技术路径**：GPT-4o Vision / Claude 3.5 Sonnet 的 vision 能力 + 现有工具调用框架。

## 四、推理模型的工具使用：o1/o3 的范式变化

### 4.1 传统模型 vs 推理模型

| | GPT-4o 等传统模型 | o1/o3 推理模型 |
|--|----------------|---------------|
| **思考方式** | 一次前向传播，直接输出 | 内部思维链（Chain of Thought），多步推理 |
| **工具调用时机** | 生成过程中判断 | 在思维链中决定何时调用工具 |
| **可见性** | 思维过程不可见 | 思维过程隐藏，只输出最终结果 |

### 4.2 o1 的工具调用特点

o1 系列模型在**内部思维链**中使用工具，用户看不到中间推理过程：

```
用户：计算 2024 年北京每个月的平均气温，然后画一个折线图

o1 内部思维链：
1. 需要获取北京 2024 年气温数据 → 调用 weather API
2. 数据拿到了，需要计算月平均 → 写 Python 代码计算
3. 计算完成，需要画图 → 调用 matplotlib 生成图片
4. 返回最终结果给用户

用户看到：直接收到一张折线图 + 简要说明
```

**关键变化**：工具调用从"显式交互"变成"隐式推理的一部分"。

### 4.3 对开发者的影响

- **调试更难**：看不到模型为什么调用某个工具
- **可靠性要求更高**：模型自主决策，工具错误可能导致级联失败
- **新范式**：从"人设计调用流程"到"模型自主规划流程"

## 五、趋势总结

| 阶段 | 时间 | 特征 |
|------|------|------|
| **Function Calling 1.0** | 2023 | 文本输入 → JSON 工具调用 → 文本输出 |
| **多模态扩展** | 2024 H1 | 图片/音频输入 + 工具调用 |
| **Computer Use** | 2024 H2 | 模型直接操作计算机 GUI |
| **MCP 标准化** | 2024.11 | 跨厂商工具协议 |
| **推理模型自主规划** | 2024-2025 | o1/o3 内部思维链 + 隐式工具调用 |

**核心趋势**：工具调用从"显式 API"走向"隐式能力"，从"文本交互"走向"多模态操作"，从"厂商锁定"走向"开放协议"。
