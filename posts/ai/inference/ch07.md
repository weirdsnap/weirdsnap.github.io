## 1. 前缀缓存（Prefix Caching）

### 1.1 解决什么问题

在实际场景中，很多请求有**共同的前缀**：

- 相同的 System Prompt："你是一个 AI 助手，擅长..."
- 相同的对话历史："用户：你好\n助手：你好！有什么可以帮助你的吗？\n用户：..."
- 相同的文档上下文：RAG 场景下，所有请求都基于同一份文档

如果没有前缀缓存，每个请求都要为这些共同前缀独立计算 KV Cache，造成严重浪费。

### 1.2 实现方式

**方式一：完全匹配**

```python
cache = {}

def get_prompt_kv(prompt_tokens):
    for i in range(len(prompt_tokens), 0, -1):
        prefix = tuple(prompt_tokens[:i])
        if prefix in cache:
            cached_kv = cache[prefix]
            remaining = prompt_tokens[i:]
            new_kv = compute_kv(remaining)
            return cached_kv + new_kv
    
    kv = compute_kv(prompt_tokens)
    cache[tuple(prompt_tokens)] = kv
    return kv
```

**方式二：自动检测 + PagedAttention 共享**

在支持 PagedAttention 的推理引擎中，前缀缓存天然得到了支持：

- 计算请求的 prompt 的哈希值
- 相同的 prompt 前缀映射到相同的物理 block
- 通过 Block Table 的 Copy-on-Write 共享这些 block

### 1.3 实际效果

| 场景 | 命中率 | TTFT 改善 | 吞吐提升 |
|------|--------|----------|---------|
| 共享 System Prompt | 100%（前缀） | 减少 30~50% | 20~30% |
| RAG + 同一文档 | 80~90%（文档部分） | 减少 60~80% | 50~100% |
| 多轮对话 | 50~80%（历史部分） | 减少 40~60% | 30~50% |

前缀缓存是"投入最少，收益最大"的推理优化手段之一。

## 2. 推测解码（Speculative Decoding）

### 2.1 核心思想

LLM 的自回归解码是**串行**的：生成 token N+1 前必须等 token N 生成完毕。

推测解码的思路：用一个更小的**草稿模型**（Draft Model）快速生成一串候选 token，然后用大模型并行验证。

```
传统解码（串行）：
t₁ → t₂ → t₃ → t₄ → t₅ （5 步，每步一遍大模型前向）

推测解码（并行验证）：
草稿模型：一步生成 t₁' → t₂' → t₃' → t₄' → t₅'
大模型：验证 [t₁' t₂' t₃' t₄' t₅']（1 步）
如果全部接受：5 个 token 只需 1 步大模型推理
如果部分接受：接受前 k 个，丢弃从第 k+1 个开始
```

### 2.2 为什么能加速？

大模型推理的主要瓶颈是**显存带宽**（memory-bound），而不是**算力**（compute-bound）。

- 草稿模型通常很小（例如 100M 参数），推理延迟极低
- 大模型验证候选 token 时是并行计算，带宽利用更充分
- 理想情况下，推测解码可以获得 2~3x 的加速比

### 2.3 局限性

- 需要额外部署一个草稿模型
- 草稿模型的质量决定接受率
- 对分布式场景的支持有限
- 草稿模型和主模型的词表可能不一致

## 3. 技术全景图

### 3.1 各技术的层面关系

| 技术 | 层面 | 解决的问题 | 与 KV Cache 的关系 |
|------|------|-----------|:----------------:|
| **持续批处理** | 引擎调度 | GPU 利用率低 | 间接影响 |
| **PagedAttention** | 内存管理 | 显存碎片、浪费 | **直接管理分配** |
| **GQA** | 模型架构 | KV Cache 总量过大 | **从源头缩减** |
| **Flash Attention** | 计算优化 | Attention 带宽瓶颈 | 优化读取计算 |
| **前缀缓存** | 引擎策略 | 重复计算 | 共享相同前缀 |
| **推测解码** | 解码策略 | 串行解码延迟 | 不直接影响 |

### 3.2 层级调用链

```
  用户请求
      │
      ▼
┌──────────────────────┐
│  持续批处理调度        │  ← 决定哪些请求一起算
└──────────┬───────────┘
           │
           ▼
┌──────────────────────┐
│  前缀缓存              │  ← 检查公共前缀
└──────────┬───────────┘
           │
           ▼
┌──────────────────────┐
│  GQA（模型架构决定）    │  ← KV Cache 总量由模型设计决定
└──────────┬───────────┘
           │
           ▼
┌──────────────────────┐
│  PagedAttention       │  ← 管理 KV Cache 的物理存储
└──────────┬───────────┘
           │
           ▼
┌──────────────────────┐
│  Flash Attention      │  ← 高效计算 attention
└──────────────────────┘

  推测解码 ─────────── 跨越整个解码循环做加速
```
