# 推理引擎横评：vLLM、TGI、TensorRT-LLM 与 llama.cpp

前面我们讲了量化、推测解码、Flash Attention、GQA 等一系列推理优化技术。但要把这些技术真正跑起来，通常不会从零写代码，而是用一个**推理引擎**。

这一章我们对比当前主流的四个 LLM 推理引擎：

- **vLLM**：开源高吞吐服务的代表
- **TGI（Text Generation Inference）**：Hugging Face 出品
- **TensorRT-LLM**：NVIDIA 官方性能优化方案
- **llama.cpp**：本地运行和边缘部署的首选

---

## 1. 为什么需要专门的推理引擎？

理论上，用 PyTorch 加载模型，写一个循环就能做推理：

```python
model = AutoModelForCausalLM.from_pretrained("model_path")
for _ in range(max_tokens):
    logits = model(input_ids)
    next_token = sample(logits)
    input_ids = torch.cat([input_ids, next_token])
```

但生产环境完全不是这么简单。真实推理引擎要解决：

| 问题 | 说明 |
|------|------|
| **显存管理** | 模型权重 + KV Cache + 激活值，如何不 OOM |
| **批处理** | 多个请求如何高效合并，Continuous Batching 怎么做 |
| **Attention 优化** | Flash Attention、PagedAttention 如何集成 |
| **量化** | INT8/INT4/FP8 的加载和推理 |
| **服务化** | OpenAI-compatible API、流式输出、超时控制 |
| **调度** | 请求优先级、抢占、动态批大小 |
| **多卡并行** | TP/PP 怎么配置 |

一个好的推理引擎把这些都封装好了，用户只需提供模型和配置。

---

## 2. 评测维度

在对比具体引擎前，先明确我们用什么维度评价：

```
1. 性能：吞吐、延迟、显存效率
2. 易用性：安装、配置、部署难度
3. 生态：支持模型数量、社区活跃度
4. 硬件：支持哪些 GPU / CPU
5. 量化：支持哪些量化格式
6. 服务化：API、监控、容器化支持
```

不同场景对这些维度的权重不一样。比如本地运行最看重易用性和 CPU 支持，在线服务最看重吞吐和稳定性。

## 3. vLLM

### 3.1 核心设计

vLLM 最大的贡献是 **PagedAttention**：把 KV Cache 像虚拟内存一样按 block 管理，解决动态长度请求的显存碎片问题。

配合 **Continuous Batching**，vLLM 在在线服务场景下能达到很高的吞吐。

```
请求 1: "你好" → 生成中
请求 2: "解释量子力学" → 生成中
请求 3: 新到达 → 插入 batch
```

vLLM 会在每个生成 step 动态调整 batch 内容，优先把已经生成完的请求踢出，新请求加入。

### 3.2 优点

- **吞吐高**：PagedAttention + Continuous Batching 的组合非常成熟
- **生态好**：支持 Hugging Face 大多数模型，社区活跃
- **易用**：一条命令启动服务

```bash
python -m vllm.entrypoints.openai.api_server \
    --model /path/to/model \
    --tensor-parallel-size 2 \
    --quantization awq
```

- **API 兼容**：提供 OpenAI-compatible API，方便接入现有应用

### 3.3 缺点

- **冷启动慢**：首次加载模型和编译 CUDA kernel 需要时间
- **内存占用不低**：PagedAttention 的 block 粒度会预分配一些显存
- **调试复杂**：调度逻辑封装较深，出问题不易定位

### 3.4 适用场景

- 在线推理服务
- 高并发聊天机器人
- 需要 OpenAI API 兼容性的项目

## 4. TGI（Text Generation Inference）

### 4.1 背景

TGI 是 Hugging Face 推出的生产级推理服务框架。它最早是为 HF 自己的推理 API 服务的，后来开源。

### 4.2 核心设计

TGI 也实现了 Continuous Batching 和 PagedAttention（后来版本引入），但它的设计更偏向**与 Hugging Face 生态无缝集成**。

```bash
docker run --gpus all -p 8080:80 \
    -v /path/to/model:/data \
    ghcr.io/huggingface/text-generation-inference:latest \
    --model-id /data/model
```

### 4.3 优点

- **HF 生态原生**：支持 Safetensors、Transformer 模型最完整
- **生产特性齐全**：
  - Prometheus metrics
  - Token streaming（Server-Sent Events）
  - 健康检查、优雅关闭
  - 多 LoRA 适配
- **容器化成熟**：官方 Docker 镜像维护得很好

### 4.4 缺点

- **灵活性不如 vLLM**：一些高级调度策略不好自定义
- **性能上限略低于 vLLM**：在部分 benchmark 中吞吐稍逊
- **量化支持有限**：相比 vLLM 和 TensorRT-LLM，量化格式支持较少

### 4.5 适用场景

- 已经深度使用 Hugging Face 生态的团队
- 需要快速部署 HF 模型
- 需要完整的监控和运维特性

## 5. TensorRT-LLM

### 5.1 背景

TensorRT-LLM 是 NVIDIA 基于 TensorRT 开发的 LLM 推理优化库。它不是完整的服务框架，更偏向**高性能推理后端**。

### 5.2 核心设计

TensorRT-LLM 在编译阶段做大量优化：

- 算子融合
- Kernel 自动调优
- FP8/INT8/INT4 量化集成
- Multi-GPU 通信优化

它会针对具体模型和 GPU 生成一个优化后的 engine 文件，推理时直接加载这个 engine。

```bash
# 1. 构建 engine
python build.py --model_dir /path/to/model \
    --dtype float16 \
    --use_gpt_attention \
    --output_dir /path/to/engine

# 2. 运行推理
python run.py --engine_dir /path/to/engine
```

### 5.3 优点

- **性能天花板最高**：在 NVIDIA GPU 上通常比 vLLM 快 10~30%
- **量化支持最好**：FP8、INT8、INT4 AWQ/GPTQ 都有深度优化
- **多卡扩展强**：NVLink、InfiniBand 优化到位

### 5.4 缺点

- **NVIDIA 专属**：只支持 NVIDIA GPU
- **构建复杂**：需要为每个模型、每个 GPU 架构生成 engine
- **调试困难**：编译失败时错误信息不够友好
- **生态封闭**：不如 vLLM 社区活跃

### 5.5 适用场景

- NVIDIA 数据中心部署
- 对延迟和吞吐要求极高的场景
- 有专门工程团队维护的项目

## 6. llama.cpp

### 6.1 背景

llama.cpp 是一个用 C/C++ 从头实现的 LLM 推理库，目标是让大模型能在消费级硬件上运行。它支持 CPU、GPU（通过 CUDA/Metal/Vulkan），并且定义了 GGUF 格式。

### 6.2 核心设计

llama.cpp 的核心是**极致的本地运行效率**：

- 纯 C++ 实现，无 Python 依赖
- 支持各种量化格式（Q4_0、Q4_K_M、Q5_K_M、Q8_0）
- CPU 推理优化（AVX、AVX2、ARM NEON）
- 支持 Apple Silicon 的 Metal GPU

```bash
./main -m model.Q4_K_M.gguf \
    -p "You are a helpful assistant." \
    -n 256
```

### 6.3 优点

- **跨平台**：Windows、macOS、Linux、Android 都能跑
- **硬件要求低**：CPU 就能跑 7B 模型
- **GGUF 生态成熟**：Hugging Face 上有大量转换好的 GGUF 模型
- **工具链完整**：提供 server、quantize、perplexity 等工具

### 6.4 缺点

- **服务功能弱**：虽然有 `llama-server`，但不如 vLLM/TGI 成熟
- **高并发能力有限**：设计初衷不是在线服务
- **模型支持有限**：主要针对 Llama 架构，其他架构支持较慢

### 6.5 适用场景

- 本地个人电脑运行大模型
- 边缘设备、嵌入式设备
- 快速验证和原型开发
- 没有高端 GPU 的环境

## 7. SGLang 和其他新兴引擎

### 7.1 SGLang

SGLang 由伯克利大学开发，特点是**结构化生成**和 **RadixAttention**。

RadixAttention 可以看作 PagedAttention 的扩展，专门针对多轮对话和共享前缀做了优化，比 vLLM 的前缀缓存更高效。

SGLang 还提供了类似 Python 的 DSL，方便描述复杂的生成流程：

```python
# SGLang 风格
@function
def multi_turn(state, question):
    s = state + system("You are a helpful assistant.")
    s += user(question)
    s += assistant(gen("answer", max_tokens=256))
    return s
```

### 7.2 其他

| 引擎 | 特点 |
|------|------|
| **LightLLM** | 与 vLLM 类似，更早支持 Continuous Batching |
| **FastChat** | 侧重多模型对话服务，不是纯推理引擎 |
| **MLC-LLM** | 针对手机、浏览器等边缘设备优化 |
| **DeepSpeed-MII** | 微软推出，适合和 DeepSpeed 生态结合 |

## 8. 选型建议

### 8.1 一张表对比

| 维度 | vLLM | TGI | TensorRT-LLM | llama.cpp |
|------|------|-----|--------------|-----------|
| **性能** | 高 | 中高 | 最高 | 中（CPU）/ 高（Metal） |
| **易用性** | 高 | 高 | 低 | 高 |
| **生态** | 很好 | 很好 | 一般 | 好（GGUF） |
| **硬件** | NVIDIA/AMD | NVIDIA | NVIDIA only | CPU/GPU/Metal |
| **量化** | AWQ/GPTQ/FP8 | 有限 | 最全 | GGUF |
| **服务化** | 成熟 | 成熟 | 需自己封装 | 简单 server |
| **本地运行** | 不适合 | 不适合 | 不适合 | 最适合 |
| **在线服务** | 最适合 | 适合 | 适合 | 不适合 |

### 8.2 按场景推荐

| 场景 | 推荐 | 理由 |
|------|------|------|
| 高并发在线 API 服务 | **vLLM** | 吞吐高、API 兼容、生态好 |
| HF 模型快速上线 | **TGI** | 与 HF 生态集成最好 |
| NVIDIA 数据中心极致性能 | **TensorRT-LLM** | 性能天花板最高 |
| 本地笔记本/台式机 | **llama.cpp** | 跨平台、CPU 可跑、GGUF 丰富 |
| 边缘设备/手机 | **MLC-LLM / llama.cpp** | 针对低功耗优化 |
| 复杂结构化生成 | **SGLang** | DSL 和 RadixAttention 有优势 |

---

## 9. 一句话总结

> 没有最好的推理引擎，只有最合适的场景。在线服务首选 vLLM，HF 生态用 TGI，NVIDIA 极致性能选 TensorRT-LLM，本地和边缘部署用 llama.cpp。关键是先明确你的瓶颈是吞吐、延迟、成本还是部署灵活性。
