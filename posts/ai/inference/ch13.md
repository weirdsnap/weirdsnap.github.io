# 用 vLLM 部署一个本地大模型

前面我们讲了推理引擎横评和模型格式。这一章我们真正动手，用 **vLLM** 把一个本地大模型跑成 OpenAI 风格的 API 服务。

例子用 **Qwen2.5-7B-Instruct**，这是目前中文支持和生态都很成熟的模型。文末也会提一下自定义小模型（比如 minimind）部署时可能遇到的坑。

---

## 1. 环境准备：版本匹配是第一坑

vLLM 对 CUDA、PyTorch、Python 版本比较敏感。最常见的问题就是装完后一运行就报 CUDA 相关错误。

### 1.1 版本三角关系

```
vLLM 版本  ←→  PyTorch 版本  ←→  CUDA 版本
```

vLLM 的每个 release 都明确声明了支持的 PyTorch 和 CUDA 版本。去官方 release 页面或 PyPI 看说明：

```
vllm 0.5.x → PyTorch 2.3.x → CUDA 12.1
vllm 0.6.x → PyTorch 2.4.x → CUDA 12.1
```

### 1.2 先查自己的 CUDA 驱动

```bash
nvidia-smi
```

看右上角显示的 CUDA Version，比如 `CUDA Version: 12.4`。这表示驱动最高支持的 CUDA runtime 版本。实际安装的 CUDA toolkit 可以低于这个版本。

```bash
nvcc --version
```

这个显示的是实际安装的 CUDA toolkit 版本。

### 1.3 推荐安装方式

如果有 conda：

```bash
conda create -n vllm python=3.10 -y
conda activate vllm

# 按官方推荐的组合安装
pip install torch==2.4.0 --index-url https://download.pytorch.org/whl/cu121
pip install vllm==0.6.0
```

如果图省事，直接用官方 Docker：

```bash
docker run --runtime nvidia --gpus all \
    -v ~/.cache/huggingface:/root/.cache/huggingface \
    -p 8000:8000 \
    vllm/vllm-openai:v0.6.0 \
    --model Qwen/Qwen2.5-7B-Instruct
```

> 注意：Docker 方式需要提前装好 NVIDIA Container Toolkit。

## 2. 准备模型

### 2.1 从 Hugging Face 下载

```bash
pip install huggingface-hub

huggingface-cli download Qwen/Qwen2.5-7B-Instruct \
    --local-dir ./Qwen2.5-7B-Instruct \
    --local-dir-use-symlinks False
```

如果网络慢，可以换 ModelScope 镜像：

```bash
pip install modelscope
modelscope download --model qwen/Qwen2.5-7B-Instruct --local_dir ./Qwen2.5-7B-Instruct
```

### 2.2 检查模型文件

下载完后，目录结构应该是：

```
Qwen2.5-7B-Instruct/
├── config.json
├── generation_config.json
├── tokenizer.json
├── tokenizer_config.json
├── model.safetensors.index.json
├── model-00001-of-00004.safetensors
├── ...
```

重点检查：
- `config.json` 里有 `architectures` 字段，比如 `"Qwen2ForCausalLM"`
- tokenizer 文件存在
- 权重文件是 `.safetensors` 或 `.bin`

## 3. 启动 vLLM 服务

### 3.1 最简命令

```bash
python -m vllm.entrypoints.openai.api_server \
    --model ./Qwen2.5-7B-Instruct \
    --host 0.0.0.0 \
    --port 8000
```

第一次启动会比较慢，因为 vLLM 在：
- 加载模型权重
- 编译 CUDA kernel
- 生成 Triton cache

第二次启动会快很多。

### 3.2 常用参数解释

```bash
python -m vllm.entrypoints.openai.api_server \
    --model ./Qwen2.5-7B-Instruct \
    --host 0.0.0.0 \
    --port 8000 \
    --tensor-parallel-size 1 \
    --dtype auto \
    --max-model-len 8192 \
    --gpu-memory-utilization 0.9
```

| 参数 | 说明 |
|------|------|
| `--model` | 模型路径或 HF model id |
| `--tensor-parallel-size` | 张量并行 GPU 数，单卡填 1 |
| `--dtype` | 权重精度，`auto` / `float16` / `bfloat16` |
| `--max-model-len` | 最大上下文长度，超出会报错 |
| `--gpu-memory-utilization` | GPU 显存使用率上限，默认 0.9 |

### 3.3 验证服务

```bash
curl http://localhost:8000/v1/models
```

如果返回模型信息，说明服务起来了。

## 4. 调用 API

vLLM 提供 OpenAI-compatible API，可以直接用 OpenAI 的 client 调用。

### 4.1 curl 调用

```bash
curl http://localhost:8000/v1/chat/completions \
    -H "Content-Type: application/json" \
    -d '{
        "model": "Qwen2.5-7B-Instruct",
        "messages": [
            {"role": "system", "content": "You are a helpful assistant."},
            {"role": "user", "content": "你好，介绍一下自己"}
        ],
        "max_tokens": 512
    }'
```

### 4.2 Python 调用

```python
from openai import OpenAI

client = OpenAI(
    base_url="http://localhost:8000/v1",
    api_key="not-needed"
)

response = client.chat.completions.create(
    model="Qwen2.5-7B-Instruct",
    messages=[
        {"role": "user", "content": "你好，介绍一下自己"}
    ],
    stream=True
)

for chunk in response:
    if chunk.choices[0].delta.content:
        print(chunk.choices[0].delta.content, end="")
```

### 4.3 流式输出

vLLM 默认支持 `stream=True`，和 OpenAI API 行为一致。

## 5. 量化模型部署

如果显存不够，可以部署量化模型。

### 5.1 AWQ 模型

```bash
python -m vllm.entrypoints.openai.api_server \
    --model Qwen/Qwen2.5-7B-Instruct-AWQ \
    --quantization awq \
    --dtype auto
```

### 5.2 GPTQ 模型

```bash
python -m vllm.entrypoints.openai.api_server \
    --model Qwen/Qwen2.5-7B-Instruct-GPTQ-Int4 \
    --quantization gptq \
    --dtype auto
```

### 5.3 KV Cache 量化

vLLM 还支持 KV Cache 用 FP8 存储，进一步省显存：

```bash
python -m vllm.entrypoints.openai.api_server \
    --model ./Qwen2.5-7B-Instruct \
    --kv-cache-dtype fp8
```

> 需要 GPU 支持 FP8（如 H100、4090 不支持硬件 FP8，但可以用 fp8_e4m3 软件实现）。

## 6. Docker 部署

Docker 是生产环境最常用的部署方式。

### 6.1 基本命令

```bash
docker run -d \
    --name vllm-server \
    --runtime nvidia \
    --gpus all \
    -p 8000:8000 \
    -v /path/to/Qwen2.5-7B-Instruct:/models \
    vllm/vllm-openai:v0.6.0 \
    --model /models/Qwen2.5-7B-Instruct \
    --served-model-name qwen2.5-7b
```

### 6.2 用 docker-compose

```yaml
version: "3.8"
services:
  vllm:
    image: vllm/vllm-openai:v0.6.0
    runtime: nvidia
    environment:
      - NVIDIA_VISIBLE_DEVICES=all
    volumes:
      - /path/to/models:/models
    ports:
      - "8000:8000"
    command:
      - --model
      - /models/Qwen2.5-7B-Instruct
      - --max-model-len
      - "8192"
```

## 7. 常见坑点与排查

### 7.1 CUDA out of memory

最常见。解决方法：

```bash
# 1. 降低最大上下文长度
--max-model-len 4096

# 2. 降低 GPU 显存使用率，留更多给 KV Cache
--gpu-memory-utilization 0.85

# 3. 减少同时处理的序列数
--max-num-seqs 128

# 4. 用量化模型
```

### 7.2 启动时 kernel 编译卡住

vLLM 第一次启动会编译 Triton kernel，可能需要几分钟。如果卡住：

```bash
# 检查 ~/.cache/vllm 和 ~/.triton 目录权限
# 设置临时目录到空间大的分区
export TMPDIR=/path/to/large/disk
```

### 7.3 tokenizer 特殊 token 报错

有些模型的 tokenizer_config.json 里 special token 配置和 vLLM 预期不一致。可以尝试：

```bash
--trust-remote-code
```

或者手动修改 tokenizer_config.json。

### 7.4 端口被占用

```bash
# 换端口
--port 8001
```

### 7.5 多卡张量并行失败

```bash
# 2 张卡
--tensor-parallel-size 2
```

注意：TP 对 GPU 间通信有要求，最好在 NVLink 环境下使用。PCIe 也能跑，但速度慢。

## 8. 自定义架构模型的坑（以 minimind 为例）

如果你尝试部署过 minimind 这类自定义架构的小模型，可能会遇到：

```
ValueError: Model architecture minimind is not supported for now.
```

vLLM 只内置支持主流架构（Llama、Qwen、Baichuan、ChatGLM 等）。自定义架构需要：

1. **在 vLLM 源码中注册模型**  
   参考 `vllm/model_executor/models/` 目录，新增一个模型定义文件。

2. **实现 attention 和 forward 逻辑**  
   需要把原始 PyTorch 实现改写成 vLLM 的接口（PagedAttention、KV Cache 管理等）。

3. **重新编译安装 vLLM**  
   修改源码后要从源码安装：
   ```bash
   pip install -e .
   ```

4. **更简单的方式**  
   如果只是想跑通自定义小模型，不一定非用 vLLM。用原生 PyTorch 或 transformers 的 `pipeline` 反而更简单。vLLM 的收益在高并发服务场景，单用户本地体验差距不大。

## 9. 性能调参速查

| 参数 | 作用 | 建议 |
|------|------|------|
| `--max-model-len` | 最大序列长度 | 按实际需求设，别盲目设太大 |
| `--gpu-memory-utilization` | 显存使用上限 | 默认 0.9，OOM 时降到 0.85 |
| `--max-num-seqs` | 最大并发序列数 | 高并发场景调高，低延迟场景调低 |
| `--enable-prefix-caching` | 开启前缀缓存 | 多轮对话/共享 system prompt 时打开 |
| `--num-scheduler-steps` | 调度步数 | 默认 1，提高可降低调度开销 |
| `--kv-cache-dtype` | KV Cache 精度 | fp8 / fp16 / auto |

---

## 10. 一句话总结

> vLLM 的部署流程可以概括为：装对版本 → 下好模型 → 一条命令启动 → OpenAI API 调用。真正花时间的是环境版本匹配和 OOM 调参。对于自定义架构的小模型，如果不需要高并发，用原生 PyTorch 可能比硬上 vLLM 更省心。
