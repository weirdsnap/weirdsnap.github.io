# GQA 与 Flash Attention：注意力层的双重优化

[上一章：PagedAttention：KV Cache 的分页管理](./blog.html?post=ai/inference/ch05.md) 我们聊了 KV Cache 的存储管理。但存储只是问题的一半——**Attention 的计算本身也需要优化**。

这一章从两个角度来看注意力层的优化：GQA 在模型架构层面减少 KV Cache 总量，Flash Attention 在计算层面加速 Attention 运算。

---

## 1. GQA：从模型架构缩减 KV Cache

### 1.1 从 MHA 到 MQA 到 GQA

在标准 Transformer 的 Multi-Head Attention（MHA）中，每个注意力头都独立维护自己的 Key、Query、Value。

```
MHA（多头注意力，原始 Transformer）：
 Head 0:    Q0    K0    V0
 Head 1:    Q1    K1    V1
 Head 2:    Q2    K2    V2
 ...（假设 32 个头）
 
→ KV Cache = 2 × 32 × head_dim × seq_len
```

KV Cache 的大小和**注意力头数成正比**。一个很自然的想法：能不能减少 K 和 V 的头数？

**MQA（Multi-Query Attention）** 由谷歌在 2019 年提出：

```
MQA：所有 Query 头共享 1 组 K、V
 Head 0:    Q0 ──┐
 Head 1:    Q1 ──┤── 所有 Query 头共享 1 组 K、V
 Head 2:    Q2 ──┘
                 K0  V0     （只有 1 组 Key/Value）

→ KV Cache 缩小到 1/32！
```

MQA 把 KV Cache 降低了一个数量级，但代价是所有 query 头被迫用相同的 key/value，信息多样性受损。

**GQA（Grouped Query Attention）** 由谷歌在 2023 年提出，是 MHA 和 MQA 的折中：

```
GQA（分组查询注意力）：
               Query 头    Key/Value 头
 Group 0:      Q0  Q1     K0  V0
 Group 1:      Q2  Q3     K1  V1
 
→ KV Cache = 2 × num_kv_heads × head_dim × seq_len
  如果 num_kv_heads = 2（32 个 query 头分为 2 组）
  KV Cache 缩小到原来的 2/32 = 1/16
```

### 1.2 三种注意力机制的对比

| 注意力机制 | KV 头数 | KV Cache 相对大小 | 表达能力 | 典型应用 |
|-----------|--------|:----------------:|:--------:|---------|
| **MHA** | 等于 query 头数（如 32） | 1×（基准线） | 最强 | 原始 Transformer |
| **MQA** | 1 | 1/32 | 最弱 | PaLM |
| **GQA** | 2~8 | 1/16 ~ 1/4 | 接近 MHA | Llama 2/3, Qwen2.5 |

### 1.3 GQA 的实际效果

GQA 是"免费午餐"式的优化：

- **KV Cache 降到 1/4 ~ 1/8**：同样显存可以服务更多请求或更长的上下文
- **质量损失极小**：实验表明 GQA 在几乎不损失模型质量的前提下显著降低推理成本。这也是 Llama 2 70B（GQA-8）、Llama 3 系列（GQA-8）都在用它的原因

### 1.4 GQA 与 PagedAttention 的互补关系

GQA 和 PagedAttention 的关系是**正交且互补**的：

- **GQA**：在模型架构层面减少了 KV Cache 的总量
- **PagedAttention**：在运行时层面优化了 KV Cache 的分配管理

两者叠加的效果非常明显——用 GQA 让 KV Cache 需求降到 1/4，再用 PagedAttention 把它用满。

## 2. Flash Attention：Attention 计算的效率革命

### 2.1 传统 Attention 的显存瓶颈

标准的 Scaled Dot-Product Attention 过程：

```python
S = Q @ K^T              # (1) 得分矩阵
P = softmax(S)           # (2) 概率矩阵
O = P @ V                # (3) 输出
```

每一步都会产生巨大的中间矩阵。以 seq_len=4096、hidden_dim=4096 为例：

- `Q @ K^T` 产生一个 4096 × 4096 = 16M 个元素的矩阵（FP16 → 32 MB）
- 加上 softmax 结果 → 又一个 32 MB
- 对于 32 层 × 32 个头 → 总中间显存 ≈ **32 GB**

Attention 的 HBM 访问量是 O(seq_len² × head_dim)，而 GPU 的算力远超显存带宽，所以 Attention 的瓶颈**不在于算力，而在于显存读写带宽**。

### 2.2 Flash Attention 的核心思想

**Flash Attention**（Tri Dao, 2022）的洞察是：GPU 内部有**多级存储**——显存（HBM）慢但容量大，SRAM（片上缓存）快但容量小。

```
┌─────────────────────────────────────────────┐
│  GPU 芯片                                    │
│  ┌────────────────────┐                      │
│  │  SRAM (~20 MB)     │  最快、最小           │
│  └────────┬───────────┘                      │
│           ↑ ↓                                │
│  ┌────────┴───────────┐                      │
│  │  HBM (24~80 GB)    │  较慢、大容量        │
│  └────────────────────┘                      │
└─────────────────────────────────────────────┘
```

传统方案把 Q、K、V 和中间结果都放在 HBM 中，频繁读写 HBM 成为瓶颈。

Flash Attention 的核心策略：**利用 SRAM 做计算，避免向 HBM 写入中间结果**。

具体做法是**分块计算（Tiling）**：

1. 把 Q、K、V 分成小块（Tile），每块小到可以放进 SRAM
2. 在 SRAM 中完成一小块的 attention 计算
3. 把结果写回 HBM，加载下一块

```
传统方案：
HBM: [Q][K][V] → 计算 S(写HBM) → 读S → softmax(写P) → 读P → 计算O
     3次读         2次写           1次读    1次写       1次读

Flash Attention：
HBM: [Q tile][K tile][V tile] → 加载到 SRAM → 在 SRAM 完成全部计算 → 写回 O
                                循环处理所有 tile
```

### 2.3 收益

| 指标 | 传统 | Flash Attention | 提升 |
|------|------|---------------|:----:|
| HBM 读写量 | O(seq_len² × head_dim) | O(seq_len² × head_dim² / SRAM_size) | 2~4x |
| 训练吞吐 | 基准 | +15~40% | — |

Flash Attention 在推理中的更重要意义是：**释放的显存可以留给 KV Cache 和更大的 batch**。

### 2.4 FA + PagedAttention：算和存的配合

两者解决 attention 的两个不同层面的问题：

```
PagedAttention（存的层面）          Flash Attention（算的层面）
  KV Cache 存在哪里？                怎么用这些 KV 计算 attention？
  → 分块管理、动态分配                 → 利用 SRAM、Tiling
  → 消除碎片、COW共享                 → 减少 HBM 读写
```

- PagedAttention 决定了 KV Cache 在物理显存中的布局
- Flash Attention 负责读取这些 block 并在 SRAM 中高效计算

## 3. 小结

| 技术 | 层面 | 解决的问题 |
|------|------|-----------|
| **GQA** | 模型架构 | 从源头缩减 KV Cache 总量 |
| **Flash Attention** | 计算优化 | 减少 Attention 的 HBM 带宽消耗 |

两者都与 KV Cache 紧密相关：GQA 管"少存点"，PagedAttention 管"存得好"，Flash Attention 管"算得快"。
