# LLM 推理的内存都去哪了？

当你第一次尝试在本地跑一个开源大模型（比如 Llama、Qwen、ChatGLM），最大的拦路虎往往不是代码，而是**显存/内存不够**。报错信息通常是：

```
CUDA out of memory
RuntimeError: CUDA error: out of memory
```

要理解为什么不够，首先要搞清楚：LLM 在推理时，内存主要被**两大部分**吃掉：

1. **模型权重（Model Weights）**：把训练好的海量参数加载到内存里
2. **KV 缓存（KV Cache）**：推理过程中保存历史上下文的状态

今天我们就来仔细分析一下，LLM 在推理运行的时候，到底是怎么样使用我们的显存/内存的。

---

## 1. 模型权重：参数量的直观感受

### 1.1 一个参数占多少字节？

模型权重本质上就是一大堆数字。每个数字用多少位（bit）存储，直接决定占多少内存。
当然因为数字的存储有很多中不同的方式，定点数，浮点数；很多不同的精度，这些都影响模型权重的大小。
其实使用不同的类型来存储参数数据，本身也是一种叫做量化的模型推理加速手段。

| 精度 | 每个参数占用 | 说明 |
|------|-------------|------|
| FP32（单精度浮点） | 4 字节 | 训练时常用，推理很少直接用 |
| FP16 / BF16（半精度） | 2 字节 | 推理最常见，速度和精度平衡 |
| INT8 | 1 字节 | 量化推理，显存减半 |
| INT4 / FP4 | 0.5 字节 | 极致压缩，精度有损失 |

**核心公式**：

```
权重内存 = 参数量 × 每个参数的字节数
```

### 1.2 实际算一笔账

以 Llama 2 7B 为例，"7B" 表示 70 亿个参数：

| 精度 | 计算 | 权重内存 |
|------|------|---------|
| FP32 | 7 × 10⁹ × 4 B | **28 GB** |
| FP16/BF16 | 7 × 10⁹ × 2 B | **14 GB** |
| INT8 | 7 × 10⁹ × 1 B | **7 GB** |
| INT4 | 7 × 10⁹ × 0.5 B | **3.5 GB** |

看到没有？同样是 7B 模型，FP32 需要 28GB，INT4 只需要 3.5GB。这就是为什么"量化"（Quantization）在本地部署里如此重要——它直接决定你的 8GB 显存能不能跑起来。

### 1.3 常见模型的 FP16 权重内存

| 模型 | 参数量 | FP16 权重内存 |
|------|--------|--------------|
| Llama 2 / 3 8B | 8B | ~16 GB |
| Llama 2 13B | 13B | ~26 GB |
| Llama 2 70B | 70B | ~140 GB |
| Qwen2.5 7B | 7B | ~14 GB |
| Qwen2.5 32B | 32B | ~64 GB |

这还仅仅是**把模型加载进来**需要的内存。实际推理时，你还要给 KV Cache、激活值、框架开销留空间。

---

## 2. KV 缓存：上下文长度的代价

### 2.1 为什么需要 KV Cache？

LLM 生成文本是一个字一个字（或一个 token 一个 token）往外吐的。生成第 N 个 token 时，模型需要看前面 N-1 个 token 的信息。

如果每次生成新 token 都把前面所有 token 重新计算一遍，速度会慢到无法接受。KV Cache 的解决办法是：**把前面 token 计算出的 Key 和 Value 矩阵缓存起来**，下次直接复用。

可以简单理解为：

> KV Cache = 模型对"已经看过的上下文"的记忆快照

### 2.2 KV Cache 怎么算？

KV Cache 的大小和以下因素有关：

- **batch size（同时处理多少条请求）**：通常本地推理是 1
- **序列长度（上下文长度）**：你输入的 prompt 加上已经生成的文本长度
- **模型层数（num_layers）**：比如 32 层、40 层
- **注意力头数（num_heads）** 和 **每个头的维度（head_dim）**
- **精度**：FP16 每个数 2 字节，FP32 每个数 4 字节

简化公式（按每个 token 计算）：

```
每个 token 的 KV Cache = 2 × num_layers × num_heads × head_dim × 精度字节数
                      = 2 × num_layers × hidden_size × 精度字节数
```

其中 `hidden_size = num_heads × head_dim`，乘以 2 是因为要存 Key 和 Value 两份。

对于整个序列：

```
总 KV Cache = batch_size × seq_len × 2 × num_layers × hidden_size × 精度字节数
```

### 2.3 一个具体例子

以 Llama 2 7B 为例：
- `num_layers = 32`
- `hidden_size = 4096`
- 精度 FP16（2 字节）

每个 token 的 KV Cache：

```
2 × 32 × 4096 × 2B = 524,288 字节 ≈ 0.5 MB
```

如果上下文长度是 4096 tokens：

```
4096 × 0.5 MB = 2048 MB ≈ 2 GB
```

如果上下文长度拉到 32K：

```
32768 × 0.5 MB = 16 GB
```

也就是说，光是 KV Cache，32K 上下文就要吃掉 16GB。这还不算权重本身占的 14GB。

### 2.4 上下文长度 vs KV Cache

| 上下文长度 | Llama 2 7B FP16 KV Cache |
|-----------|-------------------------|
| 512 | ~0.25 GB |
| 2048 | ~1 GB |
| 4096 | ~2 GB |
| 8192 | ~4 GB |
| 32768 | ~16 GB |

注意：这是**单个请求**。如果 batch size 增加到 4（同时处理 4 个对话），KV Cache 也要乘以 4。

## 3. 总内存占用的快速估算

把两部分合起来：

```
总推理内存 ≈ 权重内存 + KV Cache + 激活值/框架开销
```

激活值和框架开销通常占 10% ~ 20%，粗略估算时可以加 1~2 GB 的余量。

### 3.1 实例：Llama 2 7B FP16，4K 上下文

- 权重内存：14 GB
- KV Cache：2 GB
- 激活值/开销：约 1~2 GB
- **总计：约 17~18 GB**

所以一张 24GB 显存的 RTX 3090/4090 可以比较舒服地跑 FP16 的 7B 模型，4K 上下文。如果是 8GB 显存，就必须上 INT4 量化。

### 3.2 实例：Llama 2 70B FP16，4K 上下文

- 权重内存：140 GB
- KV Cache：更大（hidden_size 和层数都更多）
- **总计：远超单卡 24GB/48GB 显存**

这就是大模型需要多卡并行（Tensor Parallelism、Pipeline Parallelism）的原因。

---

## 4. 量化：在精度和内存之间做取舍

既然内存是瓶颈，最常见的解法就是**量化**（Quantization）。

| 精度 | 7B 模型权重 | 13B 模型权重 | 70B 模型权重 |
|------|-----------|------------|------------|
| FP16 | 14 GB | 26 GB | 140 GB |
| INT8 | 7 GB | 13 GB | 70 GB |
| INT4 | 3.5 GB | 6.5 GB | 35 GB |

量化不仅能省权重内存，通常也能省 KV Cache 内存（如果 KV Cache 也量化）。但代价是：

- **精度下降**：INT4 在某些任务上会有可感知的质量下降
- **计算不一定更快**：低精度矩阵运算需要硬件支持才能加速
- **部署复杂度增加**：需要对应的量化框架（GGUF、AWQ、GPTQ、AutoAWQ 等）

### 4.1 常见的量化格式

- **GGUF**：llama.cpp 使用的格式，支持 Q4_0、Q5_K_M、Q8_0 等多种级别，适合 CPU + GPU 混合推理
- **GPTQ**：主要针对 4bit 权重量化，适合单卡 GPU
- **AWQ / AutoAWQ**：激活感知的权重量化，4bit 下精度保留较好

---

## 5. 为什么你的 8GB 显卡跑不动 7B FP16？

看完上面的计算，这个问题应该很清楚了：

- 7B FP16 权重：14 GB
- 你的显存：8 GB
- 14 > 8，所以直接报错

即使你用 INT4 把权重压到 3.5 GB，再加上 KV Cache 和框架开销，8GB 显存也只是刚刚够用。这就是为什么：

- **7B 模型 + INT4 量化 ≈ 8GB 显存的极限**
- **13B 模型 + INT4 量化 ≈ 16GB 显存的极限**
- **70B 模型基本需要多卡或云服务**

---

## 6. 小结

LLM 推理时的内存主要由两部分组成：

| 组成部分 | 决定因素 | 如何降低 |
|---------|---------|---------|
| **模型权重** | 参数量、精度 | 量化（INT8/INT4）、选更小的模型 |
| **KV Cache** | 层数、隐藏维度、上下文长度、batch size | 缩短上下文、降低 batch size、KV Cache 量化 |

快速估算公式：

```
权重内存(GB) ≈ 参数量(B) × 精度字节数 × 2 / 1024

KV Cache(GB) ≈ batch_size × seq_len × 2 × num_layers × hidden_size × 精度字节数 / (1024³)
```
