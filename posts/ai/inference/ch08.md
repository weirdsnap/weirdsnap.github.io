# 前缀缓存与推测解码

前面几章聊的都是注意力层内部的优化。这一章看两个独立于 Attention 核心的"外挂式"加速技巧——不需要改模型结构，直接加到推理流程里就能生效。

---

## 1. 前缀缓存（Prefix Caching）

### 1.1 解决什么问题

想象这个场景：你给 AI 助手配了一段很长的 System Prompt，比如"你是一个精通 C++ 的编程助手，遵循以下规则：1. 优先用 RAII 2. 避免裸指针 3. ...（500 字）"。然后用户发了 100 个不同的问题。

如果没有前缀缓存：

```
问题 1: 计算完整 KV Cache（含 System Prompt 部分 × 1）
问题 2: 计算完整 KV Cache（含 System Prompt 部分 × 1，重复！）
...
问题 100: 计算完整 KV Cache（含 System Prompt 部分 × 1，重复！）
```

System Prompt 占的 KV Cache 被重复算了 100 次。

在实际场景中，这种共同前缀无处不在：

- **System Prompt**：所有请求共享
- **多轮对话历史**：每次追问都带着前面的对话
- **RAG 文档上下文**：同一份文档嵌入多个问题的 prompt 中
- **Few-shot 示例**：每个请求都带相同的示例

### 1.2 实现方案

**方案一：精确哈希匹配**

最简单的实现——用哈希表缓存 Token 前缀的 KV Cache：

```python
cache: dict[int, list[KVCache]] = {}  # hash → KV Cache

def get_prefilled(prompt_tokens: list[int]) -> KVCache:
    prompt_hash = hash(tuple(prompt_tokens))
    
    if prompt_hash in cache:
        return cache[prompt_hash]  # 完全命中，TTFT 几乎为 0
    
    kv = compute_kv(prompt_tokens)
    cache[prompt_hash] = kv
    return kv
```

但实际场景中，请求很少完全一致。更常见的是**部分前缀相同**：

```
请求 1: "你是一个助手。今天天气怎么样？"
请求 2: "你是一个助手。给我讲个笑话？"
                  ↑ 共同前缀
```

更实用的方案是**最长前缀匹配**：

```
请求 1: [A, B, C, D, E]  → KV Cache 已缓存
请求 2: [A, B, C, X, Y]
         ↑ 命中前 3 个 token 的 KV Cache
         → 只需计算 X, Y 的 KV，C 之前的直接复用
```

```python
def get_prefilled_partial(prompt_tokens):
    for i in range(len(prompt_tokens), 0, -1):
        prefix = tuple(prompt_tokens[:i])
        if prefix in cache:
            cached = cache[prefix]
            remaining = compute_kv(prompt_tokens[i:])
            return cached + remaining
    return compute_kv(prompt_tokens)
```

**方案二：基于 PagedAttention 的自动共享**

这是主流推理引擎（vLLM、SGLang）的实现方式：

1. 将 prompt 按 block 大小（如 16 tokens）切块
2. 计算每个 block 的哈希值
3. 相同哈希的 block 映射到相同的物理 page
4. 通过页表的 Copy-on-Write 机制共享

```
请求 1: [A][B][C][D]  → 物理 page: P1-P2-P3-P4
请求 2: [A][B][C][X]  → 物理 page: P1-P2-P3-P5（P1-P2-P3 共享）
```

这种方式的好处是**不需要显式的前缀匹配逻辑**——块级别的哈希自动实现了共享，粒度更细、更灵活。

### 1.3 实际效果

| 场景 | 命中部分 | TTFT 改善 | 首 token 延迟 |
|------|---------|----------|--------------|
| 共享 System Prompt（500 tokens） | 前缀 100% | 减少 30~50% | ~100ms → ~50ms |
| RAG + 同一文档（2000 tokens） | 文档部分 80~90% | 减少 60~80% | ~400ms → ~80ms |
| 多轮对话（历史 1000 tokens） | 历史部分 50~80% | 减少 40~60% | ~200ms → ~80ms |

**投入产出比最高的优化之一**——不改模型、不改训练、只需配一个缓存策略。

### 1.4 注意事项

- **缓存淘汰**：KV Cache 很占显存（每请求可能几百 MB），需要有 LRU 或 TTL 策略
- **哈希碰撞**：用强哈希（如 xxHash）降低概率
- **前缀对齐**：如果前缀只有部分相同（比如差一个 token），需要退化到最长前缀匹配

## 2. 推测解码（Speculative Decoding）

### 2.1 问题：自回归是串行的

大模型生成文本是**逐 token 串行**的：

```
t₁ → t₂ → t₃ → t₄ → t₅
```

每步都要跑一遍完整的前向传播。对于 70B 的模型，即使 batch size = 1，一次前向也要 ~30ms。生成 100 个 token 就要 3 秒。

关键是：**GPU 在这 3 秒里没有满负荷**——生成时 batch size = 1，显存带宽只利用了很小一部分。

### 2.2 核心思想

> 用一个小模型"打草稿"，大模型快速批处理验证。

```
传统方式（串行，5 步）：
大模型:  t₁ → t₂ → t₃ → t₄ → t₅
        每次 1 个 token，跑 5 次

推测解码（并行验证，1~2 步）：
草稿模型（快速串行生成 5 个候选）:
         t₁' → t₂' → t₃' → t₄' → t₅'
大模型（并行验证全部 5 个）:
         [t₁' t₂' t₃' t₄' t₅'] → 同时计算 5 个位置的 logits
         
如果全部接受：1 步大模型前向 = 5 个 token ✓
如果接受前 3 个：1 步 = 3 个 token ✓（从第 4 个开始重新生成）
```

加速的关键在于：**大模型验证 5 个候选 token 和生成 1 个 token 的计算量差不多**（显存带宽是瓶颈，计算量翻 5 倍但带宽消耗接近），而草稿模型跑 5 步的成本远低于大模型跑 1 步。

### 2.3 草稿模型从哪来

**方案 A：自带的投机头（Self-Speculative）**

有些大模型在训练时就附带了小模型参数（如 7B 的 LM head + 几层 transformer 作为投机头）。推理时投机头先生成候选，主体模型验证。

优点：不需要额外部署，词表天然一致。
缺点：需要训练时就设计好，已有模型不支持。

**方案 B：独立小模型**

用一个小 LLM（如 100M~1B 参数）做草稿模型。

```
草稿模型：tiny-llama（1B）    → 快速生成候选
主模型：   llama-70B           → 并行验证
```

优点：草稿模型可独立训练，接受率可以很高。
缺点：词表可能不一致，需要映射；额外部署开销。

**方案 C：Medusa / Eagle（多头投机）**

2024 年提出的改进方案——不需要草稿模型，直接在主干模型上**添加多个投机头**：

```
主干模型:  t₁ → t₂ → t₃ → t₄
                          ↓
多头投机头:   头 1 → 头 2 → 头 3（同时预测 t₅, t₆, t₇）
```

每个投机头是一个轻量 MLP，训练成本很低，推理时多个头同时预测不同位置的 token。Medusa 在 MT-Bench 上实现了 ~2.3x 的加速。

### 2.4 接受率决定加速效果

推测解码的加速比由**接受率**（Acceptance Rate）决定。粗略地，如果草稿模型很快、可以忽略其成本，那么：

```
加速比 ≈ 平均每次循环接受的 token 数
        = β × (1 - β^γ) / (1 - β)
```

- γ：每次生成的候选数量（投机窗口）
- β：接受率（草稿模型预测正确的概率）

| 接受率 β | 窗口 γ=3 | 窗口 γ=5 | 窗口 γ=10 |
|---------|---------|---------|----------|
| 90% | ~2.4x | ~3.7x | ~5.9x |
| 80% | ~2.0x | ~2.7x | ~3.6x |
| 70% | ~1.5x | ~1.9x | ~2.3x |
| 50% | ~1.2x | ~1.2x | ~1.1x |

注意：实际加速比还要考虑草稿模型本身的成本，所以通常会比这个理论值低一些。

当接受率低于 50% 时，推测解码基本没有收益——退化的候选还会拖慢速度。

提高接受率的方法：

- **草稿模型和主模型越像越好**（同系列、同词表、同训练数据）
- **投机窗口不宜过大**——通常 3~5 个最合适，多了接受率会急剧下降
- **Medusa 风格的多头**可以通过训练让不同头负责不同位置的预测

### 2.5 局限性

- **额外部署开销**：草稿模型虽然小，但仍需占用 GPU 显存
- **吞吐 vs 延迟的权衡**：推测解码主要降低**首 token 后**的单 token 延迟，对吞吐提升有限
- **批处理场景效果差**：当 batch 已经很大时，GPU 带宽已被充分利用，推测解码的"并行验证"优势没那么明显
- **分布式场景复杂**：草稿模型和主模型在不同机器上时，通信延迟可能抵消收益

---

## 3. 小结

| 技术 | 解决的问题 | 原理 | 典型加速 | 适用场景 |
|------|-----------|------|---------|---------|
| 前缀缓存 | 重复计算公共前缀的 KV Cache | 缓存 + 块级哈希共享 | TTFT 降低 30~80% | 共享 system prompt、多轮对话、RAG |
| 推测解码 | 自回归串行瓶颈 | 小模型打草稿 + 大模型并行验证 | 1.5~3x token 生成速度 | 低 batch 交互式场景 |

两个技术的共同点是：**不做模型架构改动，纯推理引擎层面的优化**。这也是它们被广泛采用的原因——对已部署的模型直接加配置就能生效。

量化是另一个不改动模型架构就能降低显存占用的方向：用 INT4/INT8 表示权重，让大模型瘦身。
