## 二、当前结构的优点

1. **顺序合理**  
   内存 → 推理流程 → 分布式 → 服务调度 → Attention 底层优化 → KV Cache 压缩 → 系统级技巧，符合"先懂问题，再看解法"的认知路径。

2. **软硬结合**  
   既有算法层面的 Flash Attention / GQA，也有系统层面的 PagedAttention / Continuous Batching，还有硬件通信层面的 RDMA。

3. **小步快跑**  
   每篇 4-10 分钟阅读，适合碎片化阅读。

## 三、存在的问题

### 1. ~~缺少"量化"这条主线~~ ✅ 已补充
ch10 已覆盖 INT8/INT4、对称/非对称量化、PTQ/QAT、GPTQ、AWQ、FP8 和 GGUF 命名解读。

### 2. 缺少推理引擎/框架
vLLM、TensorRT-LLM、TGI、llama.cpp、SGLang 这些工业界常用的推理引擎没有系统介绍。读者看完优化原理后，不知道实际该用什么工具。

### 3. 缺少部署实践
现在全是原理，缺少：
- 模型格式转换（Safetensors、GGUF、ONNX）
- Docker 化部署
- API 服务化（OpenAI-compatible API）
- 负载均衡 / 多机服务
- 监控与日志

### 4. 部分文章偏薄
| 文章 | 字数 | 问题 |
|------|------|------|
| ch02 一次推理调用拆解 | 1301 | 偏短，Prefill/Decode 的图、shape 变化、batch 影响可以展开 |
| ch05 PagedAttention | 1333 | 分页原理讲了，但和 vLLM 的结合、内存碎片量化可以更深 |
| ch04 推理服务指标 | 1966 | 指标定义有了，但 Continuous Batching 的调度细节、preemption 可以补 |

### 5. ch09 RDMA 的位置有点突兀
RDMA 更像"分布式训练/推理的基础设施"，放在 inference 系列末尾，和前面 Attention 优化衔接不够自然。更适合：
- 单独作为"AI 基础设施"系列开篇
- 或者并入 ch03 并行与通信，作为"高速通信"一节

### 6. AI 模型系列支撑不够
inference ch02 引用 models ch01，说明有依赖。但 models 系列目前只有 2 篇，一些 inference 中需要的模型基础（如位置编码、FFN、激活函数、LayerNorm、MoE）没有铺垫。

## 七、写作风格约定

1. **不要写死相邻文章链接**  
   相邻导航由 `blog.js` 从 `index.json` 自动生成。正文里只引用**非相邻的特定文章**，用 `?post=...` 形式。

2. **代码块若含 `{{` 要处理**  
   GitHub Pages 已禁用 Jekyll（`.nojekyll`），但为防万一，含大量 `{{` 的代码块用 `{% raw %}{% endraw %}` 包裹。

3. **新增文章后必须重新生成索引**  
   ```bash
   python3 scripts/build_index.py
   ```

4. **内部规划文件命名**  
   以 `_` 开头（如此文件），不进入索引。
