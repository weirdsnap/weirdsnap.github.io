# 推测解码进阶：Tree Attention、Lookahead 与实战调参

ch08 我们已经讲过推测解码的基本思想：**用一个小模型快速生成候选 token，再用大模型并行验证**，从而把串行的自回归生成变成"批量验证"。

这一章我们不再重复基础概念，而是深入工程实现和进阶变体：

- 接受率公式到底怎么来的？
- 大模型验证多个候选时，Attention 怎么算？
- Tree Attention 和 Lookahead Decoding 怎么进一步提升接受率？
- 实际部署时怎么调参？

---

## 1. 快速回顾

传统自回归生成：

```
t1 → t2 → t3 → t4 → t5
```

每步一次大模型前向，5 个 token 需要 5 次前向。

推测解码：

```
草稿模型：t1' → t2' → t3' → t4' → t5'   （快速串行，5 步小模型）
大模型：[t1' t2' t3' t4' t5']            （1 步验证 5 个 token）
```

如果 5 个候选全部接受，相当于 1 次大模型前向产出 5 个 token。

## 2. 接受率公式推导

ch08 里我们直接给出了公式：

```
加速比 ≈ β × (1 - β^γ) / (1 - β)
```

其中：
- γ：每次生成的候选数量（投机窗口）
- β：单个候选被接受的概率（接受率）

这个公式怎么理解？

### 2.1 一次投机循环的平均收益

一次完整的投机循环包括：

1. 草稿模型生成 γ 个候选
2. 大模型并行验证这 γ 个候选
3. 从前往后依次检查，直到遇到第一个被拒绝的 token

假设每个 token 被接受的概率都是 β，且相互独立：

```
接受 1 个 token 的概率：β
接受 2 个 token 的概率：β^2
...
接受 γ 个 token 的概率：β^γ
```

**平均每次循环接受的 token 数**：

```
E_accept = 1 × β + 2 × β^2 + ... + γ × β^γ
```

这个求和可以化简为：

```
E_accept ≈ β × (1 - β^γ) / (1 - β)
```

当 β 接近 1 时，E_accept ≈ γ。

### 2.2 一次投机循环的成本

一次循环里：
- 大模型固定跑 1 次前向（验证 γ 个候选）
- 草稿模型跑 γ 次前向（生成 γ 个候选）

如果草稿模型速度是大模型的 k 倍（k 通常 5~20），那么大模型 1 次前向相当于草稿模型 k 次前向。

所以一次循环的"大模型等效前向次数"：

```
Cost = 1 + γ / k
```

当 k 很大时，Cost ≈ 1。

### 2.3 加速比

```
加速比 = E_accept / Cost
       ≈ E_accept / 1
       ≈ β × (1 - β^γ) / (1 - β)
```

当接受率 β 接近 1 时，E_accept ≈ γ，加速比接近 γ；但 β 稍微下降，加速比就会快速恶化。这就是推测解码对草稿模型质量极度敏感的原因。

### 2.4 具体数字感受

| 接受率 β | γ=3 | γ=5 | γ=10 |
|---------|-----|-----|------|
| 95% | ~2.7x | ~4.3x | ~7.6x |
| 80% | ~2.0x | ~2.7x | ~3.6x |
| 60% | ~1.2x | ~1.4x | ~1.5x |

可以看到：
- 接受率 > 80% 时，推测解码才有明显收益
- 接受率 < 60% 时，基本没意义
- γ 不是越大越好，β 不够高时，大 γ 反而拖累

## 3. 验证过程的 Attention 细节

这是推测解码的工程核心：大模型怎么在一次前向里验证 γ 个候选 token？

### 3.1 朴素方案：把候选拼成一条序列

最简单的方法是把 prompt + 候选 token 拼成一条长序列：

```
输入：[prompt, t1', t2', t3', t4', t5']
```

然后跑一次标准的前向传播。模型会为每个位置输出 logits：

```
logits[0] 对应 prompt 最后一个 token 的下一个 token
logits[1] 对应 t1' 位置的下一个 token
...
logits[5] 对应 t5' 位置的下一个 token
```

验证规则：

```
检查 logits[i] 中概率最大的 token 是否等于 t_{i+1}'
或者按概率采样接受
```

具体做法是按 token 逐个接受：

```
for i in range(γ):
    # 大模型在位置 i 预测的下一个 token 分布
    prob_main = softmax(logits[i])
    # 草稿模型生成的候选 token
    draft_token = candidates[i]
    # 按概率判断是否接受
    if random() < prob_main[draft_token] / prob_draft[draft_token]:
        accept(draft_token)
    else:
        # 从调整后的分布中采样，退出验证
        resample_from(prob_main)
        break
```

### 3.2 KV Cache 怎么复用

验证时，prompt 部分的 KV Cache 可以复用。只有候选 token 部分需要重新计算 Attention。

```
Prompt KV Cache：已经算好，直接复用
候选 token KV：新算
```

这是推测解码额外开销很小的原因之一——大部分计算量花在候选 token 上，而不是重算整个 prompt。

### 3.3 为什么验证 5 个 token 不比验证 1 个慢很多

关键原因是：**推理瓶颈在显存带宽，不在计算**。

验证 1 个 token 时，大模型需要从 HBM 读取全部权重一次。验证 5 个 token 时，虽然计算量变成 5 倍，但权重读取量几乎相同（只读一次）。所以只要显存带宽是瓶颈，验证 γ 个 token 的时间 ≈ 验证 1 个 token 的时间。

## 4. Tree Attention：从一条链到一棵树

### 4.1 链式候选的问题

前面讲的推测解码，草稿模型生成的是**一条候选链**：

```
t1' → t2' → t3' → t4' → t5'
```

如果 t3' 被拒绝，那么 t4'、t5' 即使是对的也会被丢弃。这就是链式结构的局限：**前面一个错误会浪费后面所有候选**。

### 4.2 树形候选

Tree Attention 的想法是：草稿模型不只生成一条链，而是生成一棵**候选树**。

```
              t1'
            /  |  \
          t2' t2'' t2'''
          / \
        t3' t3''
```

每个节点有多个子节点，代表对下一个 token 的多种猜测。大模型一次前向验证整棵树的所有节点。

### 4.3 Attention Mask 的设计

树形结构的 Attention 不能再用简单的因果掩码。每个节点只能看到从根到它自己的路径上的 token。

```
普通因果 Attention（链）：
  1 0 0 0
  1 1 0 0
  1 1 1 0
  1 1 1 1

Tree Attention（树）：
  每个节点只 attend 自己的祖先路径
```

这需要特殊的 Attention Mask 构造，以及树形的 KV Cache 布局。vLLM 的 Speculative Decoding 和 SGLang 都实现了 Tree Attention。

### 4.4 收益

Tree Attention 的优势：
- 同样的 γ 个候选 token，可以覆盖更多可能的序列分支
- 前面某个分支被拒绝，其他分支不受影响
- 接受率相同的情况下，Tree Attention 能产出更多 token

代价：
- Attention Mask 更复杂
- 需要更复杂的调度逻辑
- 树不能无限分叉，否则验证成本会上升

## 5. Lookahead Decoding：不用草稿模型

### 5.1 观察：文本有很强的局部重复性

很多生成任务中，模型会重复类似的模式：

```
代码生成：
  def __init__(self, ...):
      self.x = x
      self.y = y
      ...

多轮对话：
  User: ...
  Assistant: ...
  User: ...
```

这些模式在已经生成的文本中出现过。Lookahead Decoding 利用这一点：**用模型自己的输出历史做候选**。

### 5.2 核心思想

维护一个窗口，记录最近生成的 n-gram。当要生成下一个 token 时：

1. 在历史中查找以当前位置结尾的 n-gram
2. 把这些 n-gram 后面的 token 作为候选
3. 大模型并行验证这些候选

```
已生成文本：... A B C D A B C
当前要生成：?

在历史中找到 "A B C" 出现过两次
它后面的 token 分别是 D 和 ?
所以 D 是一个候选
```

### 5.3 优点和局限

优点：
- 不需要额外部署草稿模型
- 对代码、模板化文本特别有效
- 没有词表不一致问题

局限：
- 对创造性文本效果差
- 需要维护 n-gram 缓存，占用一定显存
- 候选质量不如专门训练的草稿模型稳定

## 6. Medusa / Eagle：多头投机

### 6.1 再回顾 Medusa

Medusa 在主干模型上加多个轻量投机头，每个头负责预测未来不同位置的 token：

```
主干输出 t1 后：
  头 1 预测 t2
  头 2 预测 t3
  头 3 预测 t4
```

这些头都是小 MLP，推理开销很小。

### 6.2 训练目标

Medusa 头的训练不需要改动主模型。它的损失函数是：

```
loss = -log P(t_{i+k} | h_i, head_k)
```

即第 k 个头要用位置 i 的隐藏状态预测位置 i+k 的 token。

### 6.3 Eagle 的改进

Eagle 比 Medusa 更进一步：
- 不仅用当前隐藏状态，还显式把已经生成的候选 token 的 embedding 传入
- 有点像"自回归的投机头"，每个头的输入包含前面头的预测结果
- 这样候选之间的依赖关系被建模得更准确

实验上 Eagle 的接受率比 Medusa 更高，但训练和推理也更复杂。

### 6.4 多头投机 vs 独立草稿模型

| 方案 | 优点 | 缺点 |
|------|------|------|
| 独立草稿模型 | 灵活，可以选任意小模型 | 额外部署，词表要对齐 |
| Medusa / Eagle | 无额外模型，和主模型共享计算 | 需要训练，主模型结构要支持 |

## 7. 实战调参与收益边界

### 7.1 投机窗口 γ 怎么选

γ 的选择要在"单次验证量"和"接受率"之间平衡：

```
γ 太小：验证开销占比高，加速不明显
γ 太大：后面候选接受率低，浪费计算
```

工程上常见的选择：
- 独立草稿模型：γ = 3~5
- Medusa：γ = 4~8（因为候选来自同一个模型，一致性更好）
- Lookahead：γ = 5~10（取决于 n-gram 命中率）

### 7.2 草稿模型选多大

草稿模型越小越快，但接受率也越低。

| 主模型 | 草稿模型 | 典型接受率 |
|--------|---------|-----------|
| 70B | 7B | ~60-70% |
| 70B | 1B | ~50-60% |
| 13B | 1B | ~70-80% |

经验：草稿模型参数最好是主模型的 1/10 到 1/5。

### 7.3 什么场景不适合推测解码

1. **batch size 已经很大**  
   大 batch 时 GPU 带宽和计算都被充分利用，推测解码的"并行验证"优势消失。

2. **草稿模型和主模型差异太大**  
   词表不同、训练数据不同，接受率会很低。

3. **创造性/开放式生成**  
   文本多样性高，草稿模型很难猜中主模型的输出。

4. **短输出场景**  
   只生成几个 token 时，投机循环的固定开销占比太高。

### 7.4 实际收益预期

在合适的场景下：
- 单用户交互式对话：1.5x ~ 2.5x
- 代码生成（配合 Lookahead）：2x ~ 3x
- 大 batch 在线服务：收益很小甚至负增长

---

## 8. 一句话总结

> 推测解码的进阶方向，本质上是让"候选生成"变得更聪明：Tree Attention 让候选从链变成树，Lookahead 利用历史重复模式，Medusa/Eagle 把投机能力内建到主模型里。但无论哪种方案，**接受率始终是生命线**——低于 60% 时，再花哨的结构也救不了收益。
