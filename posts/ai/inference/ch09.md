# RDMA：让 CPU 通信不再成为 AI 训练的瓶颈

上一章我们聊了前缀缓存和推测解码——两个不修改模型架构的推理加速技巧。

但所有这些优化都有一个共同的前提：**GPU 之间必须能快速地传数据。**

如果 GPU 分布在不同机器上，数据怎么传？传统思路是：数据从 GPU 拷贝到 CPU 内存，CPU 通过网络协议栈把数据发出去，对方 CPU 收到后再拷贝到 GPU。

这个路径长、延迟高、CPU 占用大。对于 AI 训练这种"数据量巨大、通信极其频繁"的场景，根本不够用。

**RDMA（Remote Direct Memory Access，远程直接内存访问）** 就是来解决这个问题的。它是现代 AI 集群网络的底层支柱之一。

---

## 1. 传统网络通信的问题

### 1.1 数据拷贝的层层关卡

传统的 TCP/IP 网络通信，数据从一台机器传到另一台机器，要经历很多步骤：

```
应用缓冲区
    ↓ 拷贝
Socket 缓冲区（用户态）
    ↓ 拷贝
内核 Socket 缓冲区
    ↓ 协议处理（TCP/IP）
网卡 DMA 缓冲区
    ↓ 物理网络
对方网卡 → 对方内核 → 对方用户态 → 对方应用
```

一次网络发送，数据可能要被拷贝 3~4 次，还要多次进出内核态。每一步都带来延迟和 CPU 开销。

### 1.2 对 AI 训练的影响

在数据并行训练中，每次迭代都要 All-Reduce 整个模型的梯度。如果模型有 70B 参数，FP16 下梯度就有 140GB。

就算把 batch 拆小、梯度压缩，通信量依然巨大。如果网络延迟高、CPU 忙于处理网络协议，GPU 就会**空等数据**，整体训练效率暴跌。

> GPU 算得再快，数据传不过来也是白搭。

---

## 2. RDMA 的核心思想：绕过 CPU，直接访问内存

### 2.1 什么是 RDMA？

RDMA 允许一台机器的网卡**直接读写另一台机器的内存**，几乎不需要 CPU 参与。

```
本机 GPU/CPU 内存  ←──→  本机 RDMA 网卡
                                  ↕
                            物理网络（InfiniBand / RoCE）
                                  ↕
对方 RDMA 网卡  ←──→  对方 GPU/CPU 内存
```

关键点是：**两端网卡直接打交道，数据不经过操作系统内核，也不需要 CPU 逐字节搬运。**

### 2.2 RDMA 的三大特性

| 特性 | 含义 | 效果 |
|------|------|------|
| **Kernel Bypass（内核绕过）** | 应用程序直接和网卡交互，不走操作系统内核网络栈 | 延迟从几十微秒降到 1~2 微秒 |
| **Zero-Copy（零拷贝）** | 数据从发送方内存直接到接收方内存，中间不经过多次缓冲拷贝 | 减少 CPU 和内存带宽占用 |
| **CPU Offload（CPU 卸载）** | 网络传输由网卡处理，CPU 几乎不参与 | CPU 可以专注于计算任务 |

这三点叠加起来，让 RDMA 的通信延迟和吞吐量都远超传统 TCP/IP。

---

## 3. RDMA 的三种网络实现

RDMA 不是一种具体的网络技术，而是一套能力。它可以通过不同的物理网络实现：

### 3.1 InfiniBand（IB）

- **专用网络协议栈**，从物理层到传输层都为 HPC 和 AI 设计
- 延迟最低（~1 μs），带宽最高（目前 NDR 400Gbps，XDR 800Gbps）
- 需要专用 IB 网卡和 IB 交换机
- NVIDIA 收购 Mellanox 后，InfiniBand 成为 NVIDIA DGX 集群的标准网络

**优点**：性能极致，生态成熟。

**缺点**：贵，专用设备，部署复杂。

### 3.2 RoCE（RDMA over Converged Ethernet）

- 在标准以太网上跑 RDMA
- **RoCEv1**：无损以太网，需要 PFC（Priority Flow Control）等机制保证不丢包
- **RoCEv2**：基于 UDP/IP，可以跨子网路由，部署更灵活
- 延迟略高于 IB，但远低于 TCP/IP

**优点**：复用现有以太网交换机（部分需要支持 DCB 特性），成本比 IB 低。

**缺点**：对网络质量要求高，丢包会严重影响 RDMA 性能；RoCEv2 需要无损网络配置。

### 3.3 iWARP（Internet Wide Area RDMA Protocol）

- 基于 TCP/IP 实现 RDMA
- 可以跨广域网，兼容性好

**优点**：路由方便，适合长距离传输。

**缺点**：因为基于 TCP，延迟比 IB 和 RoCE 高；在 AI 训练场景基本不用。

### 3.4 三种方案对比

| 方案 | 底层网络 | 延迟 | 带宽 | 部署成本 | AI 训练常用度 |
|------|---------|------|------|---------|------------|
| InfiniBand | 专用 IB | 最低 | 最高 | 高 | ⭐⭐⭐⭐⭐ |
| RoCEv2 | 以太网 + UDP | 低 | 高 | 中 | ⭐⭐⭐⭐ |
| iWARP | 以太网 + TCP | 中 | 中 | 低 | ⭐ |

目前大规模 AI 集群的主流选择是 **InfiniBand** 或 **RoCEv2**。

## 4. RDMA Verbs：编程接口

RDMA 提供给应用程序的接口叫做 **Verbs**。最常用的有：

### 4.1 Send / Recv

最基础的两边配合操作：

- 接收方先 Post Receive（准备好接收缓冲区）
- 发送方执行 Send
- 数据到达后，接收方收到完成通知

类似传统 socket 的语义，但延迟低得多。

### 4.2 Read

**单边操作**：发送方直接读取接收方内存中的数据，接收方 CPU 完全不用参与。

```
本机：执行 RDMA Read，指定远程地址和长度
对方：网卡自动把内存数据发过来
```

适合"拉取数据"的场景。

### 4.3 Write

**单边操作**：发送方直接把数据写到接收方内存的指定地址，接收方 CPU 不参与。

```
本机：执行 RDMA Write，数据直接写入对方内存
对方：通过完成队列或内存标志感知数据到达
```

Write 是 RDMA 最高效的操作之一，广泛用于 NCCL 的底层实现。

### 4.4 Atomic

原子操作，比如：

- `FETCH_AND_ADD`：读取并增加远程内存值
- `COMPARE_AND_SWAP`：比较并交换

用于分布式锁、计数器等场景。

### 4.5 单边 vs 双边

| 类型 | 操作 | 接收方 CPU 是否参与 | 典型用途 |
|------|------|------------------|---------|
| 双边 | Send/Recv | 需要 Post Receive | 控制消息、不规则通信 |
| 单边 | Read/Write/Atomic | 不参与 | 大规模数据传输、NCCL 底层 |

AI 训练中的梯度同步主要用**单边 Write + 双边控制消息**的组合。

## 5. RDMA 在 AI 训练推理中的应用

### 5.1 多机数据并行训练

当模型可以放进单卡，但训练数据太多时，可以用多机数据并行：

```
机器 0: GPU 0,1,2,3 计算本地梯度
机器 1: GPU 4,5,6,7 计算本地梯度
            ↓
      跨机器 All-Reduce
            ↓
  所有机器拿到相同梯度
```

跨机器 All-Reduce 的通信量巨大，必须走 RDMA。NCCL 会自动检测并使用 RDMA 网络（IB 或 RoCE），把梯度直接从一台机器的 GPU 显存传到另一台机器的 GPU 显存。

### 5.2 大模型分布式训练

对于 GPT-4、Llama 405B、DeepSeek-V3 这类模型：

- **张量并行（TP）**：通常在同一节点内做（NVLink 更快）
- **流水并行（PP）** 和 **数据并行（DP）**：跨节点，走 RDMA
- **MoE 的专家并行（EP）**：All-to-All 通信频繁，极度依赖 RDMA 带宽

可以说，没有 RDMA，现代大模型的分布式训练根本无法扩展。

### 5.3 推理服务的参数服务器

某些推理架构会把模型参数拆分到多台机器上，推理请求根据需要的参数去远程拉取。这种情况下：

- 客户端通过 RDMA Read 拉取参数
- 参数服务器通过 RDMA Write 推送更新

低延迟的 RDMA 能显著减少单次推理的尾延迟。

### 5.4 检查点保存与加载

大模型训练需要定期保存 checkpoint。一个 70B 模型的 FP16 检查点就有 140GB，全量保存到分布式存储时：

- 通过 RDMA 把 GPU 内存数据高速写入 NVMe-oF 或分布式文件系统
- 故障恢复时再通过 RDMA 高速读回

RDMA 能把这个过程的耗时从几十分钟缩短到几分钟。

## 6. RDMA 与 NCCL 的关系

很多初学者会混淆 RDMA 和 NCCL：

| | NCCL | RDMA |
|--|------|------|
| **层级** | 集合通信库（高级） | 网络传输能力（底层） |
| **面向用户** | AI 框架开发者 | 网络/系统开发者 |
| **提供什么** | All-Reduce、All-Gather 等操作 | 远程内存直接访问能力 |
| **关系** | NCCL 的跨机通信底层经常依赖 RDMA | RDMA 可以被 NCCL、MPI、自研框架使用 |

简单说：

> **NCCL 负责"做什么通信"，RDMA 负责"怎么把数据快速搬过去"。**

在 NVIDIA 生态中，NCCL 会自动调用 IB Verbs 或 RoCE 接口，利用 RDMA 完成跨机集合通信。上层用户写 `dist.all_reduce()`，底层可能就是 RDMA Write 在狂奔。

## 7. RDMA 部署的关键考虑

### 7.1 网络拓扑

RDMA 对网络质量敏感，尤其是 RoCE。需要关注：

- **无损网络**：配置 PFC、ECN 等避免丢包
- **拥塞控制**：RDMA 有自己一套拥塞控制机制（如 DCQCN）
- **拓扑一致性**：GPU 拓扑和网络拓扑要对齐，减少跨交换机跳数

### 7.2 内存注册

RDMA 要求通信的内存先**注册（Memory Registration）**到网卡，拿到内存密钥（Memory Key / R_Key）。这会带来一些 CPU 开销，频繁注册会影响性能。

优化方法：

- 预先注册固定大小的内存池
- 使用 HugePage 减少 TLB miss
- GPU Direct RDMA：让网卡直接访问 GPU 显存，避免 CPU 内存中转

### 7.3 GPU Direct RDMA

这是 NVIDIA 的一项关键技术：

> RDMA 网卡可以直接读写 GPU 显存，数据从 GPU 到网卡，再到远端 GPU，全程不经过 CPU 内存。

对 AI 训练来说，这是标准配置。没有 GPU Direct RDMA，跨机通信要先从 GPU 拷到 CPU 内存，再走 RDMA，延迟和带宽都会大打折扣。

---

## 8. 小结

RDMA 是现代 AI 集群的"高速公路"，它让 CPU 从网络通信中解放出来，让 GPU 之间可以低延迟、高带宽地直接交换数据。

核心要点：

| 概念 | 一句话解释 |
|------|-----------|
| **Kernel Bypass** | 绕过操作系统内核，应用直接操作网卡 |
| **Zero-Copy** | 数据从发送方内存直达接收方内存，不反复拷贝 |
| **单边操作** | Read/Write 不需要接收方 CPU 参与 |
| **InfiniBand** | 专用 RDMA 网络，性能最强 |
| **RoCEv2** | 以太网上的 RDMA，成本较低 |
| **GPU Direct RDMA** | 网卡直接访问 GPU 显存，AI 训练标配 |

在 AI 训练中，RDMA 是 NCCL 的底层运力；在推理服务中，RDMA 能降低跨机参数访问延迟；在 checkpoint 读写中，RDMA 能大幅缩短存取时间。

理解 RDMA，你就理解了为什么 AI 大厂的训练集群愿意花重金铺设 InfiniBand 网络——因为**通信延迟每降低一点，整体训练时间就缩短一截**，对于动辄数千万美元的训练成本来说，网络投资完全值得。
