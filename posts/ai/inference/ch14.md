# 推理服务的监控与性能调优

前文中我们把 vLLM 跑起来了。但服务上线后，真正的挑战才刚开始：

- 延迟为什么忽高忽低？
- GPU 显存为什么满了？
- 吞吐到底能到多少？
- 加了批处理后为什么反而更慢了？

这一章讲怎么监控 LLM 推理服务，以及遇到性能问题时怎么定位、怎么调。

---

## 1. 监控什么指标？

推理服务的核心指标可以分为四类：

```
1. 用户感知指标：TTFT、TPOT、端到端延迟
2. 系统资源指标：GPU 利用率、显存占用、显存带宽
3. 服务吞吐指标：QPS、Throughput(tokens/s)、请求排队数
4. 调度指标：batch size、排队延迟、preemption 次数
```

### 1.1 TTFT（Time To First Token）

从请求到达，到第一个 token 返回的时间。

```
TTFT ≈ 请求排队时间 + Prefill 计算时间
```

Prefill 阶段要处理整个 prompt，计算量和 prompt 长度成正比。TTFT 高通常意味着：
- prompt 太长
- 当前有太多请求在 Prefill
- GPU 被 Decode 占满，新请求排不上队

### 1.2 TPOT（Time Per Output Token）

生成阶段每个 token 的平均耗时。

```
TPOT ≈ 一次 Decode step 的时间 / 该 step 生成的 token 数
```

TPOT 高通常意味着：
- batch size 太小，GPU 没吃饱
- KV Cache 太长，显存带宽成为瓶颈
- 模型太大，权重读取慢

### 1.3 Throughput

单位时间内生成的 token 数，通常用 tokens/s 表示。

```
Throughput = 总生成 token 数 / 总时间
```

注意：**吞吐高不等于单用户延迟低**。把很多请求 batch 起来可以提高吞吐，但每个请求等得更久。

## 2. vLLM 内置的 Prometheus Metrics

vLLM 默认暴露了 Prometheus 指标：

```bash
curl http://localhost:8000/metrics
```

关键指标：

| 指标名 | 含义 |
|--------|------|
| `vllm:time_to_first_token_seconds` | TTFT 分布 |
| `vllm:time_per_output_token_seconds` | TPOT 分布 |
| `vllm:e2e_request_latency_seconds` | 端到端请求延迟 |
| `vllm:request_prompt_tokens` | Prompt token 数 |
| `vllm:request_generation_tokens` | 生成 token 数 |
| `vllm:gpu_cache_usage_perc` | GPU KV Cache 使用率 |
| `vllm:cpu_cache_usage_perc` | CPU KV Cache 使用率 |
| `vllm:num_requests_running` | 正在运行的请求数 |
| `vllm:num_requests_waiting` | 排队等待的请求数 |
| `vllm:iteration_tokens_total` | 每次迭代生成的 token 总数 |

## 3. 压测：怎么知道服务能扛多少？

### 3.1 vLLM 自带 benchmark

vLLM 提供了 `benchmark_serving.py`：

```bash
python benchmark_serving.py \
    --backend vllm \
    --model /path/to/model \
    --dataset-name random \
    --random-input-len 1024 \
    --random-output-len 512 \
    --num-prompts 1000 \
    --request-rate 10
```

参数含义：
- `--random-input-len`：模拟 prompt 长度
- `--random-output-len`：模拟生成长度
- `--num-prompts`：总请求数
- `--request-rate`：每秒请求数（QPS）

输出会给出：
- TTFT 统计
- TPOT 统计
- 总 throughput
- 成功率

### 3.2 压测方法论

不要只跑一个 QPS。应该逐步加压，找到拐点：

```
QPS = 1, 5, 10, 20, 50, 100
```

观察：
- TTFT 什么时候开始明显上升？
- TPOT 什么时候开始恶化？
- GPU 显存什么时候满？
- 队列长度什么时候开始持续增长？

找到系统的**饱和点**，这就是你的服务容量上限。

## 4. 常见性能问题与定位

### 4.1 TTFT 过高

**原因 1：Prompt 太长**

Prefill 阶段计算量和 prompt 长度成正比。Prompt 从 1k 变 8k，Prefill 时间可能增加 8 倍。

解决：
- 缩短 prompt（RAG 时只取最相关文档）
- 开启前缀缓存（vLLM `--enable-prefix-caching`）
- 用更小的模型做 prefill

**原因 2：GPU 被 Decode 占满**

如果当前有很多请求在 Decode，新请求的 Prefill 就要排队。

解决：
- 增大 `--max-num-seqs`，允许更多并发
- 设置合理的 `--max-model-len`
- 分离 Prefill 和 Decode（Chimera、Splitwise 等方案）

### 4.2 TPOT 过高

**原因 1：Batch size 太小**

Decode 阶段，batch size = 1 时 GPU 利用率很低。TPOT 高但 GPU 没吃饱。

解决：
- 提高请求并发度
- 调大 `--max-num-seqs`
- 使用 Continuous Batching（vLLM 默认已开）

**原因 2：KV Cache 太长**

生成长文本时，KV Cache 会不断增长，每次都要从 HBM 读取全部 KV Cache。

解决：
- KV Cache 量化（`--kv-cache-dtype fp8`）
- 限制最大生成长度
- 使用 GQA / MLA 减少 KV Cache 大小

**原因 3：模型太大，权重读取慢**

70B 模型权重 140GB，Decode 每步都要读一遍。

解决：
- 量化模型（INT8/INT4）
- 用更快的 GPU（H100 显存带宽比 A100 高约 70%）
- 多张卡张量并行（TP）

### 4.3 GPU 显存满了但利用率不高

这是 LLM 推理的典型现象：**显存瓶颈 ≠ 计算瓶颈**。

可能原因：
- KV Cache 占用了大量显存
- `--gpu-memory-utilization` 设得太高，预分配太多
- 有太多长请求占着 KV Cache

解决：
- 调低 `--gpu-memory-utilization`
- 启用 prefix caching 减少重复 KV Cache
- 设置请求超时和最大生成长度

### 4.4 队列不断增长

说明请求到达速度超过了处理能力。

解决：
- 扩容（加 GPU、多实例负载均衡）
- 限流
- 优化单卡吞吐（量化、更大的 batch）

## 5. 调参速查

| 参数 | 作用 | 调优方向 |
|------|------|---------|
| `--max-model-len` | 最大序列长度 | 按实际需求设，不要过大 |
| `--gpu-memory-utilization` | 显存使用上限 | OOM 时降低；想多留 KV Cache 时降低 |
| `--max-num-seqs` | 最大并发序列数 | 高并发提高；低延迟敏感降低 |
| `--max-num-batched-tokens` | 每次迭代最大 batch token 数 | 长 prompt 场景调大 |
| `--num-scheduler-steps` | 调度器 lookahead 步数 | 默认 1，高吞吐可尝试调大 |
| `--enable-prefix-caching` | 前缀缓存 | 多轮对话/RAG 必开 |
| `--kv-cache-dtype` | KV Cache 精度 | fp8 省显存，但需要 GPU 支持 |
| `--dtype` | 模型权重精度 | auto / float16 / bfloat16 |
| `--tensor-parallel-size` | 张量并行卡数 | 单卡放不下时增加 |
| `--pipeline-parallel-size` | 流水线并行层数 | 超大规模模型使用 |

## 6. Profiling 工具

### 6.1 PyTorch Profiler

可以在 vLLM 中接入 torch profiler：

```python
with torch.profiler.profile(
    activities=[torch.profiler.ProfilerActivity.CPU,
                torch.profiler.ProfilerActivity.CUDA],
    record_shapes=True,
    profile_memory=True
) as prof:
    # 跑推理
    pass

print(prof.key_averages().table())
```

可以看到：
- 每个算子的耗时
- CUDA kernel 执行时间
- 显存分配情况

### 6.2 NVIDIA Nsight Systems

更专业的 GPU profiling 工具：

```bash
nsys profile -o vllm_profile \
    python -m vllm.entrypoints.openai.api_server \
    --model /path/to/model
```

可以分析：
- GPU 空闲时间
- 显存拷贝开销
- Kernel launch 延迟

### 6.3 看日志

vLLM 启动时会打印很多配置信息：

```
INFO 06-30 10:00:00 gpu_executor.py:91] # GPU blocks: 11776, # CPU blocks: 32768
INFO 06-30 10:00:00 model_runner.py:123] Loading model weights took 13.4567 GB
```

`# GPU blocks` 表示 KV Cache 能分成多少个 block，直接决定了最大并发能力。

## 7. 性能优化的权衡

优化 LLM 推理服务时，经常要在几个目标之间取舍：

```
低延迟  ←→  高吞吐
低成本  ←→  高精度
大模型  ←→  低延迟
```

没有万能配置。关键是：

1. 明确你的核心 KPI：是首 token 延迟？还是总生成时间？还是单位成本？
2. 先监控，找到瓶颈
3. 小步调整参数，压测验证
4. 不要同时改多个变量

---

## 8. 一句话总结

> 推理服务的性能优化不是玄学，而是数据驱动的工程：先通过 Prometheus 和压测拿到 TTFT/TPOT/吞吐/显存数据，找到真正的瓶颈，再有针对性地调整 batch size、上下文长度、量化策略和调度参数。监控不到位，调参就是盲猜。
