# 并行技术与集合通信：AI 训练推理的分布式基础

上一章我们聊了 LLM 推理时的内存占用。结论是：大模型的权重和 KV Cache 都很大，单卡根本装不下。

那怎么办？

答案是：**把模型拆到多张卡、多台机器上，一起算。**

这就是并行技术。而并行训练/推理的核心问题，其实可以归结为一句话：

> **多个计算单元之间，怎么高效地交换数据？**

要回答这个问题，必须先理解几个最基础的**集合通信原语**：Broadcast、Scatter、Gather、All-Gather、Reduce、All-Reduce、All-to-All，以及 NVIDIA 专门为 GPU 通信打造的 **NCCL** 库。

---

## 1. 为什么 AI 离不开并行？

### 1.1 单卡的瓶颈

假设你要训练一个 70B 参数的模型：

- FP16 权重：140 GB
- 优化器状态（Adam）：每个参数需要两份动量，再加一份梯度副本
- 激活值、梯度、临时缓冲区

加起来轻松超过 500 GB。单张 GPU 显存通常只有 24GB、48GB 或 80GB，远远不够。

推理也同样困难。即使把 70B 模型量化到 INT4，权重还有 35GB，再加上 KV Cache 和框架开销，单卡也很难吃得消。

### 1.2 三种基本并行策略

| 并行方式 | 拆分对象 | 解决什么问题 | 典型场景 |
|---------|---------|------------|---------|
| **数据并行（DP）** | 数据批次 | 加速训练 | 模型能放进单卡，但 batch 太大 |
| **模型并行 / 张量并行（TP）** | 模型参数 | 单卡放不下模型 | 大模型推理、训练 |
| **流水并行（PP）** | 模型层 | 进一步分摊显存 | 超大规模模型 |

数据并行最简单：每张卡上都有完整模型，各自处理不同的数据，最后把梯度汇总平均。

模型并行更复杂：把模型横向或纵向切开，每张卡只存一部分参数。这时就需要大量集合通信来同步中间结果。

流水并行则是把模型按层分段，像流水线一样依次处理。

现实中，像 GPT-4、Llama 405B、DeepSeek-V3 这种超大规模模型，往往是 **数据并行 + 张量并行 + 流水并行 + 专家并行（MoE）** 一起上。

## 2. 集合通信原语：七张图看懂

集合通信（Collective Communication）指的是**一组进程之间同时进行的数据交换操作**。理解这些原语，是理解分布式 AI 的钥匙。

下面用一个简单的类比：把 4 张 GPU 想象成 4 个工人，每个工人有一块小黑板。

### 2.1 Broadcast：广播

**定义**：一个进程把数据发给所有其他进程。

```
Before:
  GPU 0: [A]   GPU 1: []   GPU 2: []   GPU 3: []

After:
  GPU 0: [A]   GPU 1: [A]   GPU 2: [A]   GPU 3: [A]
```

**应用场景**：
- 数据并行训练开始时，把模型参数从主卡广播到所有卡
- 推理时把输入 prompt 广播到所有参与张量并行的卡

### 2.2 Scatter：散播

**定义**：一个进程把一份数据的不同部分发给不同进程。

```
Before:
  GPU 0: [A|B|C|D]   GPU 1: []   GPU 2: []   GPU 3: []

After:
  GPU 0: [A]   GPU 1: [B]   GPU 2: [C]   GPU 3: [D]
```

**应用场景**：
- 数据并行训练时，把一批数据拆成 4 份，每份给一张卡
- 张量并行中，把输入特征按某种维度切分开

### 2.3 Gather：收集

**定义**：所有进程把数据发给一个进程，由它汇总。

```
Before:
  GPU 0: [A]   GPU 1: [B]   GPU 2: [C]   GPU 3: [D]

After:
  GPU 0: [A|B|C|D]   GPU 1: [B]   GPU 2: [C]   GPU 3: [D]
```

**应用场景**：
- 张量并行推理后，把每块卡上的输出片段收集起来
- 调试时汇总各卡的中间结果

### 2.4 All-Gather：全收集

**定义**：每个进程最后都拥有完整的数据副本。

```
Before:
  GPU 0: [A]   GPU 1: [B]   GPU 2: [C]   GPU 3: [D]

After:
  GPU 0: [A|B|C|D]   GPU 1: [A|B|C|D]   GPU 2: [A|B|C|D]   GPU 3: [A|B|C|D]
```

**应用场景**：
- **张量并行（TP）的核心操作之一**
- 比如在 Megatron-LM 中，线性层按列切分后，需要 All-Gather 把结果拼起来

### 2.5 Reduce：归约

**定义**：所有进程把数据发给一个进程，由它做聚合运算（比如求和、求平均、取最大）。

```
Before:
  GPU 0: [1]   GPU 1: [2]   GPU 2: [3]   GPU 3: [4]

After (SUM):
  GPU 0: [10]   GPU 1: [2]   GPU 2: [3]   GPU 3: [4]
```

**应用场景**：
- 数据并行训练中，把各卡的梯度汇总到主卡
- 但现代训练更常用 All-Reduce，避免单卡成为瓶颈

### 2.6 All-Reduce：全归约

**定义**：所有进程都拿到归约后的结果。

```
Before:
  GPU 0: [1]   GPU 1: [2]   GPU 2: [3]   GPU 3: [4]

After (SUM):
  GPU 0: [10]   GPU 1: [10]   GPU 2: [10]   GPU 3: [10]
```

**应用场景**：
- **数据并行训练的灵魂操作**
- 每张卡算出本地梯度，All-Reduce 后所有卡拿到一样的平均梯度，再各自更新参数

All-Reduce 是分布式训练中最常用的通信原语，没有之一。

### 2.7 All-to-All：全交换

**定义**：每个进程都把自己数据的不同部分发给其他进程，同时从其他进程接收数据。

```
Before:
  GPU 0: [A0|B0|C0|D0]
  GPU 1: [A1|B1|C1|D1]
  GPU 2: [A2|B2|C2|D2]
  GPU 3: [A3|B3|C3|D3]

After:
  GPU 0: [A0|A1|A2|A3]
  GPU 1: [B0|B1|B2|B3]
  GPU 2: [C0|C1|C2|C3]
  GPU 3: [D0|D1|D2|D3]
```

**应用场景**：
- **MoE（混合专家模型）的核心通信**
- 每个 token 需要根据路由规则发给对应的专家，训练完再把结果交换回来
- DeepSeek-V2/V3、Mixtral 都大量依赖 All-to-All

## 3. 原语对比速查表

| 操作 | 输入特点 | 输出特点 | 主要用途 |
|------|---------|---------|---------|
| **Broadcast** | 一卡有数据 | 所有卡都有相同数据 | 参数同步、输入广播 |
| **Scatter** | 一卡有完整数据 | 每卡拿到一部分 | 数据分片、任务分发 |
| **Gather** | 每卡有一部分 | 一卡拿到完整数据 | 结果汇总、调试 |
| **All-Gather** | 每卡有一部分 | 每卡都有完整数据 | 张量并行结果拼接 |
| **Reduce** | 每卡有数据 | 一卡拿到聚合结果 | 梯度汇总（较少用） |
| **All-Reduce** | 每卡有数据 | 每卡都有聚合结果 | 数据并行梯度同步 |
| **All-to-All** | 每卡有按行/列分块的数据 | 每卡拿到转置后的分块 | MoE 路由、数据重排 |

## 4. NCCL：GPU 之间的高速公路

### 4.1 什么是 NCCL？

**NCCL**（NVIDIA Collective Communications Library）是 NVIDIA 专门为 GPU 集群设计的高性能集合通信库。

可以理解为：NCCL 就是把上面讲的 Broadcast、All-Reduce、All-Gather 等操作，在 NVIDIA GPU 上做极致优化的实现。

它自动选择最快的通信路径：

- **同一块 GPU 内部**：直接访问显存
- **同节点内的多张 GPU**：通过 NVLink 或 PCIe P2P
- **跨节点**：通过 InfiniBand 或 RoCE 网卡

### 4.2 为什么 NCCL 这么快？

NCCL 做了大量工程优化：

1. **拓扑感知**：自动检测 GPU 和网络的连接拓扑，选择最优路径
2. **Ring、Tree、NVLS 算法**：不同规模用不同算法，比如 All-Reduce 常用 Ring 算法
3. **与 CUDA 内核融合**：通信和计算可以重叠，隐藏延迟
4. **多流并发**：利用多个 CUDA Stream 同时做不同通信

PyTorch 的 `torch.distributed`、DeepSpeed、Megatron-LM、vLLM 等框架，底层都依赖 NCCL。

### 4.3 NCCL 的使用示例

在 PyTorch 中，NCCL 通常作为后端使用：

```python
import torch
import torch.distributed as dist

# 初始化进程组，使用 NCCL 后端
dist.init_process_group(backend="nccl")

# 创建 local rank 对应的 GPU
torch.cuda.set_device(dist.get_rank())

# 创建张量
tensor = torch.randn(1024, 1024, device="cuda")

# All-Reduce：所有 GPU 的 tensor 求和，结果分发到所有 GPU
dist.all_reduce(tensor, op=dist.ReduceOp.SUM)

# All-Gather：每个 GPU 的 tensor 拼成一个大的 tensor
gather_list = [torch.zeros_like(tensor) for _ in range(dist.get_world_size())]
dist.all_gather(gather_list, tensor)
```

## 5. 并行 + 通信在训练和推理中的具体应用

### 5.1 数据并行训练

```
1. 每张卡加载相同模型副本
2. 每张卡处理不同 mini-batch 数据
3. 各自计算梯度
4. All-Reduce 同步梯度（所有卡拿到平均梯度）
5. 每张卡用相同梯度更新模型
```

通信量：每次迭代都要 All-Reduce 整个模型的梯度。

优点：实现简单，扩展性好（模型能放进单卡时）。

缺点：模型太大时单卡放不下，需要模型并行。

### 5.2 张量并行推理（以 vLLM 为例）

```
1. 把模型的每一层按列或按行切分到多张卡
2. 输入数据广播到所有参与卡
3. 每张卡计算自己那部分矩阵乘法
4. All-Gather 或 All-Reduce 合并结果
5. 继续下一层
```

通信量：每一层都要通信。

优点：单卡显存不够时的标准解法。

缺点：通信开销大，卡数太多时收益递减。

### 5.3 流水线并行

```
1. 把模型按层分成若干段，每段放到一组卡上
2. 数据像流水线一样依次经过各段
3. 前向传播时传递激活值，反向传播时传递梯度
```

通信量：段与段之间传递激活值，比张量并行通信少。

优点：适合超深模型，通信压力小。

缺点：有流水线气泡（某些卡在某些时刻空闲），利用率不如数据并行。

### 5.4 MoE 的专家并行

```
1. 每个 token 经过路由网络，决定发给哪个专家
2. All-to-All 把 token 路由到对应专家所在的卡
3. 专家计算完成后，All-to-All 把结果换回来
```

通信量：每个 token 都要 All-to-All 两次。

优点：激活参数少，推理成本低。

缺点：All-to-All 对网络带宽要求极高，是 MoE 模型的主要瓶颈。

---

## 6. 小结

并行技术是 AI 从大模型走向超大模型的必经之路。而并行的本质，是**多个计算单元之间的高效协作**，协作的核心就是集合通信。

记住这张图：

| 你遇到的问题 | 需要的并行 | 关键通信原语 |
|-----------|-----------|------------|
| 模型能放进单卡，但训练太慢 | 数据并行 | All-Reduce |
| 模型太大，单卡放不下 | 张量并行 | All-Gather、All-Reduce |
| 模型层数太多，显存爆炸 | 流水并行 | Point-to-Point（Send/Recv） |
| 使用 MoE 架构 | 专家并行 | All-to-All |

NCCL 把这些通信原语在 GPU 上做了极致优化，是 PyTorch、DeepSpeed、Megatron、vLLM 等框架的底层支柱。
