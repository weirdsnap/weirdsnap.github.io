# 模型格式与转换：Safetensors、GGUF、ONNX 与 PyTorch 权重

当你下载一个大模型时，可能会看到各种各样的文件后缀：

```
model.safetensors
pytorch_model.bin
model.pth
model.gguf
model.onnx
model.engine
```

它们不是随意命名的，每种格式对应不同的**使用场景、加载方式和推理框架**。

这一章我们讲清楚这些格式是什么、有什么区别、什么时候该用哪个，以及怎么互相转换。

---

## 1. 为什么有这么多模型格式？

模型格式本质上解决的是两个问题：

1. **怎么存权重**：把训练好的参数高效、安全地保存到磁盘
2. **怎么跑推理**：让不同框架、不同硬件都能加载和计算

训练阶段和推理阶段的诉求不一样：

| 阶段 | 关注点 |
|------|--------|
| 训练 | 方便保存/恢复、支持优化器状态、和训练框架绑定 |
| 推理 | 加载快、占用小、跨平台、支持量化、服务化友好 |

所以训练出来的模型往往需要**转换**成更适合推理的格式。

---

## 2. PyTorch 原生格式：.pth / .pt / .bin

这是最常见的一类格式，本质上是 PyTorch `state_dict` 的序列化文件。

### 2.1 里面存了什么

```python
import torch

state_dict = torch.load("pytorch_model.bin", map_location="cpu")
print(state_dict.keys())
# dict_keys(['model.embed_tokens.weight', 'model.layers.0.self_attn.q_proj.weight', ...])
```

`state_dict` 就是一个 Python 字典：
- key：参数名，比如 `layers.0.attention.q_proj.weight`
- value：对应的 `torch.Tensor`

### 2.2 .pth / .pt / .bin 的区别

| 后缀 | 说明 |
|------|------|
| `.pth` / `.pt` | PyTorch 传统后缀，通常用 pickle 序列化 |
| `.bin` | Hugging Face 上常见的后缀，其实也是 PyTorch 权重 |

它们的内部结构几乎一样，只是后缀习惯不同。

### 2.3 优点

- **生态最原生**：PyTorch、Hugging Face Transformers 直接支持
- **信息完整**：可以保存整个模型对象，而不仅是权重
- **通用性最强**：几乎所有训练框架都能导出

### 2.4 缺点

- **只能 PyTorch 用**：其他框架需要转换
- **加载慢**：pickle 反序列化开销大
- **安全性问题**：`.pth` 用 pickle，可能执行恶意代码，不要加载来源不明的 `.pth`
- **不支持内存映射**：大模型加载时必须先完整读入内存

## 3. Safetensors

Safetensors 是 Hugging Face 推出的模型权重格式，专门解决 PyTorch 权重格式的安全和性能问题。

### 3.1 设计目标

```
1. 安全：不执行代码，只存储张量数据
2. 快速：支持零拷贝内存映射
3. 简单：格式规范清晰，跨语言容易解析
4. 惰性加载：可以只加载需要的张量
```

### 3.2 和 .bin 的对比

| 特性 | Safetensors | PyTorch .bin |
|------|-------------|--------------|
| 安全性 | 高，无代码执行 | 低，pickle 可执行任意代码 |
| 加载速度 | 快，支持 mmap | 慢，需反序列化 |
| 内存占用 | 可零拷贝加载 | 需要额外内存拷贝 |
| 跨语言 | 容易 | 依赖 PyTorch |
| 生态 | HF 主推 | 传统主流 |

### 3.3 一个 Safetensors 文件里有什么

```json
{
  "model.embed_tokens.weight": {
    "dtype": "F16",
    "shape": [32000, 4096],
    "data_offsets": [0, 262144000]
  },
  "model.layers.0.self_attn.q_proj.weight": {
    "dtype": "F16",
    "shape": [4096, 4096],
    "data_offsets": [262144000, 295698432]
  }
}
```

本质上就是一张元数据表 + 一段连续的二进制张量数据。

### 3.4 怎么转换

```python
from safetensors.torch import save_file
import torch

state_dict = torch.load("pytorch_model.bin", map_location="cpu")
save_file(state_dict, "model.safetensors")
```

Hugging Face 也提供命令行工具：

```bash
python -m transformers.convert_to_safetensors --model_id /path/to/model
```

## 4. GGUF

GGUF 是 llama.cpp 使用的模型格式，全称是 GPT-Generated Unified Format。它是 GGML 的后继版本。

### 4.1 特点

- **为本地推理设计**：目标是让大模型在消费级 CPU/GPU 上跑起来
- **内置量化信息**：一个 GGUF 文件可以包含 INT4/INT5/INT8 等量化后的权重
- **自描述**：文件头里包含模型架构、超参数、tokenizer 配置等元数据
- **支持分片**：大模型可以分成多个 `.gguf` 文件

### 4.2 常见命名

```
Llama-3-8B-Instruct-Q4_K_M.gguf
Llama-3-8B-Instruct-Q8_0.gguf
```

`Q4_K_M`、`Q8_0` 是量化方案，我们在 ch09 量化里讲过。

### 4.3 怎么从 Hugging Face 转换

llama.cpp 提供了转换脚本：

```bash
# 1. 克隆 llama.cpp
# 2. 安装依赖
pip install -r requirements.txt

# 3. 转换
python convert_hf_to_gguf.py /path/to/hf/model \
    --outfile /path/to/output.gguf \
    --outtype q4_k_m
```

Hugging Face 上也有很多社区预转换好的 GGUF 模型，直接下载即可。

### 4.4 适用场景

- 本地笔记本/台式机运行
- CPU 推理
- 边缘设备
- 需要小体积模型的场景

## 5. ONNX

ONNX（Open Neural Network Exchange）是一个跨框架、跨硬件的模型交换格式。

### 5.1 设计目标

让不同训练框架（PyTorch、TensorFlow、Paddle）导出的模型能在不同推理引擎（ONNX Runtime、TensorRT、OpenVINO）上运行。

```
PyTorch 模型 → ONNX → ONNX Runtime / TensorRT / OpenVINO
```

### 5.2 对 LLM 的挑战

ONNX 对 CNN、BERT 这类模型很成熟，但对大模型有一些挑战：

- **动态 shape**：LLM 的序列长度是变化的，ONNX 对动态 shape 支持有限
- **算子支持**：大模型里的一些新算子（如 RoPE、GQA）可能没有标准 ONNX 算子
- **文件大**：ONNX 通常保存 FP32/FP16，体积比量化格式大

### 5.3 怎么转 ONNX

对于较小的模型，可以用 transformers 的 ONNX 导出：

```bash
python -m transformers.onnx --model=bert-base-uncased --feature=causal-lm exported/
```

对于 LLM，更常用的是 Hugging Face Optimum：

```bash
optimum-cli export onnx --model /path/to/hf/model ./onnx_output/
```

### 5.4 适用场景

- 需要跨框架部署
- 在 ONNX Runtime、TensorRT、OpenVINO 上跑
- 传统模型（BERT、ViT）比 LLM 更适合 ONNX

## 6. 其他格式

### 6.1 TensorRT engine

TensorRT-LLM 编译后生成的二进制文件，针对 NVIDIA GPU 做了深度优化。

```
build.py → model.engine
```

特点：
- 性能最高
- 只支持 NVIDIA GPU
- 构建时间长
- 换一个 GPU 架构通常要重新构建

### 6.2 MLX

Apple 推出的机器学习框架，MLX 格式专门用于 Apple Silicon（M1/M2/M3）。

```
Hugging Face 模型 → MLX 格式 → 在 Mac 上本地跑
```

### 6.3 Paddle / OneFlow / MindSpore

国内框架有自己的格式：
- Paddle：`.pdparams`
- OneFlow：`.ofrecord`
- MindSpore：`.ckpt`

这些主要在各自生态内使用，跨框架需要转换。

## 7. 格式转换流程示例

### 7.1 HF → GGUF

```bash
python convert_hf_to_gguf.py /path/to/hf/model \
    --outfile model.gguf \
    --outtype q4_k_m
```

### 7.2 HF → Safetensors

```bash
python -m transformers.convert_to_safetensors --model_id /path/to/hf/model
```

### 7.3 HF → ONNX

```bash
optimum-cli export onnx --model /path/to/hf/model ./onnx_output/
```

### 7.4 HF → TensorRT engine

```bash
python build.py --model_dir /path/to/hf/model \
    --dtype float16 \
    --output_dir /path/to/engine
```

## 8. 选型建议

| 场景 | 推荐格式 | 理由 |
|------|---------|------|
| Hugging Face 下载/上传 | **Safetensors** | 安全、加载快、HF 主推 |
| 本地 CPU/GPU 跑 | **GGUF** | 体积小、量化内置、llama.cpp 生态成熟 |
| NVIDIA 极致性能 | **TensorRT engine** | 针对 GPU 深度优化 |
| 跨框架部署 | **ONNX** | 通用性强 |
| Apple Silicon | **MLX** | 针对 Mac 优化 |
| 训练/微调中间态 | **PyTorch .bin/.pth** | 原生支持，信息完整 |

---

## 9. 一句话总结

> 模型格式是训练框架和推理框架之间的"翻译官"。PyTorch 权重最通用，Safetensors 更安全高效，GGUF 适合本地量化推理，ONNX 适合跨平台，TensorRT engine 适合 NVIDIA 极致性能。理解它们的定位，就能根据部署场景选对格式，少走弯路。
