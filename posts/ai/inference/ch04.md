# 大模型推理引擎原理：从静态批处理到 PagedAttention

[上一章：RDMA：让 CPU 通信不再成为 AI 训练的瓶颈](./blog.html?post=ai/inference/ch03.md) 我们深入到了网络层面，理解了 RDMA 如何让分布式训练和推理的跨机通信变得高效。

但有了硬件基础设施，还得有高效的**软件引擎**来调度、管理、执行推理请求。这一章我们聚焦推理引擎的核心技术，看看 vLLM、TensorRT-LLM、TGI、llama.cpp 这些框架到底在解决什么问题。

---

## 1. 推理服务的关键指标

在深入技术细节之前，先明确衡量推理服务质量的几个核心指标：

### 1.1 TTFT（Time to First Token）

用户发出请求到收到第一个 token 的时间。

```
TTFT = 预填充时间（Prefill Phase）
```

输入 prompt 越长，TTFT 越长。用户感知到的"首字延迟"就是这个值。好的推理引擎应该让 TTFT 尽可能短（通常要求 < 500ms）。

### 1.2 TPOT（Time Per Output Token）

生成**每个**输出 token 的平均时间。

```
TPOT = 总解码时间 / 输出 token 数
```

这决定了模型"打字"的速度。通常以 **ms/token** 为单位。对于交互式应用（对话、实时翻译），TPOT 需要低至 20~50ms/token。对于离线批处理任务（文档分类、批量翻译），TPOT 可以容忍更高。

### 1.3 Throughput（吞吐量）

系统在单位时间内能处理的请求数或 token 数。

```
Throughput = 总输出 token 数 / 总时间
```

常见的单位是 **tokens/s**。高吞吐对于节省成本和提升资源利用率至关重要。

### 1.4 三个指标的权衡

这三个指标之间存在矛盾：

```
低 TTFT   ←──→  高 Throughput
低 TPOT   ←──→  高 Throughput
```

- 要降低单个请求的延迟（TTFT、TPOT），最好独占 GPU 资源
- 要提高吞吐，最好把多个请求拼在一起算（批处理）

推理引擎的核心设计目标就是：**在延迟可接受的前提下，最大化吞吐**。

| 场景 | 对延时的要求 | 对吞吐的要求 | 代表框架 |
|------|------------|------------|---------|
| 在线对话 | TTFT < 500ms, TPOT < 50ms | 高 | vLLM, TGI, TensorRT-LLM |
| 离线批处理 | 不敏感 | 极高 | FMS, DeepSpeed MII |
| 本地单用户 | TTFT < 2s | 中 | llama.cpp, Ollama |
| 流式 API 服务 | TTFT < 1s, TPOT < 30ms | 极高 | vLLM, TensorRT-LLM |

## 2. 从朴素批处理到持续批处理

### 2.1 朴素批处理（Naive Batching）

最早的推理方案很简单：把一批请求拼成一个 batch，统一走一遍 prefill + decode。

```
时间 → 
┌─────────────────────────────────────┐
│ Prefill(Batch A)                    │
├─────────────────────────────────────┤
│ Decode t1(Batch A)                  │
├─────────────────────────────────────┤
│ Decode t2(Batch A)                  │
├─────────────────────────────────────┤
│ ...                                 │
├─────────────────────────────────────┤
│ Decode tn(Batch A)                  │
└─────────────────────────────────────┘
                                    ┌─────────────────────────────────────┐
                                    │ Prefill(Batch B)                    │
                                    ├─────────────────────────────────────┤
                                    │ ...                                 │
                                    └─────────────────────────────────────┘
```

**问题**：

1. **等待时间长**：新请求必须等当前 batch 全部处理完才能开始
2. **资源浪费**：每个请求生成长度不同，短的请求完成后 GPU 还要等长的
3. **低效批处理**：批处理的优势在于 GPU 矩阵运算的并行性，但输入输出形状不固定导致利用率不高

### 2.2 动态批处理（Dynamic Batching）

一个简单的改进：服务层面维护一个队列，攒够一定数量请求或超时后，组成一个 batch 处理。

```
请求到达 → 入队 → 攒够 batch_size → 推理 → 返回
```

这比朴素批处理好一些，但仍然是**同步**的——同一 batch 内的所有请求必须一起开始、一起结束。

### 2.3 持续批处理（Continuous Batching / Iteration-level Batching）

这是推理引擎的**关键突破**，由 ORCA 论文提出，被 vLLM、TensorRT-LLM、TGI 等主流框架采用。

核心思想：**不用等整个 batch 都处理完，每生成一个 token 后就调整 batch 成员**。

```
时间 → 
┌────────────────────────────────────────────
│ Step 1: Prefill(A) + Prefill(B) + Prefill(C)
├────────────────────────────────────────────
│ Step 2: Decode(A) + Decode(B) + Decode(C)
├────────────────────────────────────────────
│ Step 3: Decode(A) + Decode(B) + Decode(C)
│         (C 在这步完成，从 batch 移除)
├────────────────────────────────────────────
│ Step 4: Decode(A) + Decode(B) + Prefill(D)
│         (D 是新来的请求，直接加入)
├────────────────────────────────────────────
│ ...
└────────────────────────────────────────────
```

**优势**：

- **零等待**：新请求来了，在下一次迭代就可以加入 batch
- **高 GPU 利用率**：随时都有请求在跑，GPU 不会空闲
- **低延迟**：每个请求按自己的节奏完成，不会被长的请求拖慢

这是目前所有高性能推理引擎的标准做法。

## 3. KV Cache 管理：推理引擎的核心挑战

### 3.1 问题回顾

我们之前在 [ch01](./blog.html?post=ai/inference/ch01.md) 讲过 KV Cache 的显存占用。以 Llama 2 7B 为例，32K 上下文的 KV Cache 要吃掉 16GB。

对于推理引擎来说，KV Cache 管理面临三个难题：

1. **显存碎片化**：每个请求的 KV Cache 大小随生成动态增长，频繁分配释放导致碎片
2. **无法预测**：不知道请求最终会生成长度，预先分配最大长度浪费显存
3. **无法共享**：多条请求如果有相同的前缀（如 system prompt），传统方式下各自存一份

### 3.2 传统方式：静态预分配

在 PagedAttention 之前，KV Cache 的管理方式是 **预先分配一大块连续显存**：

```
|←─────── 为 Request A 分配的最大长度 ──────→|
|  KV Cache for A (实际使用)  |   浪费的空间   |
```

```python
# 伪代码：静态预分配
max_seq_len = 4096
hidden_dim = 4096
num_layers = 32
dtype_size = 2  # FP16

# 为每个请求预先分配最大可能大小的 KV Cache
per_request_cache = max_seq_len * hidden_dim * num_layers * 2 * dtype_size
# ≈ 2 GB 每个请求（即使实际只生成了 100 个 token）
```

这种方式的浪费非常严重：
- 大多数请求实际长度远小于最大长度
- 即使只生成 10 个 token，也要预先占着 4096 个 token 的坑
- 显存不足导致 batch size 受限，吞吐上不去

### 3.3 PagedAttention：像操作系统管理内存一样管理 KV Cache

**PagedAttention** 是 vLLM 的核心创新。它借鉴了操作系统的**分页（Paging）**思想——把虚拟内存分页映射到物理内存。

核心思路：

> KV Cache 不再一次性分配连续的显存，而是分成固定大小的块（Block/Page），按需分配、动态映射。

```
传统方式（连续显存）：
┌─────────────────────────────────────────────────────┐
|  Request A: KV Cache (连续的一大块)                  |
└─────────────────────────────────────────────────────┘

PagedAttention（分页显存）：
物理显存块池：
┌──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┐
│ Blk 0│ Blk 1│ Blk 2│ Blk 3│ Blk 4│ Blk 5│ Blk 6│ Blk 7│
└──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┘
                ↑       ↑       ↑
Request A 的块表（Block Table）：
┌──────────┬──────────┬──────────┐
│ Logical 0│ Logical 1│ Logical 2│
│   → Blk 2│   → Blk 1│   → Blk 7│
└──────────┴──────────┴──────────┘
```

**关键概念**：

| 概念 | 类比操作系统 | 含义 |
|------|------------|------|
| **Block（Page）** | 内存页 | KV Cache 的最小分配单元，通常 16 或 32 个 token |
| **Block Table** | 页表 | 逻辑块到物理块的映射关系 |
| **Physical KV Block Pool** | 物理内存 | 预先分配好的固定大小的显存块池 |
| **Copy-on-Write** | COW | 多个请求共享相同块时，写时复制 |

### 3.4 PagedAttention 的优势

**优势 1：消除内部碎片**

不再为每个请求预留最大长度，只用多少 token 就分配多少 block：

```
传统方案：
|←──── 预留 4096 tokens ────→|
|  实际 150 tokens  | 3946 浪费 |
                        ↓
PagedAttention：
| Block 0 | Block 1 | ... | Block 9 | （150 tokens = 10 blocks）
                              ↑
                     没有浪费、精确分配
```

**优势 2：减少外部碎片**

物理块池是统一管理的，请求结束后块被回收，归还到池子中。

**优势 3：实现近乎 100% 的显存利用率**

社区报告的数据显示，PagedAttention 可以将显存利用率从传统的 20~40% 提升到 **95% 以上**。

**优势 4：前缀共享（Prefix Sharing）**

如果多个请求使用相同的 system prompt 或对话历史：

```
Request A: [system prompt] + "如何..."
Request B: [system prompt] + "什么是..."

传统方案：两份独立的 KV Cache
PagedAttention：system prompt 部分只存一份，通过 Block Table 共享
```

这通过 **Copy-on-Write** 机制实现：共享的 block 是只读的，当某个请求需要修改时才复制出新的 block。

### 3.5 PagedAttention 的代价

没有任何技术是免费的：

- **空间换时间**：Block Table 查询增加了一次间接寻址
- **块大小选择问题**：块太小 → 间接寻址开销大；块太大 → 内部碎片又回来了
- **计算复杂度增加**：Attention 计算时需要根据 Block Table 把非连续的物理块拼接成逻辑序列

但实践证明，这些代价相比其带来的显存利用率提升，是完全值得的。

## 4. 推理引擎的完整处理流程

一个请求经过推理引擎时，大致经历以下阶段：

### 4.1 请求接入

```
用户请求 → API Gateway → 请求排队 → 调度器
```

- 请求进入队列
- 调度器根据当前负载和资源情况决定何时处理

### 4.2 调度策略

调度器需要回答一个问题：**下一轮迭代，应该让哪些请求参与计算？**

常见的调度策略：

| 策略 | 逻辑 | 优点 | 缺点 |
|------|------|------|------|
| **FCFS**（先来先服务） | 按到达顺序处理 | 公平 | 短请求可能被长请求阻塞 |
| **SJF**（最短作业优先） | 优先处理预期短的请求 | 降低平均延迟 | 长请求可能饿死 |
| **MLFQ**（多级反馈队列） | 短请求优先，长请求降级 | 平衡延迟和公平 | 实现复杂 |
| **带优先级的调度** | 根据请求的 SLA 级别调度 | 服务质量保障 | 需要用户标记优先级 |

大多数生产级推理引擎（如 vLLM）使用**基于动态规划的调度算法**，在每次迭代时考虑：
- 当前可用的显存（KV Cache 块池还有多少空闲）
- 请求的等待时间
- 请求的预期生成长度
- QoS 要求

### 4.3 预填充（Prefill）

一旦调度器决定哪些请求参与本轮迭代：

```
输入 prompt → Tokenizer → Token IDs → Embedding → Transformer 前向计算
                                                ↓
                                    生成第一个 token + 构建 KV Cache
```

预填充阶段的特点是：
- **计算密集型**：大矩阵乘法的密集计算
- **GPU 利用率高**：大批量矩阵运算，适合 GPU
- **生成 KV Cache**：把整个 prompt 的 Key/Value 缓存起来

### 4.4 解码（Decode）

从第二个 token 开始，每次只生成一个 token：

```
当前位置的输入 → Transformer 前向计算（复用 KV Cache）
                ↓
          生成下一个 token
                ↓
        KV Cache 追加新 token 的 K/V
```

解码阶段的特点是：
- **显存密集型**：主要瓶颈在 KV Cache 的读写带宽
- **GPU 利用率低**：一次只算一个 token，矩阵运算较小
- **延迟敏感**：决定 TPOT 指标

这也是为什么持续批处理能显著提升解码阶段的 GPU 利用率——把多个请求的解码步骤合并成一个较大的矩阵运算。

### 4.5 推理引擎中的调度循环

```
while True:
    # 1. 检查是否有新请求，尝试加入 batch
    new_requests = scheduler.get_new_requests()
    
    # 2. 更新当前 batch 的成员
    batch = scheduler.update_batch(running_requests, new_requests)
    
    # 3. 执行一次模型前向计算
    #    同时处理 batch 中所有请求的当前 token
    outputs = model.forward(batch)
    
    # 4. 处理结果
    for req in batch:
        token = outputs[req]
        req.append_token(token)
        
        if token == EOS or req.max_tokens_reached:
            # 请求完成，清理 KV Cache
            scheduler.finish_request(req)
            # 返回结果给用户
            response_queue.put(req.result())
        else:
            # 请求继续，等待下一轮迭代
            req.step += 1
            scheduler.keep_request(req)
    
    # 5. 内存管理：回收已完成的 block，分配新的 block
    block_manager.garbage_collect()
```

这个循环每步迭代一次（即每生成一个 token 就执行一次），是推理引擎的调度核心。

## 5. 前缀缓存（Prefix Caching）

### 5.1 解决什么问题

在实际场景中，很多请求有**共同的前缀**：

- 相同的 System Prompt："你是一个 AI 助手，擅长..."
- 相同的对话历史："用户：你好\n助手：你好！有什么可以帮助你的吗？\n用户：..."
- 相同的文档上下文：RAG 场景下，所有请求都基于同一份文档

如果没有前缀缓存，每个请求都要为这些共同前缀独立计算 KV Cache，造成严重浪费。

### 5.2 实现方式

**方式一：完全匹配**

缓存触发的 Key-Value 对，新请求到达时检测前缀是否命中缓存。

```python
# 伪代码：前缀缓存
cache = {}

def get_prompt_kv(prompt_tokens):
    # 逐段检测
    for i in range(len(prompt_tokens), 0, -1):
        prefix = tuple(prompt_tokens[:i])
        if prefix in cache:
            # 命中缓存，只需要计算剩余部分
            cached_kv = cache[prefix]
            remaining_prompt = prompt_tokens[i:]
            new_kv = compute_kv(remaining_prompt)
            return cached_kv + new_kv
    
    # 完全没有命中，从头计算
    kv = compute_kv(prompt_tokens)
    cache[tuple(prompt_tokens)] = kv
    return kv
```

**方式二：自动检测 + PagedAttention 共享**

在 vLLM 等支持 PagedAttention 的引擎中，前缀缓存天然得到了支持：

- 计算请求的 prompt 的哈希值
- 相同的 prompt 前缀映射到相同的物理 block
- 通过 Block Table 的 Copy-on-Write 共享这些 block

### 5.3 实际效果

社区报告的前缀缓存效果：

| 场景 | 命中率 | TTFT 改善 | 吞吐提升 |
|------|--------|----------|---------|
| 共享 System Prompt | 100%（前缀） | 减少 30~50% | 20~30% |
| RAG + 同一文档 | 80~90%（文档部分） | 减少 60~80% | 50~100% |
| 多轮对话 | 50~80%（历史部分） | 减少 40~60% | 30~50% |

前缀缓存可以说是"投入最少，收益最大"的推理优化手段之一。

## 6. 推测解码（Speculative Decoding）

### 6.1 核心思想

LLM 的自回归解码是 **串行** 的：生成 token N+1 前必须等 token N 生成完毕。这限制了解码速度。

推测解码的思路是：

> 用一个更小的"草稿模型"（Draft Model）快速生成一串候选 token，然后用大模型并行验证。

```
传统解码（串行）：
t₁ → t₂ → t₃ → t₄ → t₅ （5 步，每步一遍大模型前向）

推测解码（并行验证）：
草稿模型（快）：一步生成 t₁' → t₂' → t₃' → t₄' → t₅'
大模型（并行）：验证 [t₁' t₂' t₃' t₄' t₅']（1 步）
如果全部接受：5 个 token 只需 1 步大模型推理
如果部分接受：接受前 k 个，丢弃从第 k+1 个开始，草稿模型以第 k 个为起点继续
```

### 6.2 为什么能加速？

大模型推理的主要瓶颈是**显存带宽**（memory-bound），而不是**算力**（compute-bound）。

- 草稿模型通常很小（例如 100M 参数），推理延迟极低
- 大模型验证候选 token 时，由于是并行计算，带宽利用更充分
- 理想情况下，推测解码可以获得 2~3x 的加速比

### 6.3 局限性

- 需要额外部署一个草稿模型
- 草稿模型的质量决定接受率（acceptance rate）
- 对分布式场景的支持有限
- 草稿模型和主模型的词表可能不一致，需要额外映射

## 7. 主流推理引擎对比

| 框架 | 核心特点 | 持续批处理 | PagedAttention | 前缀缓存 | 推测解码 | 分布式推理 |
|------|---------|:--------:|:------------:|:-------:|:-------:|:---------:|
| **vLLM** | 开源最流行的推理引擎 | ✅ | ✅ | ✅ | ✅ | ✅ (TP/PP) |
| **TensorRT-LLM** | NVIDIA 官方，极致性能 | ✅ | ✅ | ✅ | ✅ | ✅ (TP/PP/EP) |
| **TGI (HuggingFace)** | 与 HuggingFace 生态深度集成 | ✅ | ✅ | ✅ | ❌ | ❌ |
| **llama.cpp** | CPU 优先，本地部署 | ✅ | ✅ | ❌ | ❌ | ❌ |
| **SGLang** | 前端的结构化生成 | ✅ | ✅ | ✅ | ✅ | ✅ |
| **MLC-LLM** | 多平台（PC、手机、浏览器） | ❌ | ❌ | ❌ | ❌ | ❌ |

### 7.1 vLLM

- **优势**：开源生态最好，社区活跃，文档完善
- **适用**：大多数生产场景，作为推理 API 服务
- **底层**：支持多种后端（CUDA、ROCm、Intel Arc）

### 7.2 TensorRT-LLM

- **优势**：NVIDIA 官方优化，每代 GPU 有专属优化
- **适用**：追求极致吞吐的低延时场景，生产级部署
- **底层**：基于 TensorRT 编译优化，模型需要编译为 TensorRT 引擎

### 7.3 TGI（Text Generation Inference）

- **优势**：和 HuggingFace 生态无缝集成，开箱即用
- **适用**：中小规模、快速原型验证
- **底层**：原生的 Rust 实现，支持 safetensors 加载

### 7.4 llama.cpp

- **优势**：可以在 CPU、MacBook、甚至树莓派上运行
- **适用**：本地开发、个人使用、边缘设备
- **底层**：C++ 实现，GGUF 模型格式，支持各种 CPU 和 GPU 混合

| **PagedAttention** | 显存浪费、碎片化 | 高 | ⭐⭐⭐⭐⭐ |
