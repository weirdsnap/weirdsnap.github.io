# 大模型推理引擎原理：从静态批处理到 PagedAttention

[上一章：RDMA：让 CPU 通信不再成为 AI 训练的瓶颈](./blog.html?post=ai/inference/ch03.md) 我们深入到了网络层面，理解了 RDMA 如何让分布式训练和推理的跨机通信变得高效。

但有了硬件基础设施，还得有高效的**软件引擎**来调度、管理、执行推理请求。这一章我们聚焦推理引擎的核心技术，看看从请求进入系统到 token 输出，中间经历了哪些关键设计和优化。

---

## 1. 推理服务的关键指标

在深入技术细节之前，先明确衡量推理服务质量的几个核心指标：

### 1.1 TTFT（Time to First Token）

用户发出请求到收到第一个 token 的时间。

```
TTFT = 预填充时间（Prefill Phase）
```

输入 prompt 越长，TTFT 越长。用户感知到的"首字延迟"就是这个值。好的推理引擎应该让 TTFT 尽可能短（通常要求 < 500ms）。

### 1.2 TPOT（Time Per Output Token）

生成**每个**输出 token 的平均时间。

```
TPOT = 总解码时间 / 输出 token 数
```

这决定了模型"打字"的速度。通常以 **ms/token** 为单位。对于交互式应用（对话、实时翻译），TPOT 需要低至 20~50ms/token。对于离线批处理任务（文档分类、批量翻译），TPOT 可以容忍更高。

### 1.3 Throughput（吞吐量）

系统在单位时间内能处理的请求数或 token 数。

```
Throughput = 总输出 token 数 / 总时间
```

常见的单位是 **tokens/s**。高吞吐对于节省成本和提升资源利用率至关重要。

### 1.4 三个指标的权衡

这三个指标之间存在矛盾：

```
低 TTFT   ←──→  高 Throughput
低 TPOT   ←──→  高 Throughput
```

- 要降低单个请求的延迟（TTFT、TPOT），最好独占 GPU 资源
- 要提高吞吐，最好把多个请求拼在一起算（批处理）

推理引擎的核心设计目标就是：**在延迟可接受的前提下，最大化吞吐**。

| 场景 | 对延时的要求 | 对吞吐的要求 | 典型优化方向 |
|------|------------|------------|---------|
| 在线对话 | TTFT < 500ms, TPOT < 50ms | 高 | 持续批处理 + PagedAttention |
| 离线批处理 | 不敏感 | 极高 | 最大化 batch size |
| 本地单用户 | TTFT < 2s | 中 | 量化 + 推测解码 |
| 流式 API 服务 | TTFT < 1s, TPOT < 30ms | 极高 | 推理引擎编排 + 前缀缓存 |

## 2. 从朴素批处理到持续批处理

### 2.1 朴素批处理（Naive Batching）

最早的推理方案很简单：把一批请求拼成一个 batch，统一走一遍 prefill + decode。

```
时间 → 
┌─────────────────────────────────────┐
│ Prefill(Batch A)                    │
├─────────────────────────────────────┤
│ Decode t1(Batch A)                  │
├─────────────────────────────────────┤
│ Decode t2(Batch A)                  │
├─────────────────────────────────────┤
│ ...                                 │
├─────────────────────────────────────┤
│ Decode tn(Batch A)                  │
└─────────────────────────────────────┘
                                    ┌─────────────────────────────────────┐
                                    │ Prefill(Batch B)                    │
                                    ├─────────────────────────────────────┤
                                    │ ...                                 │
                                    └─────────────────────────────────────┘
```

**问题**：

1. **等待时间长**：新请求必须等当前 batch 全部处理完才能开始
2. **资源浪费**：每个请求生成长度不同，短的请求完成后 GPU 还要等长的
3. **低效批处理**：批处理的优势在于 GPU 矩阵运算的并行性，但输入输出形状不固定导致利用率不高

### 2.2 动态批处理（Dynamic Batching）

一个简单的改进：服务层面维护一个队列，攒够一定数量请求或超时后，组成一个 batch 处理。

```
请求到达 → 入队 → 攒够 batch_size → 推理 → 返回
```

这比朴素批处理好一些，但仍然是**同步**的——同一 batch 内的所有请求必须一起开始、一起结束。

### 2.3 持续批处理（Continuous Batching / Iteration-level Batching）

这是推理引擎的**关键突破**，由 ORCA 论文提出，现已被所有主流推理框架采用。

核心思想：**不用等整个 batch 都处理完，每生成一个 token 后就重新调度 batch 成员**。

```
时间 → 
┌────────────────────────────────────────────
│ Step 1: Prefill(A) + Prefill(B) + Prefill(C)
├────────────────────────────────────────────
│ Step 2: Decode(A) + Decode(B) + Decode(C)
├────────────────────────────────────────────
│ Step 3: Decode(A) + Decode(B) + Decode(C)
│         (C 在这步完成，从 batch 移除)
├────────────────────────────────────────────
│ Step 4: Decode(A) + Decode(B) + Prefill(D)
│         (D 是新来的请求，直接加入)
├────────────────────────────────────────────
│ ...
└────────────────────────────────────────────
```

**优势**：

- **零等待**：新请求来了，在下一次迭代就可以加入 batch
- **高 GPU 利用率**：随时都有请求在跑，GPU 不会空闲
- **低延迟**：每个请求按自己的节奏完成，不会被长的请求拖慢

这是目前所有高性能推理引擎的标准做法。但它的高效运作离不开下面要讲的几项关键技术。

## 3. KV Cache 管理：推理引擎的核心挑战

### 3.1 问题回顾

我们之前在 [ch01](./blog.html?post=ai/inference/ch01.md) 讲过 KV Cache 的显存占用。以 Llama 2 7B 为例，32K 上下文的 KV Cache 要吃掉 16GB。

对于推理引擎来说，KV Cache 管理面临三个难题：

1. **显存碎片化**：每个请求的 KV Cache 大小随生成动态增长，频繁分配释放导致碎片
2. **无法预测**：不知道请求最终会生成长度，预先分配最大长度浪费显存
3. **无法共享**：多条请求如果有相同的前缀（如 system prompt），传统方式下各自存一份

### 3.2 传统方式：静态预分配

在 PagedAttention 之前，KV Cache 的管理方式是 **预先分配一大块连续显存**：

```
|←─────── 为 Request A 分配的最大长度 ──────→|
|  KV Cache for A (实际使用)  |   浪费的空间   |
```

```python
# 伪代码：静态预分配
max_seq_len = 4096
hidden_dim = 4096
num_layers = 32
dtype_size = 2  # FP16

# 为每个请求预先分配最大可能大小的 KV Cache
per_request_cache = max_seq_len * hidden_dim * num_layers * 2 * dtype_size
# ≈ 2 GB 每个请求（即使实际只生成了 100 个 token）
```

这种方式的浪费非常严重：
- 大多数请求实际长度远小于最大长度
- 即使只生成 10 个 token，也要预先占着 4096 个 token 的坑
- 显存不足导致 batch size 受限，吞吐上不去

### 3.3 PagedAttention：像操作系统管理内存一样管理 KV Cache

**PagedAttention** 是推理引擎领域的一项核心创新。它借鉴了操作系统的**分页（Paging）**思想——把虚拟内存分页映射到物理内存。

核心思路：

> KV Cache 不再一次性分配连续的显存，而是分成固定大小的块（Block/Page），按需分配、动态映射。

```
传统方式（连续显存）：
┌─────────────────────────────────────────────────────┐
|  Request A: KV Cache (连续的一大块)                  |
└─────────────────────────────────────────────────────┘

PagedAttention（分页显存）：
物理显存块池：
┌──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┐
│ Blk 0│ Blk 1│ Blk 2│ Blk 3│ Blk 4│ Blk 5│ Blk 6│ Blk 7│
└──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┘
                ↑       ↑       ↑
Request A 的块表（Block Table）：
┌──────────┬──────────┬──────────┐
│ Logical 0│ Logical 1│ Logical 2│
│   → Blk 2│   → Blk 1│   → Blk 7│
└──────────┴──────────┴──────────┘
```

**关键概念**：

| 概念 | 类比操作系统 | 含义 |
|------|------------|------|
| **Block（Page）** | 内存页 | KV Cache 的最小分配单元，通常 16 或 32 个 token |
| **Block Table** | 页表 | 逻辑块到物理块的映射关系 |
| **Physical KV Block Pool** | 物理内存 | 预先分配好的固定大小的显存块池 |
| **Copy-on-Write** | COW | 多个请求共享相同块时，写时复制 |

### 3.4 PagedAttention 的优势

**优势 1：消除内部碎片**

不再为每个请求预留最大长度，只用多少 token 就分配多少 block：

```
传统方案：
|←──── 预留 4096 tokens ────→|
|  实际 150 tokens  | 3946 浪费 |
                        ↓
PagedAttention：
| Block 0 | Block 1 | ... | Block 9 | （150 tokens = 10 blocks）
                              ↑
                     没有浪费、精确分配
```

**优势 2：减少外部碎片**

物理块池是统一管理的，请求结束后块被回收，归还到池子中。

**优势 3：实现近乎 100% 的显存利用率**

社区报告的数据显示，PagedAttention 可以将显存利用率从传统的 20~40% 提升到 **95% 以上**。

**优势 4：前缀共享（Prefix Sharing）**

如果多个请求使用相同的 system prompt 或对话历史：

```
Request A: [system prompt] + "如何..."
Request B: [system prompt] + "什么是..."

传统方案：两份独立的 KV Cache
PagedAttention：system prompt 部分只存一份，通过 Block Table 共享
```

这通过 **Copy-on-Write** 机制实现：共享的 block 是只读的，当某个请求需要修改时才复制出新的 block。

### 3.5 PagedAttention 的代价

没有任何技术是免费的：

- **空间换时间**：Block Table 查询增加了一次间接寻址
- **块大小选择问题**：块太小 → 间接寻址开销大；块太大 → 内部碎片又回来了
- **计算复杂度增加**：Attention 计算时需要根据 Block Table 把非连续的物理块拼接成逻辑序列

但实践证明，这些代价相比其带来的显存利用率提升，是完全值得的。

## 4. GQA：从架构层面缩减 KV Cache

### 4.1 从 MHA 到 MQA 到 GQA

这一节我们来聊一个**模型架构层面**的优化，它直接在源头减少了 KV Cache 的体积。

在标准 Transformer 的 Multi-Head Attention（MHA）中，每个注意力头都独立维护自己的 Key、Query、Value：

```
MHA（多头注意力，原始 Transformer）：
┌───Query──┐  ┌Key──┐  ┌Value──┐      每个头都有自己的 K、V
 Head 0:    Q0    K0    V0
 Head 1:    Q1    K1    V1
 Head 2:    Q2    K2    V2
 Head 3:    Q3    K3    V3
 ...（假设 32 个头）
 
→ KV Cache 大小 = 2 × 32 × head_dim × seq_len
```

这意味着 KV Cache 的大小和**注意力头数成正比**。既然 KV Cache 是瓶颈，一个很自然的想法就是：能不能减少 K 和 V 的头数？

**MQA（Multi-Query Attention）** 由谷歌在 2019 年提出：

```
MQA（多查询注意力，2019）：
 Head 0:    Q0 ──┐
 Head 1:    Q1 ──┤
 Head 2:    Q2 ──┤── 所有 Query 头共享 1 组 K、V
 Head 3:    Q3 ──┘
                 K0  V0     （只有 1 组 Key/Value）

→ KV Cache 大小缩小到 1/32！
```

MQA 把 KV Cache 降低了一个数量级，但代价是注意力表达能力的下降——所有 query 头被迫用相同的 key/value，信息多样性受损。

**GQA（Grouped Query Attention）** 由谷歌在 2023 年提出，是 MHA 和 MQA 之间的一种折中：

```
GQA（分组查询注意力，2023）：
               Query 头    Key/Value 头
 Group 0:      Q0  Q1     K0  V0
 Group 1:      Q2  Q3     K1  V1
 
→ KV Cache 大小 = 2 × num_kv_heads × head_dim × seq_len
  如果 num_kv_heads = 2（32 个 query 头分为 2 组）
  KV Cache 缩小到原来的 2/32 = 1/16
```

### 4.2 三种注意力机制的对比

| 注意力机制 | KV 头数 | KV Cache 相对大小 | 表达能力 | 典型应用 |
|-----------|--------|:----------------:|:--------:|---------|
| **MHA** | 等于 query 头数（如 32） | 1×（基准线） | 最强 | 原始 Transformer |
| **MQA** | 1 | 1/32 | 最弱 | PaLM |
| **GQA** | 2~8（取决于分组） | 1/16 ~ 1/4 | 接近 MHA | Llama 2 70B、Llama 3、Qwen2.5 |

### 4.3 实际效果

GQA 是"免费午餐"式的优化：

- **KV Cache 降到 1/4 ~ 1/8**：同样显存可以服务更多请求或更长的上下文
- **质量损失极小**：实验表明 GQA 在几乎不损失模型质量的前提下，显著降低推理成本。这也是 Llama 2 70B（GQA-8）、Llama 3 系列（GQA-8）都在用它的原因

### 4.4 GQA 对 PagedAttention 的影响

GQA 和 PagedAttention 的关系是**正交且互补**的：

- **GQA**：在模型架构层面减少了 KV Cache 的总量
- **PagedAttention**：在运行时层面优化了 KV Cache 的分配管理

两者叠加的效果非常明显——用 GQA 让 KV Cache 需求降到 1/4，再用 PagedAttention 把它用满。

## 5. Flash Attention：Attention 计算的效率革命

### 5.1 问题：Attention 的显存瓶颈

传统的 Attention 计算过程是这样的：

```python
# 标准的 Scaled Dot-Product Attention
S = Q @ K^T              # (1) 得分矩阵，形状 [seq_len, seq_len]
P = softmax(S)           # (2) 概率矩阵
O = P @ V                # (3) 输出
```

每一步都会产生巨大的中间矩阵（得分矩阵 S 和概率矩阵 P）。以 seq_len=4096、hidden_dim=4096 为例：

- `Q @ K^T` 产生一个 `4096 × 4096 = 16M` 个元素的矩阵
- 每个元素 FP16 → 32 MB
- 再加上 softmax 的结果 → 又是一个 32 MB
- 对于 32 层，每层 32 个头 → 总中间显存 = 32 × 32 × 32 MB ≈ **32 GB**

这些中间结果被写入显存（HBM），然后在后续步骤中又被读出来。Attention 的 HBM 访问量是：

```
HBM 访问量 = O(seq_len² × head_dim)
```

而 Attention 的计算量是：

```
计算量 = O(seq_len² × head_dim)
```

看起来一样？但 GPU 的算力（FLOPs）远超显存带宽（HBM bandwidth），所以 Attention 的瓶颈**不在于算力，而在于显存读写带宽**。

### 5.2 Flash Attention 的核心思想

**Flash Attention**（Tri Dao, 2022）的洞察是：GPU 内部有**多级存储**——显存（HBM）速度慢但容量大，SRAM（片上缓存）速度快但容量小。

```
┌─────────────────────────────────────────────────┐
│  GPU 芯片                                        │
│  ┌──────────────────────────┐                    │
│  │  SRAM (片上缓存)           │  速度最快、容量最小 │
│  │  ~20 MB                   │                    │
│  └──────────────────────────┘                    │
│         ↑ ↓                                      │
│  ┌──────────────────────────┐                    │
│  │  HBM (显存)               │  速度较慢、容量大  │
│  │  24/48/80 GB             │                    │
│  └──────────────────────────┘                    │
└─────────────────────────────────────────────────┘
```

传统方案把 Q、K、V 和中间结果都放在 HBM 中，频繁读写 HBM 成为瓶颈。

Flash Attention 的核心策略：

> **利用 SRAM 做计算，避免向 HBM 写入中间结果。**

具体做法是**分块计算（Tiling）**：

1. 把 Q、K、V 分成小块（Tile），每块小到可以放进 SRAM
2. 在 SRAM 中完成一小块的 attention 计算
3. 把结果写回 HBM，加载下一块

```
传统方案：
HBM:  [Q] [K] [V] → 计算 S（写入 HBM） → 读 S → softmax（写入 P） → 读 P → 计算 O
      3 次读    2 次写 HBM    1 次读    1 次写 HBM    1 次读

Flash Attention：
HBM:  [Q tile] [K tile] [V tile] → 加载到 SRAM → 在 SRAM 中完成全部计算 → 写回 O
                                    循环处理所有 tile
      一次性全部在 SRAM 里完成（S、P 都不需要写回 HBM）
```

### 5.3 收益

| 指标 | 传统 Attention | Flash Attention | 提升 |
|------|---------------|----------------|:----:|
| HBM 读写量 | O(seq_len² × head_dim) | O(seq_len² × head_dim² / SRAM_size) | 2~4x |
| 实际训练吞吐 | 基准 | +15~40% | — |
| 推理时的 KV Cache 读取 | 可能需要多次读 | 一次性读取更多 | + |

但 Flash Attention 不只是加速——它在推理中的更重要意义是：**释放的显存可以留给 KV Cache 和更大的 batch**。

### 5.4 FA + PagedAttention：算和存的配合

Flash Attention 和 PagedAttention 分别解决 attention 的两个不同层面的问题：

```
PagedAttention（存的层面）                    Flash Attention（算的层面）
┌──────────────────────┐                    ┌──────────────────────┐
│  KV Cache 存在哪里？   │                    │  怎么用这些 KV 计算    │
│                      │                    │  attention 更高效？   │
│  → 分块管理、动态分配   │  ←── 数据流 ──→   │  → 利用 SRAM、Tiling  │
│  → 消除碎片、COW共享   │                    │  → 减少 HBM 读写     │
└──────────────────────┘                    └──────────────────────┘
```

在推理引擎中：
- PagedAttention 决定了 KV Cache 在物理显存中的布局（可能是不连续的 block）
- Flash Attention 负责读取这些 block 并在 SRAM 中高效计算
- 两者互补：PagedAttention 让显存利用率最大化，FA 让计算效率最大化

## 6. 推理引擎的完整处理流程

### 6.1 请求接入与调度

```
用户请求 → 请求排队 → 调度器
```

- 请求进入队列
- 调度器根据当前负载和资源情况决定何时处理

调度器需要回答一个问题：**下一轮迭代，应该让哪些请求参与计算？**

常见的调度策略：

| 策略 | 逻辑 | 优点 | 缺点 |
|------|------|------|------|
| **FCFS**（先来先服务） | 按到达顺序处理 | 公平 | 短请求可能被长请求阻塞 |
| **SJF**（最短作业优先） | 优先处理预期短的请求 | 降低平均延迟 | 长请求可能饿死 |
| **MLFQ**（多级反馈队列） | 短请求优先，长请求降级 | 平衡延迟和公平 | 实现复杂 |
| **带优先级的调度** | 根据请求的 SLA 级别调度 | 服务质量保障 | 需要用户标记优先级 |

现代推理引擎通常使用**基于动态规划的调度算法**，在每次迭代时考虑：
- 当前可用的显存（KV Cache 块池还有多少空闲）
- 请求的等待时间
- 请求的预期生成长度
- QoS 要求

### 6.2 预填充（Prefill）

一旦调度器决定哪些请求参与本轮迭代：

```
输入 prompt → Tokenizer → Token IDs → Embedding → Transformer 前向计算
                                                ↓
                                    生成第一个 token + 构建 KV Cache
```

预填充阶段的特点是：
- **计算密集型**：大矩阵乘法的密集计算
- **GPU 利用率高**：大批量矩阵运算，适合 GPU
- **生成 KV Cache**：把整个 prompt 的 Key/Value 缓存起来（GQA 和 PagedAttention 会影响这一步的布局）

### 6.3 解码（Decode）

从第二个 token 开始，每次只生成一个 token：

```
当前位置的输入 → Transformer 前向计算（复用 KV Cache）
                ↓
          生成下一个 token
                ↓
        KV Cache 追加新 token 的 K/V
```

解码阶段的特点是：
- **显存密集型**：主要瓶颈在 KV Cache 的读写带宽（Flash Attention 在这里发挥关键作用）
- **GPU 利用率低**：一次只算一个 token，矩阵运算较小
- **延迟敏感**：决定 TPOT 指标

这也是为什么持续批处理能显著提升解码阶段的 GPU 利用率——把多个请求的解码步骤合并成一个较大的矩阵运算。

### 6.4 推理引擎中的调度循环

```
while True:
    # 1. 检查是否有新请求，尝试加入 batch
    new_requests = scheduler.get_new_requests()
    
    # 2. 更新当前 batch 的成员
    batch = scheduler.update_batch(running_requests, new_requests)
    
    # 3. 执行一次模型前向计算
    #    同时处理 batch 中所有请求的当前 token
    #    （底层使用 PagedAttention + Flash Attention 加速）
    outputs = model.forward(batch)
    
    # 4. 处理结果
    for req in batch:
        token = outputs[req]
        req.append_token(token)
        
        if token == EOS or req.max_tokens_reached:
            # 请求完成，清理 KV Cache（block 归还到池中）
            scheduler.finish_request(req)
            response_queue.put(req.result())
        else:
            # 请求继续，等待下一轮迭代
            req.step += 1
            scheduler.keep_request(req)
    
    # 5. 内存管理：回收已完成的 block，分配新的 block
    block_manager.garbage_collect()
```

这个循环每步迭代一次（即每生成一个 token 就执行一次），是推理引擎的调度核心。

## 7. 前缀缓存（Prefix Caching）

### 7.1 解决什么问题

在实际场景中，很多请求有**共同的前缀**：

- 相同的 System Prompt："你是一个 AI 助手，擅长..."
- 相同的对话历史："用户：你好\n助手：你好！有什么可以帮助你的吗？\n用户：..."
- 相同的文档上下文：RAG 场景下，所有请求都基于同一份文档

如果没有前缀缓存，每个请求都要为这些共同前缀独立计算 KV Cache，造成严重浪费。

### 7.2 实现方式

**方式一：完全匹配**

缓存触发的 Key-Value 对，新请求到达时检测前缀是否命中缓存。

```python
# 伪代码：前缀缓存
cache = {}

def get_prompt_kv(prompt_tokens):
    # 逐段检测
    for i in range(len(prompt_tokens), 0, -1):
        prefix = tuple(prompt_tokens[:i])
        if prefix in cache:
            # 命中缓存，只需要计算剩余部分
            cached_kv = cache[prefix]
            remaining_prompt = prompt_tokens[i:]
            new_kv = compute_kv(remaining_prompt)
            return cached_kv + new_kv
    
    # 完全没有命中，从头计算
    kv = compute_kv(prompt_tokens)
    cache[tuple(prompt_tokens)] = kv
    return kv
```

**方式二：自动检测 + PagedAttention 共享**

在支持 PagedAttention 的推理引擎中，前缀缓存天然得到了支持：

- 计算请求的 prompt 的哈希值
- 相同的 prompt 前缀映射到相同的物理 block
- 通过 Block Table 的 Copy-on-Write 共享这些 block

### 7.3 实际效果

社区报告的前缀缓存效果：

| 场景 | 命中率 | TTFT 改善 | 吞吐提升 |
|------|--------|----------|---------|
| 共享 System Prompt | 100%（前缀） | 减少 30~50% | 20~30% |
| RAG + 同一文档 | 80~90%（文档部分） | 减少 60~80% | 50~100% |
| 多轮对话 | 50~80%（历史部分） | 减少 40~60% | 30~50% |

前缀缓存可以说是"投入最少，收益最大"的推理优化手段之一。

## 8. 推测解码（Speculative Decoding）

### 8.1 核心思想

LLM 的自回归解码是 **串行** 的：生成 token N+1 前必须等 token N 生成完毕。这限制了解码速度。

推测解码的思路是：

> 用一个更小的"草稿模型"（Draft Model）快速生成一串候选 token，然后用大模型并行验证。

```
传统解码（串行）：
t₁ → t₂ → t₃ → t₄ → t₅ （5 步，每步一遍大模型前向）

推测解码（并行验证）：
草稿模型（快）：一步生成 t₁' → t₂' → t₃' → t₄' → t₅'
大模型（并行）：验证 [t₁' t₂' t₃' t₄' t₅']（1 步）
如果全部接受：5 个 token 只需 1 步大模型推理
如果部分接受：接受前 k 个，丢弃从第 k+1 个开始，草稿模型以第 k 个为起点继续
```

### 8.2 为什么能加速？

大模型推理的主要瓶颈是**显存带宽**（memory-bound），而不是**算力**（compute-bound）。

- 草稿模型通常很小（例如 100M 参数），推理延迟极低
- 大模型验证候选 token 时，由于是并行计算，带宽利用更充分
- 理想情况下，推测解码可以获得 2~3x 的加速比

### 8.3 局限性

- 需要额外部署一个草稿模型
- 草稿模型的质量决定接受率（acceptance rate）
- 对分布式场景的支持有限
- 草稿模型和主模型的词表可能不一致，需要额外映射

## 9. 技术全景图

这一章我们讨论了多项推理优化技术，它们从不同层面解决不同的问题：

| 技术 | 层面 | 解决的问题 | 与 KV Cache 的关系 |
|------|------|-----------|:----------------:|
| **持续批处理** | 引擎调度 | GPU 利用率低 | 间接影响（更大 batch → 更多 KV Cache） |
| **PagedAttention** | 内存管理 | 显存碎片、浪费 | **直接管理 KV Cache 的分配** |
| **GQA** | 模型架构 | KV Cache 总量过大 | **从源头缩减 KV Cache** |
| **Flash Attention** | 计算优化 | Attention 显存带宽瓶颈 | 优化 KV Cache 的读取计算效率 |
| **前缀缓存** | 引擎策略 | 重复计算 | 共享相同前缀的 KV Cache |
| **推测解码** | 解码策略 | 串行解码延迟 | 不直接影响 |

**各技术的层级关系**：

```
  用户请求
      │
      ▼
┌──────────────────────┐
│  持续批处理调度        │  ← 决定哪些请求一起算
│  (Continuous Batching)│
└──────────┬───────────┘
           │
           ▼
┌──────────────────────┐
│  前缀缓存              │  ← 检查有没有算过的公共前缀
│  (Prefix Caching)     │
└──────────┬───────────┘
           │
           ▼
┌──────────────────────┐
│  GQA（模型架构决定）    │  ← KV Cache 总量由模型设计决定
│  (Grouped Query Attn) │
└──────────┬───────────┘
           │
           ▼
┌──────────────────────┐
│  PagedAttention       │  ← 管理 KV Cache 的物理存储
│  (Block-level 分配)   │
└──────────┬───────────┘
           │
           ▼
┌──────────────────────┐
│  Flash Attention      │  ← 如何高效计算 attention
│  (SRAM tiling 加速)   │
└──────────────────────┘
```

如果把大模型部署比作一个物流系统：

- **ch01** 讲的是仓库存储（显存规划）
- **ch02** 讲的是货物拆装（并行策略）
- **ch03** 讲的是运输网络（网络基础设施）
- **ch04** 讲的是配送中心的调度与分拣优化（推理引擎 + GQA + Flash Attention）
