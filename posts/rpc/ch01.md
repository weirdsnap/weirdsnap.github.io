# 从共享内存到 RPC：先补齐 socket 这一课

之前我们写了一些同机 IPC 的代码：共享内存、环形缓冲区、Chunk 池。这些方案的核心目标是在**同一台机器**的两个进程之间快速搬数据。

但如果要跨机器通信，或者进程不一定在同一台机器上，共享内存就无能为力了。这时候就要用到 socket。

---

## socket 是什么

Socket 编程本质上是**打开一个特殊的文件描述符，然后通过它收发数据**。和共享内存不同，socket 的 fd 背后连的是**内核网络协议栈**，而不是内存页。

共享内存像是两个人共用一块白板，写完对方直接看。socket 更像是打电话：你说一句，内核帮你传一句，对方在另一边听到。

---

## TCP Socket 建立流程

```
服务端                              客户端
─────────────────────────────────────────────────────────
socket()    ──创建fd──▶            socket()
   │                                   │
bind()      ──绑定地址──▶              │
   │                                   │
listen()    ──开始监听──▶              │
   │                                   │
accept()    ◀──等待连接──    connect()  ──发起连接──▶
   │              │                    │
recv()      ◀────收发────▶   send()     │
   │              │                    │
send()      ────▶  │        recv()    ◀──
   │              │                    │
close()     ──关闭fd──▶            close()
```

服务端先创建 fd、绑定地址、开始监听，然后阻塞在 accept 上等客户端。客户端创建 fd 后直接 connect，内核帮两边完成三次握手。连接建立后，双方就可以用 send/recv 收发数据。

---

## 关键 API 逐个看

### `socket()` — 创建通信端点

```cpp
int socket(int domain, int type, int protocol);
```

| 参数 | 含义 | 常用值 |
|---|---|---|
| `domain` | 协议族 | `AF_UNIX`（同机 UDS）、`AF_INET`（IPv4）、`AF_INET6`（IPv6） |
| `type` | 套接字类型 | `SOCK_STREAM`（TCP，可靠流式）、`SOCK_DGRAM`（UDP，数据报） |
| `protocol` | 具体协议 | 通常填 `0`，由系统根据 domain + type 自动选择 |

返回值是一个 fd（≥0），失败返回 -1。这个 fd 必须 `close()`，否则就是 fd 泄漏。中间件服务长期运行，泄漏会耗尽系统 fd 上限。

### `bind()` — 给 fd 绑定一个地址

```cpp
int bind(int sockfd, const struct sockaddr *addr, socklen_t addrlen);
```

服务端必须 bind，告诉内核“我在这个地址上提供服务”。客户端通常不需要 bind，connect 时系统会分配临时端口。

两种常见地址结构：

```cpp
// IPv4
struct sockaddr_in {
    sa_family_t    sin_family; // AF_INET
    in_port_t      sin_port;   // 端口号，必须用 htons() 转网络字节序
    struct in_addr sin_addr;   // IP 地址，INADDR_ANY = 监听所有网卡
};

// Unix Domain Socket
struct sockaddr_un {
    sa_family_t sun_family;    // AF_UNIX
    char        sun_path[108]; // 文件路径，如 "/tmp/my.sock"
};
```

注意端口号和 IP 地址要用 `htons` / `htonl` 转成网络字节序。x86 是小端，网络协议统一大端，不转就会乱。

### `listen()` — 开始监听连接请求

```cpp
int listen(int sockfd, int backlog);
```

`backlog` 是已完成三次握手、等待 accept 的连接队列最大长度。listen 只是让内核开始处理握手，真正的连接还要 accept 来取。

### `accept()` — 接受一个连接

```cpp
int accept(int sockfd, struct sockaddr *addr, socklen_t *addrlen);
```

返回一个新的 fd，专门用于和**这个客户端**通信。这里有个重要区分：

- `socket()` 创建的 fd 是“监听 fd”，只负责 accept
- `accept()` 返回的是“连接 fd”，负责 recv/send
- **两个 fd 都要 close**

默认情况下，如果没有客户端连接，accept 会阻塞。这也是后续引入 epoll 的原因。

### `connect()` — 客户端发起连接

```cpp
int connect(int sockfd, const struct sockaddr *addr, socklen_t addrlen);
```

客户端调用后，内核自动完成 TCP 三次握手。成功返回 0，失败返回 -1。

### `send()` / `recv()` — 收发数据

```cpp
ssize_t send(int sockfd, const void *buf, size_t len, int flags);
ssize_t recv(int sockfd, void *buf, size_t len, int flags);
```

这两个函数**不保证一次收发完整数据**。

- `send(fd, buf, 100, 0)` 可能只发了 30 字节，返回 30
- `recv(fd, buf, 100, 0)` 可能只读了 20 字节，返回 20

TCP 是流式协议，没有消息边界。应用层必须自己知道“一条消息多长”，循环读够为止。这就是 RPC 协议里必须有 `length` 字段的原因。

`flags` 常用值：

- `0`：默认阻塞
- `MSG_NOSIGNAL`：对端断开时不触发 `SIGPIPE` 信号，中间件通常要加这个，否则进程可能被信号杀死

### `close()` — 关闭 fd

```cpp
int close(int fd);
```

关闭 fd，释放内核资源，TCP 层面触发四次挥手。如果多个进程/线程共享 fd，需要 `shutdown()` 来强制断开。

## 一个完整的服务端示例

```cpp
#include <sys/socket.h>
#include <netinet/in.h>
#include <arpa/inet.h>
#include <unistd.h>
#include <cstdio>

int main() {
    // 1. 创建 socket
    int listen_fd = socket(AF_INET, SOCK_STREAM, 0);
    if (listen_fd < 0) { perror("socket"); return 1; }

    // 2. 绑定地址和端口
    struct sockaddr_in addr{};
    addr.sin_family = AF_INET;
    addr.sin_port = htons(8080);
    addr.sin_addr.s_addr = htonl(INADDR_ANY);

    if (bind(listen_fd, (struct sockaddr*)&addr, sizeof(addr)) < 0) {
        perror("bind");   // 常见原因：端口被占用
        return 1;
    }

    // 3. 监听
    if (listen(listen_fd, 128) < 0) { perror("listen"); return 1; }

    // 4. 接受连接（默认阻塞）
    struct sockaddr_in client_addr;
    socklen_t client_len = sizeof(client_addr);
    int conn_fd = accept(listen_fd, (struct sockaddr*)&client_addr, &client_len);
    if (conn_fd < 0) { perror("accept"); return 1; }

    // 5. 收发数据
    char buf[1024];
    ssize_t n = recv(conn_fd, buf, sizeof(buf), 0);
    if (n > 0) {
        send(conn_fd, buf, static_cast<size_t>(n), 0);  // 回显
    }

    // 6. 关闭两个 fd
    close(conn_fd);
    close(listen_fd);
    return 0;
}
```

## 常见错误与面试考点

| 错误现象 | 原因 | errno |
|---|---|---|
| `bind` 失败 | 端口被占用，或监听 <1024 端口没有 root 权限 | `EADDRINUSE` |
| `accept` 返回 -1 | 被信号中断 | `EINTR`，中间件通常要重试 |
| `recv` 返回 0 | 对端正常关闭连接 | — |
| `recv` 返回 -1 | 连接异常，或非阻塞模式下暂无数据 | `ECONNRESET`, `EAGAIN` |
| `send` 返回 -1 | 连接已断开 | `EPIPE`，配合 `MSG_NOSIGNAL` 使用 |

## 与共享内存 IPC 的 API 对照

| 能力 | 共享内存 | Socket |
|---|---|---|
| 创建/打开 | `shm_open()` | `socket()` |
| 配置大小/权限 | `ftruncate()` + `mmap()` | `bind()` |
| 等待连接 | 自己实现同步机制 | `listen()` + `accept()` |
| 发送数据 | `memcpy()` 到共享区 | `send()` |
| 接收数据 | `memcpy()` 从共享区 | `recv()` |
| 关闭 | `munmap()` + `shm_unlink()` | `close()` |

核心差异：共享内存的“连接”是隐式的，两个进程都打开同一段内存就行。socket 的“连接”是显式协议化的，内核帮你做了三次握手、四次挥手、同步和可靠性保证。

## 从 socket 到 RPC 的三个台阶

搞懂基础 API 后，代码里放了三个由浅入深的示例：

### 1. `ipc/socket/`：最基础的 TCP echo

只演示核心 API，没有处理短读短写，适合先看懂 socket 长什么样。

### 2. `ipc/minichat/`：引入协议帧

用 Unix Domain Socket 实现同机双进程聊天。核心改进：

- 协议帧：`4 字节长度头 + 变长 payload`
- `SendAll` / `RecvAll` 循环处理短读短写
- 解决 TCP/UDS 流式传输的粘包/拆包问题

### 3. `ipc/tinyrpc/`：最小 RPC 骨架

在协议帧基础上，把 payload 换成 JSON，加入方法分发：

```json
// 请求
{"method":"Add","a":1,"b":2}

// 响应
{"result":3}
```

服务端做的事情就是：接收帧 → 解析 JSON → 按 method 名字分发 → 拼响应 → 发帧。后续把 JSON 换成 protobuf，把硬编码分发换成反射或代码生成，就是一个真正的 RPC 框架。
