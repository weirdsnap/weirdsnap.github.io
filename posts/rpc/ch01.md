# 从共享内存到 RPC：先补齐 socket 这一课

之前我们写了一些同机 IPC 的代码：共享内存、环形缓冲区、Chunk 池。这些方案的核心目标是在**同一台机器**的两个进程之间快速搬数据。

但如果要跨机器通信，或者进程不一定在同一台机器上，共享内存就无能为力了。这时候就要用到 socket。

---

## 共享内存和 socket 的区别

| 维度 | 共享内存 IPC | Socket IPC |
|------|--------------|------------|
| 通信范围 | 同一台机器 | 可跨机器 |
| 数据拷贝 | 零拷贝，直接读写映射内存 | 用户态 → 内核态 → 网卡 → 对端 |
| 可靠性 | 没有内置协议，自己设计同步 | TCP 可靠传输，但应用层仍需处理断连 |
| C++ API | `shm_open` + `mmap` | `socket` + `bind` + `listen` + `accept` + `send/recv` |

简单理解：共享内存像是两个人共用一块白板，socket 像是两个人打电话。

---

## 代码里的三个台阶

我在 `ipc/` 目录下放了三个由浅入深的示例：

### 1. `ipc/socket/`：最基础的 TCP echo

这个示例只演示 socket 最核心的 API：

- `socket(AF_INET, SOCK_STREAM, 0)` 创建 socket
- `bind` 绑定地址和端口
- `listen` / `accept` 接收连接
- `read` / `write` 收发数据

它的注释里特别强调了一个容易忽略的点：**TCP 是流式协议**，`read` 一次不一定读完一条消息，`write` 一次也不一定发完所有字节。生产环境需要循环处理"短读/短写"。

### 2. `ipc/minichat/`：引入协议帧

`minichat` 用 **Unix Domain Socket（UDS）** 实现同机双进程聊天。UDS 的 API 和 TCP 几乎一样，但地址是文件路径而不是 IP:端口，同机通信时更快。

这里引入了 RPC 里非常关键的一个概念：**协议帧**。

因为 TCP/UDS 都是流，没有消息边界，所以必须自己定义消息的边界。`minichat` 的协议格式是：

```
| 4 字节长度头（网络字节序） | 变长 payload |
```

接收方先读 4 字节得到长度，再读对应长度的数据，就能准确拿到一条完整消息。这解决了**粘包/拆包**问题。

代码里有两个关键函数：

- `SendAll` / `RecvAll`：循环发送/接收指定字节数，处理短读短写
- `SendMsg` / `RecvMsg`：在 `SendAll` / `RecvAll` 基础上加上 4 字节长度头

### 3. `ipc/tinyrpc/`：最小 RPC 骨架

`tinyrpc` 在 `minichat` 的协议帧基础上，把 payload 从纯文本换成了 JSON，并加入了**方法分发**。

请求：

```json
{"method":"Add","a":1,"b":2}
```

响应：

```json
{"result":3}
```

服务端做的事情就是：

1. 接收协议帧
2. 从 JSON 里读出 method 名字和参数
3. 调用对应的处理函数
4. 把结果拼成 JSON 响应，再按协议帧发回去

这就是 RPC 的雏形。后续如果把 JSON 换成 protobuf，把硬编码的方法分发换成反射或代码生成，就是一个真正的 RPC 框架。

## 几个必须搞懂的问题

跑完这三个示例后，建议你能回答这几个问题：

1. **为什么 `send` 返回的长度可能小于你想发送的长度？怎么处理？**
2. **TCP 没有消息边界，你的应用层怎么知道"一条消息读完了"？**
3. **`htonl` / `ntohl` 是干什么用的？为什么要用网络字节序？**
4. **Unix Domain Socket 和 TCP Socket 在 API 上有什么区别？为什么 UDS 同机更快？**
5. **如果服务端 `recv` 时客户端断开，`recv` 返回什么？程序该怎么处理？**

这些问题在面试和写中间件时都会遇到。

## 下一步

搞懂 socket 和协议帧之后，就可以正式进入 TinyRPC 阶段一：

- 把 `tinyrpc` 的 JSON 换成 protobuf
- 引入服务注册和方法反射
- 加上连接池、超时、重试、心跳
- 最后把 UDS 扩展成 TCP，支持跨机调用

但不管后面怎么复杂，底层都绕不开我们今天写的这几个函数：`socket`、`bind`、`listen`、`accept`、`SendAll`、`RecvAll`。
