# 六、并发与内存模型（85–90）

---

### 85. `mutex`

`std::mutex` 是最原始的互斥锁，提供 `lock()` 和 `unlock()` 接口，但需手动配对管理；若中间发生异常或提前返回，极易导致死锁或异常泄漏。

**原理解析 + 常见误区**

- ❌ 裸锁性能最好，热路径应该用裸锁 — 错误。现代 `lock_guard` 通常零额外开销，且编译器可完全内联。
- ❌ `try_lock()` 失败后可以不管 — 错误。`try_lock()` 返回 `false` 表示未获取锁，此时访问临界区是数据竞争。

**延伸点**：`std::mutex` 的 `lock()` 与 `unlock()` 必须成对且在同一线程执行，这是锁的线程所有权要求。

---

### 86. `lock_guard`

`std::lock_guard` 是极简的 RAII 锁包装器，构造时加锁，析构时解锁，无额外状态开销。

| 特性 | 支持情况 |
|------|---------|
| 手动 `lock()`/`unlock()` | ❌ 不支持 |
| 移动语义 | ❌ 不可移动 |
| 条件变量配合 | ❌ 不支持 |
| 性能开销 | 零额外 |

**原理解析 + 常见误区**

- ❌ `lock_guard` 可以提前解锁 — 错误。它没有 `unlock()` 成员函数。
- ❌ `lock_guard` 可以配合 `std::condition_variable` — 错误。`condition_variable::wait` 要求 `std::unique_lock`。

**延伸点**：C++17 `std::scoped_lock` 是 `lock_guard` 的多锁升级版，支持同时锁定多个 `mutex` 并避免死锁。

---

### 87. `unique_lock`

`std::unique_lock` 提供可移动、可延迟加锁、可手动解锁的 RAII 封装，是唯一能与 `std::condition_variable` 配合的标准锁包装器。

| 特性 | 支持情况 |
|------|---------|
| 延迟加锁 | ✅ `std::defer_lock` |
| 手动 `unlock()` | ✅ |
| 移动语义 | ✅ |
| 条件变量 | ✅ |

**原理解析 + 常见误区**

- ❌ `unique_lock` 一定比 `lock_guard` 慢很多 — 片面。单次加解锁差异仅一个 `bool` 判断。
- ❌ `unique_lock` 析构时总是 `unlock()` — 错误。若已通过 `unlock()` 手动释放，析构不会重复调用。

**延伸点**：`unique_lock` 的移动语义允许将锁所有权安全转移给被调函数或返回给调用者，实现锁粒度细化。

---

### 88. 三者选择策略

| 场景 | 推荐选择 |
|------|---------|
| 简单临界区，作用域固定 | `lock_guard` / `scoped_lock` |
| 需条件变量 `wait`/`notify` | `unique_lock` |
| 需尝试加锁、超时、延迟加锁 | `unique_lock` |
| 需同时锁定多个 `mutex` | `scoped_lock` (C++17) |

**延伸点**：C++20 起 `std::unique_lock` 可与 `std::latch`、`std::barrier` 及 `std::atomic::wait` 等同步原语配合。

---

### 89. `std::atomic`

`std::atomic` 提供对单个数据类型的原子读写操作，保证多线程并发访问时的数据竞争自由。

| 操作 | 说明 |
|------|------|
| `load()` | 原子读 |
| `store()` | 原子写 |
| `exchange()` | 读写交换旧值 |
| `compare_exchange_weak` | 可能伪失败，适合循环 |
| `compare_exchange_strong` | 保证真失败，适合单次检查 |

**原理解析 + 常见误区**

- ❌ `std::atomic` 对象可以拷贝 — 错误。`atomic` 的拷贝构造和拷贝赋值被删除。
- ❌ `counter = counter + 1` 是原子的 — 错误。这是先 `load`、再计算、再 `store` 的三步操作。

**延伸点**：C++20 `std::atomic_ref` 允许对非 `atomic` 对象（如共享内存中的裸变量）执行原子操作。

---

### 90. `std::atomic` 内存序

从 `relaxed`（仅保证原子性）到 `seq_cst`（全局顺序一致性）形成六级强度谱系。

| 内存序 | 同步语义 | 适用场景 |
|--------|----------|----------|
| `relaxed` | 无顺序约束 | 纯计数器 |
| `acquire` | 读同步 | 读锁、标志位检查 |
| `release` | 写同步 | 写锁、标志位设置 |
| `acq_rel` | 同时 acquire + release | CAS、fetch_add |
| `seq_cst` | 全局顺序一致性 | 默认，最强但最慢 |

**原理解析 + 常见误区**

- ❌ `relaxed` 能保证其他线程看到最新值 — 片面。`relaxed` 仅保证单个原子变量的修改是原子的，不保证修改顺序的可见性。
- ❌ `acquire` 和 `release` 可以单独使用 — 错误。二者必须配对作用于同一变量才能建立 happens-before 关系。

**延伸点**：C++20 的 `std::atomic_thread_fence` 可独立于原子操作插入屏障，用于实现更细粒度的无锁数据结构。
