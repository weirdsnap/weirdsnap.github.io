# 模板推导是模式匹配，不是解方程

模板实参推导是结构模式匹配，要求模板参数出现在可直接匹配的推导上下文中；若 `T` 被嵌套在 `std::function<void(T)>` 内部，编译器无法从 lambda 类型中解出 `T`。

**场景示例：**
```cpp
// ❌ 推导失败：T 嵌套在 std::function 内部
template<typename T>
void register_cb(std::function<void(T)> cb);
register_cb([](int x) {});  // 无法推导 T

// ✅ 方案：让 T 出现在可直接推导的位置
template<typename T>
void register_cb(std::function<void(T)> cb, T sample);
register_cb([](int x) {}, 42);  // T 推导为 int
```

**原理解析 + 常见误区**

- ❌ 编译器会分析 lambda 的 `operator()` 签名来推导 `T` — 错误。模板推导只看类型层面的结构匹配，不深入分析类的成员函数签名。
- ❌ 显式指定 `T=int` 后任何 lambda 都能匹配 — 错误。lambda 的签名必须兼容 `std::function<void(int)>`。

**延伸点**：工程实践中，接口层用万能引用替代 `std::function`：`template<typename F> void register_cb(F&& cb)`，让 `F` 直接推导为 lambda 类型。
