# IFNDR：只有空包才合法的变参模板

**IFNDR（Ill-Formed, No Diagnostic Required）** 指代码在标准上是非法的，但编译器不需要诊断（可以不报错）。程序的行为是未定义的——可能编译通过、正常运行、崩溃，也可能在换编译器版本后出错。

---

## 一、一个具体的 IFNDR 例子

```cpp
template<typename... Ts>
struct X {
    X(Ts... args) : Var(0, args...) {}
    int Var;
};

int main() {
    X<> x;   // 只用了空参数包
}
```

**这段代码看起来是合法的**：
- `X<> x` 实例化 `X` 没有模板参数
- `Ts... args` 是空包，所以构造函数等价于 `X() : Var(0) {}`
- `Var(0)` 是合法的——`int` 可以从 `0` 直接初始化

**但如果带参数实例化**：
```cpp
X<int> x(1);  // 构造函数展开为：X(int arg1) : Var(0, arg1) {}
              // Var(0, arg1) 试图用两个参数初始化 int，非法
```

标准 `[temp.res.general]¶6.3` 规定：

> 如果变参模板的**每一个合法特化**都要求参数包为空，那么程序是 **ill-formed, no diagnostic required**。

因为变参模板 `X` 的唯一合法用法是空包 `<>`，它满足了"每个有效特化都需空包"的条件，因此是 IFNDR。

---

## 二、IFNDR 为什么危险

**编译器不一定会报错**。GCC 和 Clang 对上面的代码可能编译通过、运行正常，但它标准上已经是非法程序。这意味着：

- 当前编译器版本正常 → 升级后可能突然报错
- 更换编译器 → 行为可能不同
- `-O0` 正常 → `-O2` 可能因为 UB 优化出奇怪的逻辑

```cpp
// GCC 13 和 Clang 16 上，X<> x; 编译通过，运行无问题
// 但标准上说这是 IFNDR，编译器没有被要求报错
```

---

## 三、IFNDR 的常见触发场景

| 场景 | 示例 | 相关规则 |
|------|------|---------|
| 变参模板仅空包合法 | 上面的 `X<Ts...>` | `[temp.res.general]¶6.3` |
| `if constexpr` 丢弃分支有不依赖模板参数的非法语句 | 见 `ch04/25` | `[temp.res.general]¶6.4` |
| 模板定义时即有非法语法（不依赖模板参数的部分） | 类模板成员中使用了不存在的类型 | `[temp.res.general]¶6` |

---

## 四、如何避免

```cpp
// ❌ IFNDR：只有空包合法
template<typename... Ts>
struct X {
    X(Ts... args) : Var(0, args...) {}
    int Var;
};

// ✅ 正确做法：直接写非模板特化
struct X {
    X() : Var(0) {}
    int Var;
};

// ✅ 或用概念约束阻止非法实例化
template<typename... Ts>
requires (sizeof...(Ts) == 0)
struct X {
    X(Ts... args) : Var(0, args...) {}
    int Var;
};
```

## 五、常见误区

| 误区 | 正解 |
|------|------|
| ❌ 编译通过 = 代码合法 | ✅ IFNDR 不要求编译器报错，通过不代表合法 |
| ❌ 空包合法说明模板定义没问题 | ✅ 模板定义本身有问题——它设计成了一个只有空包才能用的陷阱 |
| ❌ IFNDR 只影响变参模板 | ✅ `if constexpr` 场景、不依赖模板参数的非法语句也会触发 |
| ❌ IFNDR 的程序一定会崩溃 | ✅ 可能完全正常运行，这是它最危险的地方 |

---

## 六、延伸点

- `if constexpr` 的 IFNDR 陷阱详见 [ch04/25](../25.md)。
- C++20 Concepts 可用于约束模板参数，从根源上阻止非法实例化，比依赖空包更安全。
- IFNDR 在 `[temp.res.general]¶6` 中列出了多个具体规则，变参模板空包只是其一。
