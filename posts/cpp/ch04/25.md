# if constexpr：编译期 if 与 IFNDR 陷阱

`if constexpr` 是 C++17 引入的**编译期条件分支**。它看起来像普通的 `if`，但条件表达式必须是编译期常量，满足条件的分支才会被实例化，另一分支会被“丢弃”。

这让它在模板元编程里非常方便，但也藏着一个很深的坑：你以为被丢弃的分支不会检查，实际上如果分支里有**不依赖模板参数**的非法语句，整个模板可能变成 IFNDR（Ill-Formed, No Diagnostic Required）。

---

## 基本语法

```cpp
template<typename T>
auto add_one(T t) {
    if constexpr (std::is_integral_v<T>) {
        return t + 1;               // 只对整型实例化
    } else {
        return t;                   // 否则原样返回
    }
}
```

和普通 `if` 的关键区别：

| | 普通 `if` | `if constexpr` |
|---|---|---|
| 条件 | 运行时决定 | 必须是编译期常量 |
| 分支实例化 | 两个分支都会编译 | 只编译为真的分支 |
| 典型用途 | 运行时逻辑 | 模板里按类型分派 |

---

## 常见用法：替代递归终止

在可变参数模板里，`if constexpr` 可以省掉一个专门的重载：

```cpp
template<typename T, typename... Rest>
void print(T value, Rest... rest) {
    std::cout << value;
    if constexpr (sizeof...(rest) > 0) {
        std::cout << " ";
        print(rest...);   // 还有参数时才递归
    } else {
        std::cout << "\n";
    }
}
```

当 `rest` 为空时，`print(rest...)` 这一行不会被实例化，所以不需要额外的空参数终止函数。

---

## 一个反直觉的陷阱

看下面这道题：

```cpp
#include <type_traits>
#include <iostream>
#include <string>

template<typename T>
int f()
{
    if constexpr (std::is_same_v<T, int>) { return 0; }
    else { return std::string{}; }
}

int main()
{
    std::cout << f<int>();
}
```

`f<int>()` 只会实例化 `T == int` 的版本，`if constexpr` 为真，`else` 分支被丢弃。运行结果看起来应该输出 `0`。

但正确答案不是“输出 `0`”，而是 **undefined behaviour / IFNDR**。

---

## 为什么被丢弃的分支还会让程序非法？

C++ 标准 `[temp.res.general]¶6.4` 规定：

> 如果模板定义后立刻进行一次**假想实例化**（hypothetical instantiation）时，会因为某个**不依赖模板参数**的构造而成为 ill-formed，那么程序就是 **ill-formed, no diagnostic required**。

关键就在“不依赖模板参数”。

```cpp
else { return std::string{}; }
```

这一行里的 `std::string{}` **不依赖 `T`**。标准规则要求：即使它在 `f<int>()` 里被 `if constexpr` 丢弃，编译器仍然要在模板定义期检查这个语句是否合法。

而函数 `f` 的返回类型是 `int`，`return std::string{};` 显然无法把 `std::string` 转成 `int`，所以是 ill-formed。整个模板因此变成 IFNDR。

---

## 编译器表现并不一致

IFNDR 的意思是“标准不要求编译器报错”，所以不同编译器行为可能完全不同：

- **GCC / Clang**：会直接报错，指出无法把 `std::string` 转换为 `int`。
- **MSVC**：可能直接丢弃 else 分支，编译通过并输出 `0`。

MSVC 看起来“对了”，但代码本身仍是非法的，不可移植。

---

## 怎么改才对？

核心原则：**让被丢弃分支里的非法构造也依赖模板参数**，或者干脆让它合法。

### 方案 1：让 else 分支依赖 `T`

```cpp
template<typename T>
int f()
{
    if constexpr (std::is_same_v<T, int>) { return 0; }
    else { return T{}; }   // 现在依赖 T，只在实例化非 int 时检查
}
```

这样 `f<int>()` 完全合法；只有你去调用 `f<std::string>()` 时才会报错。

### 方案 2：两个分支返回兼容类型

```cpp
template<typename T>
int f()
{
    if constexpr (std::is_same_v<T, int>) { return 0; }
    else { return 0; }
}
```

### 方案 3：让返回类型本身也模板化

```cpp
template<typename T>
auto f()
{
    if constexpr (std::is_same_v<T, int>) { return 0; }
    else { return std::string{}; }
}
```

返回类型 `auto` 会根据 `T` 推导，函数签名本身就依赖 `T`，所以不会触发 IFNDR。

---

## 常见误区

- ❌ "`if constexpr` 丢弃的分支不会被检查" — 错误。不依赖模板参数的语句仍要在模板定义期接受检查。
- ❌ "编译通过就是合法代码" — 错误。IFNDR 的可怕之处正是编译器可以不报错。
- ❌ "MSVC 输出 `0` 说明代码没问题" — 错误。它只是没诊断出来，标准上仍是未定义行为。
- ❌ "把非法语句包在 `if constexpr` 里就安全了" — 错误。要让非法部分真正依赖模板参数才安全。

---

## 一句话总结

> `if constexpr` 丢弃的分支不会被实例化，但模板定义期仍会对其中**不依赖模板参数**的语句做合法性检查。如果这种语句本身非法，程序就是 IFNDR。安全做法是：要么让 else 分支也依赖 `T`，要么保证所有分支都合法。
