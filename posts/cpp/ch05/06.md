# 多重继承下的 vtable 布局

单继承每个派生类一张 vtable，多继承则**每个基类对应一张独立的 vtable**，派生类对象中**有多个 vptr**。

---

## 一、多重继承的 vtable 结构

```cpp
struct Base1 {
    virtual void f1();
    virtual void g1();
};

struct Base2 {
    virtual void f2();
    virtual void g2();
};

struct Derived : Base1, Base2 {
    void g1() override;     // 覆盖 Base1::g1
    void f2() override;     // 覆盖 Base2::f2
    virtual void h();       // 新增虚函数
};
```

**对象内存布局（64 位系统）**：

```
Derived 对象：
┌────────────────────┐  ← this (Derived*)
│ vptr_for_Base1[0]  │  ← 指向 Derived vtable for Base1
├────────────────────┤
│ Base1 members      │
├────────────────────┤
│ vptr_for_Base2     │  ← 指向 Derived vtable for Base2 (this + offset)
├────────────────────┤
│ Base2 members      │
├────────────────────┤
│ Derived members    │
└────────────────────┘
```

**对应的 vtable**：

```
Derived::vtable for Base1:
┌──────────────────────┐
│ &Derived::f1         │ ← 未覆盖
├──────────────────────┤
│ thunk for Derived::g1│ ← 覆盖，需要 this 调整
├──────────────────────┤
│ &Derived::h          │ ← 新增虚函数（通过 Base1 指针调用）

Derived::vtable for Base2:
┌──────────────────────┐
│ &Derived::f2         │ ← 覆盖
├──────────────────────┤
│ &Base2::g2           │ ← 未覆盖
```

---

## 二、this 指针调整（Thunk）的底层细节

当通过 `Base2*` 调用 `g1()` 时，编译器需要把 `this` 从 `Base2` 子对象的起始位置**调整回** `Derived` 的起始位置：

```cpp
Derived d;
Base2* pb2 = &d;          // pb2 指向对象的 Base2 部分（非首地址）
pb2->g1();                // ❗ 内部调用的是 Derived::g1()
                          //    但 this 指针需要减偏移才能到达 Derived* 的起始
```

编译器生成的 thunk 伪代码：

```cpp
// thunk for Derived::g1 (called via Base2*)
void Derived_g1_thunk(Base2* this) {
    // 调整 this 指针：从 Base2 子对象偏移回 Derived 起始
    Derived* adjusted = reinterpret_cast<Derived*>(
        reinterpret_cast<char*>(this) - offset_of_Base2_in_Derived
    );
    // 调用真实的 Derived::g1
    Derived::g1(adjusted);
}
```

## 三、通过不同基类指针调用的内存路径

```cpp
Derived d;

Base1* pb1 = &d;
Base2* pb2 = &d;

pb1->g1();  // ✅ pb1 = &d，this 已经是 Derived* 的值，直接调用
pb2->f2();  // ✅ pb2 指向 Base2 子对象，this 本身就有偏移，无需调整
pb2->g1();  // ⚡ pb2 → thunk → 调整 this → Derived::g1
```

| 调用 | this 值 | vtable 查找 | this 是否需要调整 |
|------|---------|-------------|-----------------|
| `pb1->g1()` | `&d` | vptr_for_Base1 表 | ❌ 不需要 |
| `pb2->f2()` | `&d + offset` | vptr_for_Base2 表 | ❌ 不需要 |
| `pb2->g1()` | `&d + offset` | vptr_for_Base2 表 | ✅ 需要减偏移回到 `&d` |

## 四、`delete` 通过非最左基类指针

```cpp
Base2* pb2 = new Derived();
delete pb2;   // ⚡ 不仅要调用 ~Derived()，还要调整 this 到起始地址再 free
```

编译器生成的**析构 thunk** 会在调用析构函数后，将指针调整回 `Derived` 的起始地址再调用 `operator delete`。

---

## 五、常见误区

- ❌ 多继承的所有基类共享同一张 vtable — ✅ 每个基类有自己的 vtable 和 vptr。
- ❌ 新增虚函数只加在第一个基类的 vtable 中 — ✅ 通过最左基类指针调用时，新增函数可通过该基类的 vtable 索引访问（具体编译器策略有差异）。
- ❌ Thunk 每次调用都执行 — ✅ thunk 只在基类指针不是最左基类时才触发，单继承无此开销。
- ❌ 多继承对象大小 = 各基类之和 — ✅ 还要加上额外的 vptr 和可能的对齐填充。

---

## 六、一句话总结

> 多继承下每个基类一张独立 vtable、对象中多个 vptr，跨越基类边界的虚函数调用需要 thunk 调整 this 指针——这是多继承最主要的内存和性能开销来源。
