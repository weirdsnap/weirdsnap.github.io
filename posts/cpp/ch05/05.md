# 虚函数表（vtable）与虚指针布局

运行时多态的底层依赖两个核心数据结构：**虚函数表（virtual table, vtable）** 和 **虚指针（virtual pointer, vptr）**。它们是 C++ 实现动态绑定的关键，虽然标准没有规定具体实现方式，但几乎所有主流编译器（GCC、Clang、MSVC）都采用相同的"vtable + vptr"方案。

---

## 一、vtable 是什么

**vtable** 是一个编译器自动生成的**函数指针数组**，每个包含虚函数（或继承自虚基类）的类都有一张。

```
类 Animal 的 vtable（简略）：
┌──────────────────────────┐
│ &Animal::speak           │ ← 第 0 项
├──────────────────────────┤
│ &Animal::~Animal()       │ ← 析构函数
├──────────────────────────┤
│ &typeinfo for Animal     │ ← RTTI 信息（可选）
└──────────────────────────┘

类 Dog 的 vtable（覆盖 speak）：
┌──────────────────────────┐
│ &Dog::speak              │ ← 覆盖了 Animal::speak
├──────────────────────────┤
│ &Dog::~Dog()             │
├──────────────────────────┤
│ &typeinfo for Dog        │
└──────────────────────────┘
```

**关键规则**：
- 每个虚函数在 vtable 中有一个**固定的槽位索引**
- 派生类覆盖虚函数时，将对应槽位的指针替换为派生类版本
- 未覆盖的函数槽位指向基类版本

---

## 二、vptr 放在哪里

每个**多态对象**内部隐藏着一个 vptr，指向所属类的 vtable。

```
内存布局（64 位系统，sizeof(Animal) = 16）：
┌──────────────────┐  ← 对象起始地址
│ vptr (8 bytes)   │  ← 指向 Animal 的 vtable
├──────────────────┤
│ other members     │  ← 类的数据成员
└──────────────────┘
```

```cpp
class Animal {
public:
    virtual void speak() { std::cout << "Animal\n"; }
    virtual ~Animal() = default;
private:
    int age;           // 4 字节
};

// sizeof(Animal) ≈ 16（vptr 8 + age 4 + padding 4）
```

![vptr 在对象中的位置](images/blog/ch05_vptr_layout.png)

**注意**：vptr 通常放在对象**最前面**（GCC/MSVC 默认），但标准未规定具体位置，编译器可以选择放在任何偏移。

## 三、单继承下的虚表结构

单继承是最简单的情况，**派生类继承基类的 vtable，并覆盖需要重写的槽位**：

```cpp
class Base {
public:
    virtual void f();
    virtual void g();
    virtual void h();
};

class Derived : public Base {
public:
    void g() override;  // 只覆盖 g
    virtual void k();   // 新增虚函数
};
```

```
Base vtable:              Derived vtable:
┌─────────────┐          ┌─────────────┐
│ &Base::f    │          │ &Base::f    │ ← 未覆盖，同 Base
├─────────────┤          ├─────────────┤
│ &Base::g    │          │ &Derived::g │ ← 已覆盖，替换
├─────────────┤          ├─────────────┤
│ &Base::h    │          │ &Base::h    │ ← 未覆盖，同 Base
└─────────────┘          ├─────────────┤
                         │ &Derived::k │ ← 新增函数追加在末尾
                         └─────────────┘
```

## 四、调用虚函数的底层过程

编译期将虚函数调用翻译为：**通过对象 vptr 找到 vtable，再通过槽位索引找到函数地址间接调用**。

```cpp
// 源码
Animal* p = getAnimal();
p->speak();

// 编译器生成的伪代码（简化）
(*p->vptr[0])(p);   // vptr[0] 即 speak 槽位
```

这个过程的开销：
- 一次读取 vptr（访存）
- 一次索引计算（偏移已知，编译期固定）
- 一次间接函数调用（无法内联）

现代 CPU 的分支预测和缓存命中率让这个开销通常只有**1~2 条额外指令**，远没有很多人想象的那么昂贵。

## 五、常见误区

| 误区 | 正解 |
|------|------|
| ❌ 所有对象共享同一张 vtable | ✅ 每个对象有自己的 vptr，但同类的所有对象共享同一张 vtable（vptr 指向同一地址） |
| ❌ vtable 是标准规定的 | ✅ 标准只要求行为（动态绑定），vtable 是实际实现细节，所有编译器都这么用但非强制 |
| ❌ 虚函数越多性能越差 | ✅ vtable 查找是 O(1) 的索引访问，与虚函数数量无关 |
| ❌ vptr 只有 4 字节 | ✅ 64 位系统上是 8 字节，32 位是 4 字节 |
| ❌ 基类构造期间 vptr 已经指向派生类对象 | ✅ 构造期间 vptr 先指向基类 vtable，进入派生类构造体后才切换为派生类 vtable |

---

## 六、延伸点

- **多继承下的虚表**：每个基类对应一张虚表，派生类对象中**有多个 vptr**，`this` 指针在调用不同基类方法时需要调整（thunk 机制）。
- **虚继承的复杂布局**：虚继承引入 **vbptr（virtual base pointer）** 和 vbtable，对象布局比单继承复杂得多。
- **`__attribute__((noprintf))` / 编译器特定选项**：部分编译器提供控制 vtable 生成的扩展属性。
- 构造函数中调虚函数不会动态派发到派生类，因为**进入基类构造体时 vptr 还没有切换**。
