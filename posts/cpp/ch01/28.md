# 整数提升与通常算术转换

`char`、`short`、`unsigned short` 这些比 `int` 窄的整数类型，在参与算术运算时**不会用自己的类型直接计算**。标准会先对它们进行**整数提升（integral promotion）**，提升到 `int` 或 `unsigned int`，然后再执行运算。

这解释了两个非常反直觉的题目：

- `unsigned short` 相减，结果可能是负数 `-1`。
- 两个 `char` 相加，调用的重载函数可能是 `f(int)`，也可能是 `f(unsigned int)`。

---

## 整数提升规则

C++23 标准 `[conv.prom]¶1` 规定：

> 一个整数类型（除了 `bool`、`char8_t`、`char16_t`、`char32_t`、`wchar_t`）的 prvalue，如果其整数转换等级低于 `int`，并且 `int` 能表示该类型的所有值，则它可以被转换为 `int`；否则转换为 `unsigned int`。

关键取决于两点：

1. **转换等级（conversion rank）**：`[conv.rank]` 规定 `char`、`signed char`、`unsigned char`、`short`、`unsigned short` 的等级都低于 `int`。
2. **`int` 能否表示原类型的全部值**：这由具体平台的类型宽度决定。

---

## 例子 1：`unsigned short` 的减法

```cpp
#include <iostream>

int main() {
    unsigned short zero = 0, one = 1;
    if (zero - one < zero)
        std::cout << "less";
    else
        std::cout << "more";
}
```

输出是 **implementation-defined（实现定义）** 的：

- 大多数平台：`unsigned short` 是 16 位，`int` 是 32 位。`unsigned short` 能提升为 `int`，因为 `int` 能表示 0~65535。
  - `zero - one` 变成 `int` 的 `0 - 1 == -1`。
  - `-1 < 0` 为真，输出 `less`。

- 如果 `unsigned short` 和 `int` 一样宽（某些特殊实现）：`int` 放不下所有 `unsigned short` 的值，提升目标是 `unsigned int`。
  - `0u - 1u` 按无符号回绕，变成很大的正数。
  - 大正数 `< 0` 为假，输出 `more`。

所以不要以为“无符号类型相减一定会回绕”。是否回绕，取决于表达式最终被提升成什么类型。

## 例子 2：两个 `char` 相加后的重载选择

```cpp
#include <iostream>

void f(unsigned int) { std::cout << "u"; }
void f(int)          { std::cout << "i"; }
void f(char)         { std::cout << "c"; }

int main() {
    char x = 1;
    char y = 2;
    f(x + y);
}
```

`x + y` 的类型**不是 `char`**。两个 `char` 先经过整数提升，再相加。

- 常见平台：`char` 是 8 位且有符号，`int` 是 32 位。
  - `char` 提升为 `int`。
  - `x + y` 类型是 `int`。
  - 调用 `f(int)`，输出 `i`。

- 如果 `char` 是 **无符号**，并且 `char` 与 `int` 一样宽：
  - `int` 无法表示所有 `unsigned char` 的值，提升目标是 `unsigned int`。
  - `x + y` 类型是 `unsigned int`。
  - 调用 `f(unsigned int)`，输出 `u`。

> 注意：这里 `char` 的符号性本身就是实现定义的（`[basic.fundamental]¶7`），所以在某些平台上，即使 `char` 和 `int` 宽度不同，只要 `char` 是 unsigned，提升后仍可能是 `unsigned int`（当 `int` 放不下 unsigned char 时）。

## 通常算术转换

整数提升只是第一步。对于 `+`、`-`、`*`、`/` 等二元算术运算符，标准还会执行**通常算术转换（usual arithmetic conversions）**，目的是把两个操作数转换成一个共同类型，这个共同类型也是结果的类型（`[expr.arith.conv]¶1.3`）。

对于整型，基本规则是：

1. 先对两个操作数分别做整数提升。
2. 如果提升后类型相同，就是结果类型。
3. 如果不同，按等级和符号性进一步转换，常见结果：
   - 同符号：取宽度更大的类型。
   - `unsigned int` 与 `int`：如果 `int` 能表示 `unsigned int` 所有值，则结果为 `int`；否则 `int` 转成 `unsigned int`。
   - 更一般情况参见标准转换层级。

典型的坑：

```cpp
unsigned int u = 10;
int i = -42;
auto x = u + i;   // i 先转成 unsigned int，结果是一个很大的正数
```

---

## 常见误区

- ❌ "`short + short` 的结果是 `short`" — 错误。表达式里会先提升到 `int` 或 `unsigned int`。
- ❌ "`char + char` 的结果是 `char`" — 错误。结果至少是 `int` 或 `unsigned int`。
- ❌ "无符号类型运算一定回绕" — 错误。如果提升成了有符号的 `int`，就不会回绕。
- ❌ "浮点转整数是四舍五入" — 错误。小数部分直接截断（虽然这和整数提升无关，但是同一类类型转换陷阱）。

## 工程建议

1. **需要精确控制类型时显式转换**：
   ```cpp
   auto r = static_cast<unsigned short>(zero - one);  // 强制回绕到 16 位
   ```

2. **混合有符号/无符号时特别小心**：
   - 用 `std::make_unsigned_t` / `std::make_signed_t` 统一符号性。
   - 或先提升到更大类型计算，再判断范围。

3. **跨平台代码不要依赖 `char` 的符号性和窄类型的宽度**：
   - 明确使用 `int8_t` / `uint8_t` / `int16_t` / `uint16_t`。
   - 避免对 `char` 做算术后重载选择敏感的操作。

---

## 一句话总结

比 `int` 窄的整数类型在表达式里不会“原样运算”。标准会先进行整数提升，必要时再做通常算术转换。这个规则让 `unsigned short` 减法和 `char` 加法的结果严重依赖平台，是面试题和跨平台 bug 的高发区。
