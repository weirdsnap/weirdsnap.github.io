# `extern "C"` 工程陷阱与类型系统暗礁

`extern "C"` 中的 `extern` 是**链接说明（linkage specification）**，它只改变**语言链接**（Name Mangling 开关），**不是**变量的 `extern` 存储类说明符。这两个 `extern` 同名不同义，是工程中链接报错的重灾区。

---

## 一、核心陷阱：块内变量是定义，不是声明

```cpp
extern "C" {
    int global_var;   // ⚠️ 这是定义！不是声明！
}
```

很多人看到 `extern` 关键字就误以为 `global_var` 只是声明，实际上它和写在全局命名空间的 `int global_var;` 完全一样——**会分配存储空间，是强符号定义**。

### 函数 vs 变量的不对称性

| 实体 | 在 `extern "C" {}` 中 | 含义 |
|------|----------------------|------|
| `void foo();` | 函数原型 | **声明**（无函数体） |
| `void foo() {}` | 函数实现 | **定义** |
| `int x;` | 变量 | **定义**（分配存储） |
| `int x = 42;` | 变量 | **定义**（显式初始化） |
| `extern int x;` | 变量 | **声明**（不分配存储） |

**关键区别**：函数天然有"有无函数体"的语法区分，而变量没有。变量必须显式加 `extern` 才是声明。

### 头文件中的经典翻车

```cpp
// mylib.h（错误示范）
extern "C" {
    int g_version;      // ❌ 灾难：每个包含此头文件的 .cpp 都会定义一个强符号
    void init();
}
// a.cpp #include "mylib.h" → 定义了 g_version
// b.cpp #include "mylib.h" → 又定义了 g_version
// 链接错误：multiple definition of `g_version'
```

```cpp
// mylib.h（正确写法）
extern "C" {
    extern int g_version;  // ✅ 显式 extern，纯声明
    void init();
}
// mylib.cpp
int g_version = 1;        // ✅ 唯一定义处
```

## 二、C 链接变量跨命名空间合并

C 语言链接的另一个特殊规则：**C 链接变量在不同命名空间中声明时，指向的是同一个实体**。

```cpp
#include <iostream>

namespace A {
    extern "C" { int x; }
};

namespace B {
    extern "C" { int x; }
};

int A::x = 0;

int main() {
    std::cout << B::x;  // 0
    A::x = 1;
    std::cout << B::x;  // 1
}
```

`A::x` 和 `B::x` 是同一个变量。根据 `[dcl.link]¶7`，如果两个声明引入了相同名字的 C 语言链接实体，它们声明的是**同一个实体**——命名空间不影响 C 链接符号的身份。

结合上一条规则：`extern "C" { int x; }` 块内变量是**定义**。因此以上代码中 `A::x` 和 `B::x` 是同一个变量的两次定义，触发 `multiple definition` 链接错误。

### 工程启示

- 不要在 `extern "C" {}` 中定义变量（除非用 `static` 限制到当前翻译单元）
- `extern "C"` 变量是全局的，不要放在命名空间中指望隔离
- 如果只需要变量声明，块内显式加 `extern`：

```cpp
extern "C" {
    extern int x;  // ✅ 纯声明，不定义
}
```

## 三、`extern "C"` 下不能用的 C++ 特性

C 语言不支持的特性，自然不能在 `extern "C"` 块内使用：

```cpp
extern "C" {
    void foo(int);
    void foo(double);   // ❌ 重载：C 不支持

    void bar(int&);     // ❌ 引用参数：C 没有引用

    void baz(int = 0);  // ❌ 默认参数：C 不支持

    template<typename T>
    void f(T t);        // ❌ 模板：C 没有模板

    struct S { void f(); };  // ✅ 结构体定义允许
    // void S::f() {}        // ❌ 成员函数不能有 C 链接
}
```

## 四、函数指针的语言链接不兼容

函数类型包含语言链接信息。`extern "C"` 函数的类型与 C++ 函数的类型不同：

```cpp
extern "C" void c_callback();
void cpp_callback();

void (*pf1)() = c_callback;     // ⚠️ 可能仅警告：函数类型语言链接不匹配
void (*pf2)() = cpp_callback;   // ✅ OK

// 正确的声明方式：在函数指针类型上也带 extern "C"
extern "C" typedef void (*c_fn_t)();
c_fn_t cb = c_callback;         // ✅ OK
// cb = cpp_callback;           // ❌ 编译错误：类型不匹配
```

典型场景：C 库的回调注册接口。

## 五、典型链接错误排查

当遇到 `undefined reference` 时，用 `nm` 检查符号表：

```bash
nm main.o | grep foo
# C++ 链接：00000000 T _Z3foo_i（mangled）
# C 链接：   00000000 T foo      （unmangled）
```

| 场景 | C++ 符号名 | C 符号名 |
|------|-----------|---------|
| `void foo()` | `_Z3foov` | `foo` |
| `void foo(int)` | `_Z3fooi` | `foo`（C 无法区分） |
| `ns::foo()` | `_ZN2ns3fooEv` | — |

## 六、C 头文件被 C++ 包含的惯用法

C 头文件必须自我保护：

```cpp
// mylib.h
#ifdef __cplusplus
extern "C" {
#endif

void init_engine();
int  get_status();

#ifdef __cplusplus
}
#endif
```

> `__cplusplus` 宏只在 C++ 编译器中定义，C 编译器看不到 `extern "C"` 语法。

---

## 七、一句话总结

> `extern "C"` 管的是"怎么编译符号名"，不管"是不是分配内存"。块内的变量必须显式加 `extern` 才是声明；C 链接变量跨命名空间视为同一实体；函数指针类型也带语言链接信息，混用可能导致 UB。
