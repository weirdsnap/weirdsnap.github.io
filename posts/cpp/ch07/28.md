# `extern "C"` 工程陷阱与类型系统暗礁

`extern "C"` 中的 `extern` ≠ 变量的 `extern`。这是工程实践中链接报错的重灾区。

---

## 一、核心陷阱：`extern "C"` 不改变声明/定义属性

`extern "C"` 中的 `extern` 是**链接说明（linkage specification）**的一部分，它只改变**语言链接**（Name Mangling 开关），**不是**变量的 `extern` 存储类说明符。

```cpp
extern "C" {
    int global_var;   // ⚠️ 这是定义！不是声明！
}
```

很多人看到 `extern` 关键字就误以为 `global_var` 只是声明，实际上它和写在全局命名空间的 `int global_var;` 完全一样——**会分配存储空间，是强符号定义**。

### 函数 vs 变量的不对称性

| 实体 | 在 `extern "C" {}` 中 | 含义 |
|------|----------------------|------|
| `void foo();` | 函数原型 | **声明**（无函数体） |
| `void foo() {}` | 函数实现 | **定义** |
| `int x;` | 变量 | **定义**（分配存储） |
| `int x = 42;` | 变量 | **定义**（显式初始化） |
| `extern int x;` | 变量 | **声明**（不分配存储） |

**关键区别**：函数天然有"有无函数体"的语法区分，而变量没有。变量必须显式加 `extern` 才是声明。

### 头文件中的经典翻车

```cpp
// mylib.h（错误示范）
extern "C" {
    int g_version;      // ❌ 灾难：每个包含此头文件的 .cpp 都会定义一个强符号
    void init();
}
// a.cpp #include "mylib.h" → 定义了 g_version
// b.cpp #include "mylib.h" → 又定义了 g_version
// 链接错误：multiple definition of `g_version'
```

```cpp
// mylib.h（正确写法）
extern "C" {
    extern int g_version;  // ✅ 显式 extern，纯声明
    void init();
}
// mylib.cpp
int g_version = 1;        // ✅ 唯一定义处
```

## 二、`extern "C"` 下不能用的 C++ 特性

```cpp
extern "C" {
    void foo(int);
    void foo(double);   // ❌ 重载：C 不支持

    void bar(int&);     // ❌ 引用参数：C 没有引用

    void baz(int = 0);  // ❌ 默认参数：C 不支持

    template<typename T>
    void f(T t);        // ❌ 模板：C 没有模板

    struct S { void f(); };  // ✅ 结构体定义允许
    // void S::f() {}        // ❌ 成员函数不能有 C 链接
}
```

## 三、函数指针的语言链接不兼容

这是最隐蔽的陷阱——**函数类型包含语言链接信息**。`extern "C"` 函数的类型与 C++ 函数的类型不同。

```cpp
extern "C" void c_callback();
void cpp_callback();

void (*pf1)() = c_callback;     // ⚠️ 可能仅警告：函数类型语言链接不匹配
void (*pf2)() = cpp_callback;   // ✅ OK

// 正确的声明方式：在函数指针类型上也带 extern "C"
extern "C" typedef void (*c_fn_t)();
c_fn_t cb = c_callback;         // ✅ OK
// cb = cpp_callback;           // ❌ 编译错误：类型不匹配
```

典型场景：C 库的回调注册接口。

## 四、典型链接错误排查

当遇到 `undefined reference` 时，用 `nm` 检查符号表：

```bash
nm main.o | grep foo
# C++ 链接：00000000 T _Z3foo_i（mangled）
# C 链接：   00000000 T foo      （unmangled）
```

| 场景 | C++ 符号名 | C 符号名 |
|------|-----------|---------|
| `void foo()` | `_Z3foov` | `foo` |
| `void foo(int)` | `_Z3fooi` | `foo`（C 无法区分） |
| `ns::foo()` | `_ZN2ns3fooEv` | — |

## 五、C 头文件被 C++ 包含的惯用法

C 头文件必须自我保护：

```cpp
// mylib.h
#ifdef __cplusplus
extern "C" {
#endif

void init_engine();
int  get_status();

#ifdef __cplusplus
}
#endif
```

> `__cplusplus` 宏只在 C++ 编译器中定义，C 编译器看不到 `extern "C"` 语法。

---

## 六、一句话总结

> `extern "C"` 管的是"怎么编译符号名"，不管"是不是分配内存"。块内的变量必须自己显式加 `extern` 才是声明；函数指针类型也带语言链接信息，混用可能导致 UB。
