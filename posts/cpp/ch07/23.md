# 性能开销：与零拷贝方案的对比

`stringstream` 每次 `<<` 可能触发虚函数调用、locale 查询和动态分配；`std::to_chars` / `std::from_chars` 零异常、零分配，热路径上快一个数量级。

| 方案 | 动态分配 | 虚函数 | 异常安全 | 相对速度 | 适用场景 |
|------|---------|--------|---------|---------|---------|
| `std::ostringstream` | 多次（字符串增长） | ✅ 有 | 可能抛 | 基准 | 复杂格式化、多类型混合 |
| `std::istringstream` | 构造时拷贝输入 | ✅ 有 | 可能抛 | 基准 | 灵活解析、多类型提取 |
| `std::to_chars` | **无** | ❌ 无 | **无异常** | 10x+ | 数字转字符串热路径 |
| `std::from_chars` | **无** | ❌ 无 | **无异常** | 10x+ | 字符串转数字热路径 |
| `std::format` | 可能有 | ❌ 无 | 可能抛 | 5x+ | 现代类型安全格式化 |

```cpp
#include <sstream>
#include <charconv>
#include <array>

// 低效：动态分配 + 虚函数
std::string slow(int n) {
    std::ostringstream oss;
    oss << n;
    return oss.str();
}

// 高效：栈缓冲区，零分配
std::string_view fast(int n) {
    static thread_local std::array<char, 20> buf;
    auto [ptr, ec] = std::to_chars(buf.data(), buf.data() + buf.size(), n);
    return std::string_view(buf.data(), ptr - buf.data());
}
```

**原理解析 + 常见误区**

- ❌ `stringstream` 和 `to_chars` 性能差不多 — 错误。`to_chars` 通常快一个数量级以上。
- ❌ 所有场景都该用 `to_chars` — 错误。复杂格式化（如同时输出多种类型、控制精度宽度）用 `stringstream` 或 `std::format` 更方便。
- ❌ `from_chars` 会跳过前导空白 — 错误。它不跳过空白，失败时 `ptr` 指向第一个非法字符。

**延伸点**：如果必须使用 `stringstream` 进行高频操作，在循环外构造对象并复用（`str("")` + `clear()`）可减少分配开销。但即便如此，locale 和虚函数开销仍无法消除。日志系统热路径上应优先 `to_chars` 或 `fmt::format_to`（写入预分配缓冲区）。
