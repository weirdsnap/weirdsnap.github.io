# 二、对象模型与内存布局（23–42）

---

### 23. 虚函数的底层支持（vtable / vptr）

含虚函数的类有虚表（vtable）存储虚函数地址，每个对象有虚表指针（vptr）指向虚表；vptr 位置是 ABI 实现细节。

| 层级 | 实体 | 数量 |
|------|------|------|
| Per-class | vtable | 每个含虚函数的类一个 |
| Per-object | vptr | 单继承通常 1 个，多重继承多个 |

**原理解析 + 常见误区**

- ❌ vptr 一定在对象首部 — 错误。这是 ABI 实现细节，非标准保证。
- ❌ vtable 只存函数指针 — 错误。典型实现还包含 RTTI 指针和偏移修正信息。

**延伸点**：构造/析构时 vptr 按继承链动态切换，这是基类构造/析构中调用虚函数无法到达派生类实现的底层原因。

---

### 24. 构造顺序与 vptr 切换

构造时基类子对象先构造，vptr 逐步切换；进入基类构造函数时，vptr 指向基类虚表。

| 阶段 | vptr 指向 | 虚函数调用结果 |
|------|-----------|--------------|
| 基类构造函数中 | `Base::vtable` | `Base::foo` |
| 派生类构造函数体中 | `Derived::vtable` | `Derived::foo` |

**原理解析 + 常见误区**

- ❌ 构造时 vptr 一开始就指向最终派生类 — 错误。进入基类构造函数前，vptr 已初始化为基类虚表。
- ❌ 析构中还能调用派生类虚函数 — 错误。基类析构中 vptr 已回切为基类虚表。

**延伸点**：若需根据派生类类型在构造期执行不同初始化，应使用 CRTP 静态多态或将初始化推迟到构造完成后。

---

### 25. 析构顺序与 vptr 回切

析构是构造的严格逆序：派生类析构函数体执行时 vptr 指向派生类虚表，完成后回切为基类虚表。

**原理解析 + 常见误区**

- ❌ 析构函数通常设为虚函数 — 这是正确做法，但很多人不知道为什么。若基类析构非虚，通过基类指针 `delete` 派生类对象时，派生类析构不会执行，导致资源泄漏。

**延伸点**：虚析构确保 vptr 机制正确分派到派生类析构，这是多态基类的强制要求。

---

### 26. 构造/析构中调用虚函数

构造/析构期间 vptr 切换导致调用虚函数始终绑定到当前正在构造/销毁的类版本，无法到达最终派生类。

**场景示例：**
```cpp
class Base {
public:
    Base() { foo(); }  // 调用 Base::foo，不是 Derived::foo
    virtual void foo() {}
};
```

**原理解析 + 常见误区**

- ❌ 构造函数中调用虚函数能实现多态初始化 — 错误。这是最常见的误解。
- ❌ 显式写 `Base::foo()` 就能绕过限制 — 错误。显式调用本来就是静态绑定。

**延伸点**：若需多态初始化逻辑，将逻辑移至工厂函数或独立的初始化方法，或使用 CRTP 编译期静态分派。

### 27. 虚函数的默认参数

虚函数默认参数采用静态绑定，由调用时指针/引用的声明类型决定，而非对象的实际动态类型。

**场景示例：**
```cpp
class Base { public: virtual void foo(int x = 10) {} };
class Derived : public Base { public: void foo(int x = 20) override {} };
Base* p = new Derived();
p->foo();  // 调用 Derived::foo，但默认参数是 10（Base 的版本）
```

**原理解析 + 常见误区**

- ❌ 重写虚函数时默认参数也会覆盖 — 错误。默认参数不参与多态。
- ❌ 默认参数不一致会导致编译错误 — 错误。编译器允许基类与派生类默认参数不同，且不警告。

**延伸点**：绝不在虚函数中使用默认参数；若需多态行为配合不同默认值，改用重载的非虚函数作为接口入口。

### 28. NVI 惯用法

公开接口声明为非虚函数，内部委托给私有/保护的虚函数实现，由基类控制流程，派生类仅定制行为。

**场景示例：**
```cpp
class Base {
public:
    void algorithm() {  // 非虚，签名稳定
        preProcess();
        doAlgorithm();  // 虚函数定制点
        postProcess();
    }
protected:
    virtual void doAlgorithm() {}
};
```

**原理解析 + 常见误区**

- ❌ 派生类不能重写私有虚函数 — 错误。C++ 中访问权限检查在编译期，虚函数分派在运行期；派生类重写 `private virtual` 完全合法。
- ❌ NVI 消除了虚函数开销 — 错误。虚函数调用开销仍在，只是从公开接口转移到内部。

**延伸点**：NVI 与 CRTP 结合可将虚函数替换为静态多态，在编译期完成分派，消除虚表和运行时开销。

---

### 29. `vptr` 与 `vbptr` 的区别

`vptr` 用于虚函数分派，`vbptr` 用于虚继承中共享基类定位；逻辑独立，不同 ABI 实现方式不同。

| 维度 | vptr | vbptr |
|------|------|-------|
| 触发条件 | 含虚函数 | 含虚继承 |
| 指向内容 | 虚函数表 | 虚基类偏移表 |
| 单继承对象 | 通常 1 个 | 0 或 1（取决于 ABI） |

**原理解析 + 常见误区**

- ❌ vbptr 和 vptr 是同一个东西 — 错误。逻辑独立；某些 ABI 将虚基类偏移存放在 vtable 负索引槽位，但语义仍是两个机制。
- ❌ 有虚函数就一定有 vbptr — 错误。只有虚继承才触发 vbptr。

**延伸点**：不同编译器 ABI 实现差异：MSVC 倾向独立 vbptr；GCC/Clang（Itanium ABI）复用 vtable 负索引。

---

### 30. 同时含 `vptr` 与 `vbptr` 的对象布局

类同时有虚函数和虚继承时，对象内同时存在 vptr 和 vbptr（或等效机制），具体布局因 ABI 而异。

**原理解析 + 常见误区**

- ❌ vptr 和 vbptr 总是同时存在 — 错误。只有同时满足"有虚函数"和"有虚继承"时才需要两者。
- ❌ 通过 B* 访问 A 成员和通过 D* 访问一样快 — 错误。通过 B* 访问需查 vbptr 获取偏移，多一次间接寻址。

**延伸点**：多重继承 + 虚继承时，对象内 vptr 数量爆炸，是 COM/ABI 边界设计中避免虚继承的根本原因。

---

### 31. `this` 指针调整（Thunk）

Thunk 是编译器在多重继承/虚继承场景下生成的底层代码片段，用于在虚函数分派前调整 `this` 指针。

| 场景 | 是否需要 Thunk |
|------|---------------|
| 单继承虚函数覆盖 | 通常不需要 |
| 多重继承非最左基类虚函数 | ✅ 需要 |
| 虚继承中通过基类指针调用虚函数 | ✅ 需要 |

**原理解析 + 常见误区**

- ❌ Thunk 是程序员可以显式调用的函数 — 错误。它是编译器生成的内部代码片段。
- ❌ Thunk 带来巨大性能开销 — 错误。通常只是一两条指令（`lea`/`add` + `jmp`）。

**延伸点**：通过非最左基类指针 `delete` 派生类对象时，Thunk 不仅要调整 this 指针，还需确保调用正确的 `operator delete` 释放地址。

---

### 32. 虚继承与菱形继承

虚继承解决菱形继承导致的基类多份拷贝问题；最底层派生类中只有一份共享虚基类实例，通过 vbptr 访问。

| 场景 | A 的份数 | 访问方式 |
|------|---------|---------|
| 普通多重继承 | 2 份 | 直接偏移 |
| `virtual public A` | 1 份 | vbptr + 偏移表 |

**原理解析 + 常见误区**

- ❌ 虚继承消除了多重继承的所有问题 — 错误。引入了更复杂的内存布局、更大的对象体积、更慢的访问速度。
- ❌ 虚基类由中间派生类构造 — 错误。虚基类始终由最终派生类直接构造。

**延伸点**：虚基类必须由最终派生类构造，中间类的基类初始化列表中对虚基类的构造被编译器忽略。

---

### 33. 虚继承的 `vbptr` 与偏移量表

虚继承下，编译器为每个含虚基类的子对象生成虚基类偏移表，记录到共享虚基类子对象的偏移量。

**原理解析 + 常见误区**

- ❌ vbptr 和 vptr 是同一个东西 — 错误。vptr 用于虚函数分派，vbptr 用于虚基类定位。
- ❌ 虚继承后访问虚基类和普通继承一样快 — 错误。普通继承是编译期固定偏移，虚继承需运行时查表。

**延伸点**：不同 ABI 实现差异：MSVC 采用独立 vbptr 指向 vbase table；Itanium ABI 将虚基类偏移存放在 vtable 负索引槽位。

---

### 34. 空类的虚继承大小

空类在普通继承下大小至少为 1 字节；引入虚继承后，即使基类为空，派生类也需承担虚基类定位机制的额外开销，且空基类优化（EBO）通常失效。

| 场景 | 空类大小 | 派生类额外开销 |
|------|---------|---------------|
| 普通空类 | 1 字节 | 无 |
| 非虚继承空基类 | 0（EBO 合并） | 无 |
| **虚继承空基类** | **1 字节** | **vptr/vbptr 管理** |

**原理解析 + 常见误区**

- ❌ 空类虚继承也不占空间 — 错误。虚继承的元数据开销通常远大于空类本身的 1 字节。
- ❌ EBO 在虚继承中仍然有效 — 错误。虚基类需要独立地址和统一偏移定位，无法零大小合并。

**延伸点**：虚继承与 EBO 的冲突是库设计中避免虚继承空基类的根本原因；非虚继承的空基类是零开销抽象的关键。

### 35. 成员函数模板不能是虚函数

C++ 禁止将成员函数模板声明为 `virtual`，因为虚表需要在编译期确定固定大小，而模板函数可实例化的数量在理论上是无限的。

**原理解析 + 常见误区**

- ❌ 函数模板可以偏特化 — 错误。函数模板不支持偏特化，与类模板不同。
- ❌ 全特化直接参与重载竞争 — 错误。全特化不是重载，而是对主模板特定实例的替代。

**延伸点**：函数模板实现"偏特化效果"的正确方式是重载 + SFINAE / C++20 Concepts 约束。

---

### 36. 虚函数内联的去虚拟化

虚函数通常通过虚表间接分派，但当编译器能在编译期确定对象的动态类型时，可退化为静态绑定并内联展开。

| 调用方式 | 能否确定动态类型 | 是否可内联 |
|----------|-----------------|-----------|
| 对象直接调用 `obj.foo()` | ✅ 能 | ✅ 可内联 |
| 基类指针/引用 `p->foo()` | ❌ 通常不能 | ❌ 通常不可 |
| `final` 类对象调用 | ✅ 无派生类 | ✅ 可内联 |

**原理解析 + 常见误区**

- ❌ `inline` 关键字能让虚函数内联 — 错误。`inline` 只是链接期提示，不影响编译器优化决策。
- ❌ 所有虚调用都有虚开销 — 错误。对象直接调用、`final` 类调用、构造/析构中的调用均无运行时虚分派开销。

**延伸点**：LTO（链接时优化）与 PGO（配置文件引导优化）可在链接期/运行期识别高频类型，优先去虚拟化。

---

### 37. `final` 与编译器优化

`final` 修饰类或虚函数时，向编译器承诺不存在进一步派生或覆盖，使编译器可将虚调用优化为静态绑定。

| `final` 用法 | 语义约束 | 优化机会 |
|-------------|---------|---------|
| `class T final` | 禁止继承 | 所有 `T` 对象虚调用可静态化 |
| `virtual void f() final` | 禁止覆盖 | 该虚函数无后续分派 |

**原理解析 + 常见误区**

- ❌ `final` 强制编译器去虚拟化 — 错误。`final` 仅提供优化可能性，编译器是否利用取决于优化等级。
- ❌ `final` 类中的虚函数不再是虚函数 — 错误。`final` 不改变虚函数语义，仍占用 vtable 槽位。

**延伸点**：若追求绝对零开销内联，应使用 CRTP 静态多态替代虚函数；`final` 是保留多态能力但提示优化的折中方案。

### 38. Pimpl 惯用法

将私有实现通过不透明指针转移到实现文件中，头文件仅暴露接口，隐藏实现、隔离编译依赖并保持 ABI 稳定。

**场景示例：**
```cpp
// Widget.h
class Widget {
    class Impl;
    std::unique_ptr<Impl> pImpl;
public:
    Widget();
    ~Widget();  // 必须在 cpp 中定义
    void draw();
};
```

**原理解析 + 常见误区**

- ❌ Pimpl 用裸指针 `Impl* pImpl` 更好 — 错误。现代 C++ 应使用 `std::unique_ptr`。
- ❌ 析构函数可以省略或在头文件中 `= default` — 错误。`unique_ptr` 的删除器需要 `Impl` 的完整定义。

**延伸点**：Pimpl 是 Bridge 模式的 C++ 特化；若需支持多平台实现切换，可引入虚函数，但会丧失部分编译隔离优势。

### 39. ABI 与二进制兼容

ABI 规定对象内存布局、虚表结构、调用约定等编译器/平台相关细节；ABI 稳定意味着修改实现后已编译的二进制库无需重新链接。

| 维度 | API | ABI |
|------|-----|-----|
| 关注点 | 函数签名、头文件 | 对象布局、虚表、调用约定 |
| 修改私有成员 | 不影响 | 可能破坏（改变对象大小/偏移） |
| 加虚函数 | 通常兼容源码 | 改变虚表大小/布局，破坏 ABI |

**原理解析 + 常见误区**

- ❌ API 不变就是 ABI 不变 — 错误。修改私有成员类型，API 完全不变，但对象布局改变。
- ❌ C++ 有标准 ABI — 错误。C++ 标准不定义 ABI，只有平台事实标准。

**延伸点**：保持 ABI 稳定的工程实践包括不修改现有非静态数据成员顺序/类型、不删除或重排现有虚函数、新增功能通过 Pimpl 的 Impl 扩展。

### 40. 名字查找的"找到即停"

派生类中声明的同名函数会在查找阶段"筑墙"，完全隐藏基类重载；重载决议仅在可见候选集中进行。

**场景示例：**
```cpp
class Base { public: void f(int) {} };
class Derived : public Base {
public:
    void f(double) {}  // 隐藏了 Base::f(int) 的全部重载
};
Derived d;
d.f(1);  // ❌ 编译错误：Derived::f(double) 隐藏了 Base::f(int)
```

**原理解析 + 常见误区**

- ❌ 派生类 f(double) 和基类 f(int) 是重载 — 错误。二者不在同一作用域，不构成重载。
- ❌ 签名不同就不会隐藏 — 错误。C++ 名字隐藏只看标识符名，不看签名。

**延伸点**：ADL（参数依赖查找）是名字查找的例外扩展，使 `std::cout << obj` 和 `swap(obj1, obj2)` 惯用法得以工作。

### 41. 数组参数退化影响函数重载

数组实参在按值传递时退化为指针，导致 `f(int[])` 与 `f(int*)` 实质为同一函数无法重载；而数组引用 `f(int (&)[N])` 阻止退化。

**原理解析 + 常见误区**

- ❌ `f(int[])` 和 `f(int*)` 是重载 — 错误。函数参数中的数组大小被忽略，二者是同一函数类型的重声明。
- ❌ `f(int[5])` 能精确匹配大小为 5 的数组 — 错误。函数参数中的数组大小是纯语法噪音，编译器完全忽略。

**延伸点**：模板推导中的数组退化差异决定了 `std::begin` 的实现策略：`T (&arr)[N]` 优先匹配未退化的数组。

---

### 42. 数组引用 vs 指针重载优先级

`int[2]` 匹配 `int (&)[2]` 是 Identity（精确匹配），而 `int[2]` 退化为 `int*` 属于标准转换；精确匹配严格优于标准转换。

| 候选 | 形参 | 对 `int[2]` 的 ICS | 等级 | 结果 |
|------|------|-------------------|------|------|
| 候选1 | `int*` | Array-to-pointer | 标准转换 | 落选 |
| 候选2 | `int (&)[2]` | 引用绑定（同类型） | 精确匹配 | ✅ 胜出 |

**原理解析 + 常见误区**

- ❌ 两者都是精确匹配，导致 ambiguous — 错误。数组到指针是标准转换，不是精确匹配；引用绑定才是精确匹配。
- ❌ 编译器随机选一个 — 错误。规则明确，所有符合标准的编译器都选择引用版本。

**延伸点**：标准库 `std::begin` 的重载设计正是利用此规则实现优先匹配：数组优先匹配引用版本（保留大小信息），而已退化的指针或容器则匹配通用模板。
