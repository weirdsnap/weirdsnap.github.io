# 三、内存管理与资源安全（43–57）

---

### 43. `new`/`delete` vs `malloc`/`free`

`new`/`delete` 是运算符，自动调用构造/析构，返回精确类型指针，失败抛异常；`malloc`/`free` 仅分配/释放原始内存。

| 维度 | `new`/`delete` | `malloc`/`free` |
|------|----------------|-----------------|
| 构造/析构 | ✅ 自动调用 | ❌ 不调用 |
| 返回类型 | 精确类型指针 | `void*` |
| 失败行为 | 抛 `std::bad_alloc` | 返回 `NULL` |

**原理解析 + 常见误区**

- ❌ 用 `malloc` 构造 C++ 对象安全 — 错误。`malloc` 不调用构造函数，对象处于未初始化状态。
- ❌ `new[]` 配 `delete` 安全 — 错误。`new[]` 必须用 `delete[]`，否则只析构第一个元素。

**延伸点**：placement new 在已分配内存上构造对象，需手动调用析构函数，且不能对该指针使用 `delete`。

### 44. `shared_ptr`

`shared_ptr` 通过控制块管理引用计数，支持自定义删除器；引用计数操作线程安全，但对象本身非线程安全。

| 操作 | shared_count | 说明 |
|------|-------------|------|
| 构造 | +1 | 初始化为 1 |
| 拷贝 | +1 | 共享所有权 |
| 移动 | 不变 | 转移所有权 |
| 析构 | -1 | 归零时释放 |

**原理解析 + 常见误区**

- ❌ `shared_ptr` 线程安全所以可以多线程无锁读写同一对象 — 错误。引用计数原子操作仅保证计数正确，不保证对象内容的线程安全。
- ❌ 用 `this` 构造新的 `shared_ptr` — 错误。形成双控制块，重复释放 → UB。应继承 `std::enable_shared_from_this`。

**延伸点**：`make_shared` 通常将对象内存与控制块连续分配，缓存友好；但缺点是只要存在 `weak_ptr`，对象内存也不会被归还。

---

### 45. `weak_ptr`

`weak_ptr` 不增加强引用计数，通过 `lock()` 安全观测对象生命周期，用于打破循环引用。

| 维度 | `shared_ptr` | `weak_ptr` |
|------|-------------|-----------|
| 引用计数 | shared_count + 1 | weak_count + 1，shared_count 不变 |
| 对象释放权限 | ✅ 控制生命周期 | ❌ 仅观测 |
| 直接访问 | 可直接解引用 | 必须先 `lock()` |

**原理解析 + 常见误区**

- ❌ `weak_ptr` 不增加任何计数 — 错误。它增加 `weak_count`，控制块因此不会立即释放。
- ❌ 直接解引用 `weak_ptr` — 错误。必须先 `lock()` 转成 `shared_ptr`。

**延伸点**：`std::enable_shared_from_this` 确保与外部已有的 `shared_ptr` 共享同一控制块，避免双控制块问题。

---

### 46. `unique_ptr` vs `shared_ptr`

`unique_ptr` 独占所有权零开销；`shared_ptr` 共享所有权有原子引用计数开销。

| 维度 | `unique_ptr` | `shared_ptr` |
|------|-------------|-------------|
| 所有权 | 独占，仅可移动 | 共享，可复制 |
| 内存开销 | 与裸指针相同 | 对象指针 + 控制块指针 |
| 删除器 | 编译期类型擦除（零开销） | 运行时类型擦除（有开销） |

**原理解析 + 常见误区**

- ❌ `shared_ptr` 总是用 `make_shared` 更好 — 错误。`make_shared` 一次性分配，但存在 `weak_ptr` 时对象内存被钉住。
- ❌ `unique_ptr` 完全无开销 — 有自定义删除器时对象体积可能大于裸指针，但无控制块分配。

**延伸点**：`unique_ptr` 的自定义删除器是模板参数（编译期类型擦除），`shared_ptr` 的删除器存储在控制块中（运行时类型擦除）。

---

### 47. 拷贝构造 vs 拷贝赋值

拷贝构造函数用于初始化新对象；拷贝赋值运算符用于给已存在的对象重新赋值。

| 维度 | 拷贝构造函数 | 拷贝赋值运算符 |
|------|-------------|--------------|
| 调用时机 | 对象创建时 | 对象已存在时 |
| 自身资源 | 无需处理 | 必须先释放旧资源 |
| 自赋值安全 | 不涉及 | 必须处理 |
| 返回值 | 无 | `T&`（支持链式赋值） |

**原理解析 + 常见误区**

- ❌ `T a = b;` 是赋值 — 错误。这是拷贝初始化，调用拷贝构造函数。
- ❌ 赋值运算符不需要返回值 — 错误。返回 `T&` 是标准约定，缺失会导致链式赋值编译失败。

**延伸点**：拷贝并交换惯用法（Copy-and-Swap Idiom）通过值传递隐式完成拷贝，利用临时对象析构自动释放旧资源，天然保证自赋值安全和强异常安全。

### 48. 三/五/零法则

若类需自定义析构、拷贝或拷贝赋值中的任一，则几乎肯定需要全部三者（三法则）；C++11 起扩展为五法则，额外需自定义移动构造和移动赋值；现代 C++ 优先遵循零法则。

| 法则 | 涉及的特殊成员 | 核心思想 |
|------|---------------|----------|
| 三法则 | 析构、拷贝构造、拷贝赋值 | 管理原始资源时三者必须同步自定义 |
| 五法则 | 三法则 + 移动构造 + 移动赋值 | 资源类必须具备移动能力 |
| 零法则 | 均不自定义 | 用智能指针、标准容器避免自定义 |

**原理解析 + 常见误区**

- ❌ 五法则意味着必须手写五个函数 — 错误。可通过 `= default` 显式恢复编译器生成，或遵循零法则。
- ❌ 定义了析构函数就必须定义拷贝 — 不严谨。三法则/五法则是设计信号而非语法强制。

**延伸点**：C++11 的特殊成员函数生成规则：用户声明析构、拷贝构造或拷贝赋值中的任一，编译器不会自动生成移动操作。老式资源类在 C++11 下会静默失去移动能力。

### 49. 成员初始化列表顺序

成员构造与析构严格按类内声明顺序执行，与初始化列表书写顺序无关。

**场景示例：**
```cpp
class A {
    int m_b;  // 先声明
    int m_a;  // 后声明
public:
    A(int v) : m_a(v), m_b(m_a) {}  // m_b 先构造，用未初始化的 m_a → UB
};
```

**原理解析 + 常见误区**

- ❌ 初始化列表顺序就是构造顺序 — 错误。编译器始终按类内声明顺序构造。
- ❌ 析构顺序与初始化列表逆序 — 错误。析构只与声明顺序逆序相关。

**延伸点**：C++11 默认成员初始化器与初始化列表共存时，未指定的成员仍按声明顺序执行默认初始化，进一步说明构造顺序由声明顺序锚定。

### 50. 移动语义

通过右值引用和移动构造函数将资源所有权从临时对象转移至新对象，避免深拷贝开销。

| 维度 | 拷贝语义 | 移动语义 |
|------|----------|----------|
| 参数类型 | `const T&` | `T&&` |
| 资源行为 | 分配新资源，复制内容 | 掠夺源对象资源，置空源指针 |
| 复杂度 | O(n) 深拷贝 | O(1) 指针转移 |

**原理解析 + 常见误区**

- ❌ 移动后源对象被销毁 — 错误。源对象仍在生命周期内，处于有效但未指定状态。
- ❌ `std::move` 是移动操作 — 错误。`std::move` 只是类型转换，真正的移动发生在移动构造函数中。

**延伸点**：移动构造必须标记 `noexcept`，否则标准库容器扩容时不会使用（回退到拷贝构造）。

### 51. RVO 与 NRVO

编译器通过省略临时对象的创建和拷贝/移动，直接在调用者栈帧上构造返回值。

| 优化类型 | C++17 前 | C++17 起 |
|----------|----------|----------|
| RVO（返回临时对象） | 编译器可选 | **强制保证**（guaranteed copy elision） |
| NRVO（返回命名局部变量） | 编译器可选 | 仍可选 |

**原理解析 + 常见误区**

- ❌ RVO 依赖移动语义 — 错误。RVO 是编译期优化，完全跳过移动/拷贝。
- ❌ 返回 `std::move(local)` 能加速 — 错误。阻止 NRVO，引入不必要的移动构造。

**延伸点**：C++17 的 prvalue 规则使 `func(S())` 中 `S()` 直接初始化形参，不经过临时对象，形成完整的零拷贝链。

---

### 52. 返回 prvalue 的零拷贝

`return S();` 中的 `S()` 是纯右值，C++17 起标准强制保证不触发任何拷贝或移动构造。

| 返回写法 | C++17 前 | C++17 起 | 建议 |
|----------|----------|----------|------|
| `return S();` | 编译器可选 RVO | **保证零拷贝** | ✅ |
| `return local;` | 编译器可选 NRVO | 仍可选 NRVO | ✅ |
| `return std::move(local);` | 强制移动构造 | 强制移动构造 | ❌ |

**原理解析 + 常见误区**

- ❌ `return S()` 会调用默认构造 + 拷贝构造 — 错误。C++17 起零拷贝。
- ❌ `return std::move(local)` 更保险 — 错误。阻止 NRVO，引入不必要的移动构造。

**延伸点**：C++17 prvalue 规则使 `func(S())` 中 `S()` 的 prvalue 直接初始化形参，不经过临时对象，形成完整的零拷贝链。

### 53. RAII 代理模式（栈管理堆资源）

堆资源必须通过栈上的 RAII 代理对象（如智能指针）间接管理，利用栈展开机制确保异常安全释放。

| 资源位置 | 栈展开时是否自动释放 | 管理手段 |
|----------|-------------------|---------|
| 栈对象 | ✅ 自动析构 | 正确构造/析构即可 |
| 堆裸指针 | ❌ 不自动释放 | 手动配对，异常即泄漏 |
| RAII 代理 | ✅ 代理析构时释放 | 栈代理对象析构触发 `delete` |

**原理解析 + 常见误区**

- ❌ 类的析构函数写得正确，堆资源就安全 — 错误。析构函数只定义了"如何释放"，但"何时调用"由对象生命周期决定。
- ❌ try-catch 能覆盖所有泄漏路径 — 错误。多个 return 点、深层调用栈抛出的异常都会导致 `delete` 路径被跳过。

**延伸点**：构造函数中的异常安全：始终在 `new` 表达式处立即交给智能指针（如 `member_(std::make_unique<T>())`），确保构造失败时也能被栈展开正确清理。

### 54. 模板静态成员 vs 非模板静态成员

模板静态成员遵循按需实例化（ODR-use 驱动），非模板静态成员在程序启动前（`main` 之前）完成初始化。

| 维度 | 模板静态成员 | 非模板静态成员 |
|------|-------------|---------------|
| 初始化时机 | 首次 ODR-use 时隐式实例化 | 程序启动前（`main` 之前） |
| 与使用关系 | 不用不生成，不初始化 | 只要链接进程序，无论是否使用 |
| 定义位置 | 通常头文件 | 通常单个 cpp |

**原理解析 + 常见误区**

- ❌ 模板静态成员在模板定义时就生成 — 错误。模板静态成员在模板定义时仅有声明，直到某处代码 ODR-use 时才触发隐式实例化。
- ❌ 未使用的非模板静态成员不会初始化 — 错误。只要目标文件被链接进最终可执行文件，其静态成员就会在 `main` 前初始化。

**延伸点**：C++17 `inline` 变量允许非模板静态成员在头文件中定义且跨翻译单元合并，打破了传统"声明在头、定义在 cpp"的分离编译限制。

### 55. 特殊成员函数的自动生成规则

C++ 编译器在类未显式声明特定特殊成员时，按规则自动生成；但一旦用户声明了析构、拷贝或移动中的任一，相关自动生成会被抑制或删除。

| 特殊成员 | 自动生成条件 | 被抑制/删除的条件 |
|----------|-------------|------------------|
| 默认构造 | 无用户声明的任何构造函数 | 声明任意构造函数即抑制 |
| 拷贝构造 | 无用户声明的拷贝构造 | 声明移动构造/移动赋值 → **删除** |
| 移动构造 | 无拷贝/移动/析构/赋值声明 | 声明上述任一即**抑制** |
| 析构 | 无用户声明的析构 | C++11 起隐式 `noexcept` |

**原理解析 + 常见误区**

- ❌ 声明了析构不影响移动生成 — 错误。声明析构会抑制移动构造和移动赋值的自动生成。
- ❌ 移动操作总是自动生成 — 错误。条件极为严格：类中不能有任何用户声明的拷贝、移动、析构或赋值操作。

**延伸点**：Rule of Zero vs Rule of Five：若类不需要自定义资源管理，不声明任何特殊成员（Rule of Zero）；一旦需要自定义，则必须显式处理全部五个（Rule of Five）。

### 56. 用户声明拷贝构造抑制移动（基于题目）

用户声明拷贝构造函数会抑制编译器自动生成移动构造函数；当移动构造不可用时，`std::move(c)` 产生的 xvalue 可通过 `const T&` 拷贝构造函数完成初始化。

**场景示例：**
```cpp
class C {
public:
    C() {}
    C(const C&) {}  // 用户声明拷贝构造 → 隐式移动构造被抑制
};
C c;
C c2(std::move(c));  // ✅ 编译通过：调用拷贝构造
```

**原理解析 + 常见误区**

- ❌ `std::move` 必须调用移动构造 — 错误。`std::move` 只是类型转换，真正的移动发生在移动构造函数中；若移动构造不存在，拷贝构造是合法回退。
- ❌ `const C&` 不能绑定到右值 — 错误。非 volatile 的 `const T&` 可以绑定到 rvalue。

**延伸点**：拷贝构造参数为何必须是 `const T&`：除了保护原对象不被修改，接受右值是另一关键原因。若拷贝构造声明为 `T&`，则不仅无法接受 `const` 左值，也无法接受 `std::move` 产生的右值。

### 57. 模板实例化延迟

模板函数或类模板的实例化不是发生在定义点，而是延迟到实际使用点（代码中首次调用、取地址或创建对象时）才生成具体代码；未被使用的成员函数不参与实例化。

**场景示例：**
```cpp
template<typename T>
struct Foo {
    void used() { T::bar(); }
    void unused() { T::baz(); }  // 从不调用，不编译检查
};
Foo<int> f;        // 类模板实例化：仅生成数据成员和虚函数相关
f.used();          // 此时才实例化 used() 的函数体
```

**原理解析 + 常见误区**

- ❌ 模板定义时编译通过 = 实例化没问题 — 错误。定义时只检查语法，不检查模板参数相关的语义。
- ❌ 模板类实例化时所有成员函数都被编译 — 错误。C++ 标准规定按需实例化：只有被 odr-used 的成员函数才会实例化。

**延伸点**：SFINAE（Substitution Failure Is Not An Error）正是依赖延迟实例化机制：当模板替换导致无效类型时，该特化不会编译报错，而是静默从候选集移除。
