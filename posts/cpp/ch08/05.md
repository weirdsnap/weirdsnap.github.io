# C++ 未定义行为（Undefined Behavior, UB）

## 一、什么是 UB

**未定义行为（UB）** 指 C++ 标准没有规定结果的程序行为。标准不保证：
- 程序会崩溃还是正常运行
- 输出什么值
- 编译器生成什么机器码

**关键认知**：编译器**假设 UB 永远不会发生**，并据此进行优化。这意味着 UB 不只是"结果不确定"，更可能让你的代码被优化成完全意想不到的逻辑。

---

## 二、为什么 UB 很危险

### 2.1 优化后的反直觉结果

```cpp
int foo(int x) {
    if (x < 0) return -1;
    int y = x + 1;              // 若 x == INT_MAX，这里 UB
    if (x < 0) return 1;        // 编译器可能删除这个分支
    return 0;
}
```

编译器推断："`x + 1` 没有溢出，所以 `x < 0` 在第一次不成立后永远不可能再成立"，于是第二次 `if` 可能被直接删除。

### 2.2 "碰巧正常"的假象

同一段 UB 代码在不同环境可能：
- 调试模式通过，发布模式崩溃
- 本地运行正常，服务器上异常
- 今天正常，明天换编译器版本后出错

## 三、常见 UB 分类

### 3.1 内存相关

| UB 场景 | 典型示例 | 常见后果 |
|---------|---------|---------|
| Use-after-free | `delete p; *p;` | 堆损坏、信息泄漏 |
| Double-free | `delete p; delete p;` | 堆管理器元数据破坏 |
| 缓冲区溢出 | `arr[n]` 其中 `n >= size` | 栈/堆金丝雀破坏 |
| 未初始化使用 | `int x; cout << x;` | 垃圾值、信息泄漏 |
| 悬空引用 | `int& f() { int x; return x; }` | 访问已释放栈空间 |

### 3.2 算术相关

| UB 场景 | 典型示例 | 说明 |
|---------|---------|------|
| 有符号整数溢出 | `INT_MAX + 1` | 未定义，不可依赖回绕 |
| 除以零 | `x / 0` | 通常触发 SIGFPE |
| 有符号整数左移溢出 | `1 << 31` | 可能改变符号位 |
| 无符号数转为有符号数越界 | `(int)UINT_MAX` | 实现定义或 UB |

> 注意：**无符号整数溢出不是 UB**，而是按模 2^N 回绕（定义良好）。详见 [无符号整型回绕](../ch01/01.md)。

### 3.3 对象生命周期相关

| UB 场景 | 典型示例 |
|---------|---------|
| 在对象生命周期开始前/后访问 | `delete p; p->foo();` |
| 访问已销毁栈对象 | 返回局部对象的引用/指针 |
| placement new 后未使用 `std::launder` | 编译器可能不认识新对象 |

### 3.4 类型系统相关

| UB 场景 | 典型示例 |
|---------|---------|
| 违反严格别名规则 | `float f; int* p = (int*)&f; *p;` |
| 对象切片破坏多态 | 按值传递/返回派生类给基类 |
| `this == nullptr` 调用成员 | `((T*)nullptr)->f();` |

### 3.5 执行顺序相关

| UB 场景 | 典型示例 |
|---------|---------|
| 同一标量对象在相邻序列点间被修改两次 | `i = ++i + 1;` |
| 标量对象在相邻序列点间同时被读写 | `a[i] = i++;` |

---

## 四、UB 受哪些因素影响

| 因素 | 影响 |
|------|------|
| **编译器** | GCC / Clang / MSVC 优化策略不同 |
| **编译器版本** | 新版本可能引入更激进优化 |
| **优化级别** | `-O0` 可能"正常"，`-O2`/`-O3` 可能删除代码 |
| **目标架构** | x86 / ARM / RISC-V 底层行为差异 |
| **操作系统** | 内存布局、栈保护、ASLR 等 |
| **运行时上下文** | 同一段代码在不同调用路径下优化结果可能不同 |

## 五、UB 检测工具

| 工具 | 检测重点 | 使用方式 |
|------|---------|---------|
| **AddressSanitizer (ASan)** | 内存越界、use-after-free、double-free | `-fsanitize=address` |
| **MemorySanitizer (MSan)** | 未初始化内存读取 | `-fsanitize=memory` |
| **UndefinedBehaviorSanitizer (UBSan)** | 有符号溢出、对齐错误、空指针解引用等 | `-fsanitize=undefined` |
| **Valgrind** | 内存泄漏、越界、未初始化 | 无需重编译 |
| **静态分析器** | Clang Static Analyzer、PVS-Studio、Coverity | 编译期检查 |

---

## 六、工程实践建议

1. **优先使用现代 C++ 特性**
   - 智能指针避免 use-after-free/double-free
   - `std::array` / `std::vector` 避免裸数组越界
   - `std::optional` / `std::variant` 避免魔数哨兵
   - `enum class` 避免隐式转换

2. **编译期启用 sanitizers**
   ```bash
   g++ -fsanitize=address,undefined -g -O1 main.cpp
   ```

3. **避免依赖"碰巧正常"**
   - 有符号溢出不要依赖回绕，用无符号类型或显式检查
   - 不要假设悬空指针访问会崩溃

4. **对关键代码使用静态分析**
   - Clang Static Analyzer、SonarQube、PVS-Studio 等能在编译前发现大量 UB
