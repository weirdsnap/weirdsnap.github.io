# C++ 异常堆栈信息的完整获取方案

> 本文是 [Python → C++：异常处理的定位信息去哪了？](/posts/python2cpp/ch04.md) 的进阶篇。
> 在了解 `__FILE__/__LINE__` 入门方案之后，这里由浅入深覆盖各类场景的完整方案。

---

## 1. 入门：`__FILE__` / `__LINE__` 宏埋点（零依赖）

最简单、零外部依赖的方案。在自定义异常类的构造函数里用预定义宏记录位置：

```cpp
#include <iostream>
#include <stdexcept>

class traced_exception : public std::runtime_error {
public:
    traced_exception(
        const std::string& msg,
        const char* file = __FILE__,
        int line = __LINE__
    ) : std::runtime_error(msg + " [" + file + ":" + std::to_string(line) + "]") {}
};

void inner() {
    throw traced_exception("出错了");
}

int main() {
    try {
        inner();
    } catch (const std::exception& e) {
        std::cout << e.what() << "\n";
        // 输出：出错了 [demo.cpp:17]
    }
}
```

**优点**：C++98 就能用，没有任何依赖，团队统一封装一个基类即可。
**局限**：只能记录抛出点的文件名和行号，没有函数名和完整调用链。

## 2. 进阶：`std::source_location`（C++20 现代写法）

如果项目允许用 C++20，`std::source_location` 比宏更干净，还能拿到函数名：

```cpp
#include <iostream>
#include <source_location>
#include <stdexcept>

class traced_exception : public std::runtime_error {
public:
    traced_exception(
        const std::string& msg,
        std::source_location loc = std::source_location::current()
    ) : std::runtime_error(
            msg + " [" + loc.function_name() + " at "
            + loc.file_name() + ":" + std::to_string(loc.line()) + "]"
        ) {}
};

void inner() {
    throw traced_exception("出错了");
}

int main() {
    try {
        inner();
    } catch (const std::exception& e) {
        std::cout << e.what() << "\n";
        // 输出：出错了 [inner() at demo.cpp:19]
    }
}
```

## 3. 调试器：GDB 动态捕获（零侵入）

不写一行代码，直接靠调试器在 `throw` 触发时截停程序看调用链。

```bash
gdb ./program
(gdb) catch throw
(gdb) run
# 程序会在 throw 处停下
(gdb) bt
# 显示完整调用链
```

Visual Studio 同理：在"异常设置"窗口中勾选 C++ 异常即可。

**适用场景**：开发阶段可复现的问题，零侵入，最快定位。

---

## 4. 野路子：调试阶段的暴力手法

当 try-catch 嵌套太深、删不干净时，可以用两种手段让程序直接死在异常源头。

### 手法 A：去掉 try-catch，让异常未被捕获

C++ 异常如果没被捕获，会一路往上抛直到 `std::terminate()`。把 try-catch 删掉，配合 GDB：

```bash
gdb ./program
(gdb) catch throw
(gdb) run
# 程序在 throw 处断下，不需要关心哪层 catch 会接住它
(gdb) bt
# 完整调用链
```

### 手法 B：在构造函数里触发段错误

异常对象在 `throw` 之前构造，所以构造函数里崩溃就等于死在抛出点：

```cpp
class killer_exception : public std::runtime_error {
public:
    killer_exception(const std::string& msg)
        : std::runtime_error(msg) {
        volatile int* p = nullptr;
        *p = 42;  // 段错误，死在 throw 的前一刻
    }
};

void inner() {
    throw killer_exception("出错了");  // 构造时直接挂掉
}
```

```bash
gdb ./program
(gdb) run
# 程序段错误停在 killer_exception 构造函数里
(gdb) bt
# 看到完整调用链
```

**适用场景**：调试阶段的孤注一掷，定位完立刻删除。

## 5. 线上：Core Dump 事后分析

针对线上偶发的崩溃，没法现场打断点。在 `catch` 里主动崩溃生成 Core 文件，事后分析。

```cpp
try {
    // ...
} catch (...) {
    std::abort();  // 触发 core dump
}
```

```bash
ulimit -c unlimited        # 允许生成 core 文件
./program                  # 运行程序，触发 abort
# 生成 core 文件
gdb program core           # 事后加载
(gdb) bt                   # 查看崩溃前的完整调用路径
```

## 6. 日志：运行时平台 API

适用于不能断点、需要把堆栈写入日志的场景。

### Linux 轻量：`<execinfo.h>`

```cpp
#include <iostream>
#include <execinfo.h>

void print_backtrace() {
    constexpr int MAX_FRAMES = 64;
    void* buffer[MAX_FRAMES];
    int frames = backtrace(buffer, MAX_FRAMES);
    char** symbols = backtrace_symbols(buffer, frames);

    for (int i = 0; i < frames; ++i) {
        std::cout << symbols[i] << "\n";
    }
    free(symbols);
}

void inner() {
    throw std::runtime_error("出错了");
}

int main() {
    try {
        inner();
    } catch (...) {
        std::cout << "Caught exception. Backtrace:\n";
        print_backtrace();
    }
}
```

编译加 `-rdynamic` 才能看到函数名：

```bash
g++ -rdynamic demo.cpp -o demo
```

### 跨平台：Boost.Stacktrace

```cpp
#include <boost/stacktrace.hpp>
#include <iostream>

try {
    // ...
} catch (const std::exception& e) {
    std::cerr << boost::stacktrace::stacktrace();
}
```

## 总结

| 层级 | 方案 | 适合场景 |
|:---|:---|:---|
| ① 入门 | `__FILE__/__LINE__` 宏埋点 | 零依赖，团队统一封装 |
| ② 进阶 | `std::source_location`（C++20） | 需要函数名，更现代化 |
| ③ 调试器 | GDB `catch throw` | 开发阶段零侵入定位 |
| ④ 野路子 | 去 try-catch / 构造时段错误 | 调试期暴力突破 |
| ⑤ 线上 | Core Dump 事后分析 | 线上偶发崩溃 |
| ⑥ 日志 | `<execinfo.h>` / Boost.Stacktrace | 需要堆栈字符串写入日志文件 |

请彻底放弃在标准 C++ 的 `catch` 块中"凭空"获取堆栈的幻想，主动出击才是正解。
