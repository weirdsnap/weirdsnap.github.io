# noexcept

`noexcept` 是函数对调用者的编译期契约，承诺不抛出异常；若违反则调用 `std::terminate`。

| 特性 | `noexcept` | 无标记 |
|------|-----------|--------|
| 异常传播 | 触发 `std::terminate` | 正常传播 |
| 栈展开保证 | ❌ 不保证 | ✅ 保证 |
| 容器移动（vector 扩容） | ✅ 优先使用 | ❌ 回退拷贝 |

## noexcept 规范隐式推导

函数没有 `noexcept` 说明符时，标准按函数类型区分默认推断结果（C++11 起）：

| 函数类型 | 隐式 noexcept 推导 |
|---------|------------------|
| 普通函数/构造函数 | 默认 potentially-throwing |
| 析构函数（无 noexcept-specifier） | 隐式 `noexcept(true)`，除非子对象析构 potentially-throwing |
| 默认构造/拷贝/移动（`= default` 首次声明） | 按其子对象对应操作推导 |
| 释放函数（`operator delete`） | 隐式 `noexcept(true)` |

**为何析构函数特殊？** C++11 起，析构中抛异常会极大破坏栈展开机制，因此标准默认给析构函数加上 `noexcept` 保护。即使是用户提供的非默认析构函数，只要没有写 `noexcept(false)` 且所有子对象析构不抛，就隐式 `noexcept(true)`。

```cpp
#include <iostream>
#include <stdexcept>
#include <type_traits>

struct S {
    S() {
        throw std::runtime_error("");
    }
    ~S() {
        throw std::runtime_error("");
    }
};

int main() {
    std::cout
        << (std::is_nothrow_constructible_v<S> ? 'y' : 'n')
        << (std::is_nothrow_destructible_v<S> ? 'y' : 'n');
}
// 输出：ny
```

推理过程：

- **构造函数** `S()`：没有 `noexcept` 说明符，不是析构/释放/默认函数 → 默认 potentially-throwing → `is_nothrow_constructible_v<S>` 为 `false` → `'n'`
- **析构函数** `~S()`：没有 `noexcept` 说明符，即使由用户提供，仍适用隐式推导规则。`S` 没有子对象（没有成员或基类），因此析构函数隐式 `noexcept(true)` → `is_nothrow_destructible_v<S>` 为 `true` → `'y'`

> `noexcept` 规范取决于函数签名上是否声明了 noexcept，**与函数体内实际是否抛异常无关**。构造函数体内有 `throw` 不影响它已是 potentially-throwing 的事实；析构函数体内有 `throw` 也不改变它隐式 `noexcept(true)` 的契约——运行时违反仍触发 `std::terminate`。

## 延伸点

- `noexcept` 运算符（编译期检查）配合条件 `noexcept` 可实现模板函数的异常安全转发。
- `std::is_nothrow_*` 类型 traits 查询的是函数的声明 noexcept 规范，不是运行时分析。
- 构造函数和析构函数在隐式 noexcept 上的不对称性（构造函数默认抛、析构函数默认不抛）是 C++ 异常安全设计的核心取舍。
