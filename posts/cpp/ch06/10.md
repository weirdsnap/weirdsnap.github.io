# `std::condition_variable`：条件变量

条件变量用于**等待某个条件成立**，配合 `std::mutex` 和共享数据，是实现生产者-消费者模型的核心机制。

---

## 一、基本用法

```cpp
#include <condition_variable>
#include <mutex>
#include <queue>

std::queue<int> q;
std::mutex mtx;
std::condition_variable cv;

void producer() {
    for (int i = 0; i < 10; ++i) {
        {
            std::lock_guard<std::mutex> lock(mtx);
            q.push(i);
        }
        cv.notify_one();   // 通知消费者
    }
}

void consumer() {
    for (int i = 0; i < 10; ++i) {
        std::unique_lock<std::mutex> lock(mtx);
        cv.wait(lock, [] { return !q.empty(); });  // 👈 等待条件
        int val = q.front();
        q.pop();
        lock.unlock();
        std::cout << val << "\n";
    }
}
```

---

## 二、`wait` 的两种形式

```cpp
// 形式 1：无谓词——可能被虚假唤醒
cv.wait(lock);                     // 被唤醒后需自己检查条件

// 形式 2：有谓词（推荐）——自动处理虚假唤醒
cv.wait(lock, [] { return ready; });  // 等价于 while (!ready) cv.wait(lock);
```

**为什么需要谓词？** 因为 `wait` 可能发生**虚假唤醒（spurious wakeup）**——没有通知也会偶尔返回。谓词版本内部自动用 `while` 循环检查条件。

---

## 三、通知方式

| 函数 | 效果 |
|------|------|
| `notify_one()` | 唤醒一个等待线程 |
| `notify_all()` | 唤醒所有等待线程 |

- 通常 `notify_one()` 就够了，除非有多个消费者等不同条件
- 通知时**不需要持有锁**，提前解锁可以避免唤醒后线程抢锁

---

## 四、`notify_one` vs `notify_all` 的选择

| 场景 | 推荐 |
|------|------|
| 单生产者/单消费者 | `notify_one()` |
| 多消费者等待同一条件 | `notify_one()`（谁抢到谁处理）|
| 多消费者等待不同条件 | `notify_all()` |

---

## 五、常见误区

- ❌ 必须先持有锁再 `wait` — ✅ `wait` 会自动解锁并阻塞，返回时重新加锁。
- ❌ `notify_one` 必须在持有锁时调用 — ✅ 不需要，提前解锁可以减少争用。
- ❌ 条件变量可以单独使用（不需要 mutex）— ✅ 条件变量必须配合 mutex，因为条件判断本身需要互斥。
- ❌ 不用谓词的 `wait` 是安全的 — ✅ 忽略虚假唤醒可能使程序在条件不满足时继续执行。
- ❌ `wait` 返回时条件一定满足 — ✅ 可能有虚假唤醒，必须用 while 或谓词重新检查。

---

## 六、一句话总结

> 条件变量 + `mutex` + 共享数据 = 线程安全的生产者-消费者模型；始终用带谓词的 `wait` 来避免虚假唤醒。
