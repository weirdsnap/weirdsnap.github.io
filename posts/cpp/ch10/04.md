# 头文件保护：#pragma once vs ifndef 守卫

C/C++ 中头文件可能被多个源文件包含，如果内容重复定义会导致编译错误。常见的两种保护方式是 `#pragma once` 和 `ifndef` 宏守卫。

## 两种写法

### `#pragma once`

```cpp
#pragma once

struct Point { int x, y; };
```

编译器保证同一个文件只被包含一次，写法简洁。

### `ifndef` 宏守卫

```cpp
#ifndef POINT_H
#define POINT_H

struct Point { int x, y; };

#endif // POINT_H
```

用宏控制是否展开头文件内容，是标准 C++ 的一部分，所有编译器都支持。

## 对比

| 特性 | `#pragma once` | `ifndef` 守卫 |
|------|----------------|---------------|
| 标准性 | 编译器扩展，非标准 | 标准 C++ |
| 简洁性 | 一行 | 三行 + 宏名 |
| 编译速度 | 通常更快（按文件路径判断） | 需扫描宏定义 |
| 宏名冲突 | 无 | 需要保证全局唯一 |
| 边缘情况 | 硬链接/特殊路径下可能误判 | 理论上更稳定 |

## 工程实践：两者结合

现代 C++ 项目常用**两者结合**的方式：

```cpp
#pragma once
#ifndef POINT_H
#define POINT_H

// 头文件内容

#endif // POINT_H
```

**好处**：
- `#pragma once` 让现代编译器快速跳过重复包含
- `ifndef` 提供标准兼容的兜底
- 即使 `#pragma once` 失效，宏守卫仍能工作

## 命名约定

宏名通常与头文件名对应，全大写，路径分隔符换成下划线：

| 文件路径 | 宏名 |
|----------|------|
| `point.h` | `POINT_H` |
| `utils/math.h` | `UTILS_MATH_H` |
| `fundamental_types_utils.h` | `FUNDAMENTAL_TYPES_UTILS_H` |

## 常见误区

- ❌ 头文件完全不写保护 — 错误。只要可能被多个文件包含，就要保护。
- ❌ `ifndef` 宏名和其他头文件冲突 — 会导致奇怪的错误，比如一个头文件被意外跳过。
- ❌ 在 `#pragma once` 头文件里定义全局变量或函数（非 inline）— 仍会导致链接重复定义，这不是头文件保护能解决的。

## 一句话总结

个人项目用 `#pragma once` 足够；追求可移植和工程规范时，用 `#pragma once` + `ifndef` 两者结合。
